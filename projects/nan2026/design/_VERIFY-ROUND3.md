■ ★ 실패 1 — **`bossTimeoutRate.min = 0.02`가 정본 자기 콘텐츠에서 깨진다**
`timeout = s(1−d)(1−(ds)⁶)/(1−ds)`를 s = 0.93(§13.2-⑨의 저작 계약 평균)에 대해 전수 계산:
  d=0.950 → runClear 0.476 · timeout **0.209** ✔
  d=0.980 → runClear 0.573 · timeout **0.090** ✔
  d=0.995 → runClear 0.628 · timeout **0.0232** ✔ (하한 붙음)
  d=0.9988 → timeout **0.0056** ✗ / d=0.9998 → timeout **0.0009** ✗
→ **허용 창 d ∈ [0.939, 0.995].** 그런데 §13.6.2가 인쇄한 균형 killTime(125.8~135.7s)이 주는 **Πd(probe) = 0.9966 → d ≤ 0.9994**. 즉 **probe 천장이 창의 밖**이고, 창 안으로 끌어내리는 유일한 장치가 probe(maxFarm) ↔ run(balanced)의 화력 차이인데 **`dpsRef`의 farm 정책이 미정이라 그 차이를 아무도 모른다.** §13.2-④는 d ∈ {0.95, 0.97, 0.98}만 표로 대입하고 **자기 콘텐츠의 d를 넣지 않았다** — v1.0이 `stanceValue`에서 「0.55를 가정으로 넣은」 것과 같은 사인이며, v1.1이 §13.2-③에서 그 사인을 진단하고 고친 **바로 다음 항목**에서 재발했다.

■ ★ 실패 2 — **§13.2가 Πs에 상호 배타적인 두 값을 쓴다**
②③: `Πs ≈ 0.88` → 스테이지당 s = **0.9789** → ⑨가 **저작 계약**으로 선언한 s_t ≤ 0.97의 **밖**(계약상 Πs ≤ 0.833).
⑨: `Πs(mean) = 0.93⁶ = 0.647`.
정본 자기 콘텐츠(Πd = 0.9966) 대입 → **Πs 0.88이면 runClear = 0.877 → `runClearRate ≤ 0.65` 실패** / Πs 0.647이면 **0.645 → 통과(여유 0.5%p)**. ②③이 쓴 값이 **자기 게이트를 깨는 값**이다. (판정은 뒤집히지 않는다 — 0.647로 재계산하면 ② runClear(random) = 0.300 ≥ 0.20 ✔, ③ Δ = 0.347~0.647 ≥ 0.20 ✔. 그러나 「12계열 전수 재검산」이 2계열에서 자기모순 입력을 썼고, 밸런서가 Πs의 목표를 두 개 받는다.)

7. 【§8.10의 「새떼 = 스테이지 총 XP의 ~25%」를 어떻게 할 것인가】 v1.1이 04에게 「웨이브 총 개체 수를 늘려라」고 명령해 04가 이행하자 새떼 지분이 **12~13%로 반토막**났다. 내 계산: **S22(≤0.30)의 상한까지 올려도 총 XP 대비 18.2%가 천장**이므로 ~25%는 도달 불가능하다. 선택: ⓐ ~25%를 폐기하고 실측(~13%)을 정본에 기입 ⓑ `crisisTotal`(정본 소유)을 60 → 90 이상으로 올리고 S22를 재계산 ⓒ S22의 분모를 「저작 리스트 XP」에서 「스테이지 총 XP」로 바꿔 두 수를 같은 자로 잰다(정본 §13.1.1의 「분모가 없으면 지표가 아니다」 원칙과 정합).
