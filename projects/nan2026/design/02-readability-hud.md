# 가독성 · 비주얼 언어 · HUD — 섹션 v1.2 (정본 v1.2 정합)

> **지위**: 이 문서는 `design/CANON.md`(**정본 v1.2**)의 **하위 섹션**이다. 정본의 값·공식·어휘·스키마는 **인용만** 하며 재정의하지 않는다(C-1·C-2). 이 문서가 정본과 충돌하면 **이 문서가 틀린 것이다.**
> **위임 범위** (정본 §17, v1.2 판본): 「HUD」 = **패널 내부 레이아웃 · 폰트 사다리 · 아이콘 9종 · 글리프 LOD · 강등 사다리의 값**. 다음은 **정본이 고정**한다 — 요소 목록 · 순서 `Q W E R` · 기체 부착 규격 · **§1.2 띠 내용** · **§7.12 경계면 8건** · ★ **`palette`·`visual` 전 키(§9.4.3)**.
> ★ **v1.2에서 이 섹션은 JSON 키를 하나도 소유하지 않게 되었다**(§11). 정본 §21의 인쇄 감사가 02의 채택 8키 + 미심사 신규 1건에 **전부 인쇄 자리를 주었기 때문**이다 — 위임된 것은 이제 **값이 아니라 좌표**이고, 좌표는 JSON이 아니라 이 문서의 표에 산다(정본 §7.12.5가 명문화).
> **단위**: 모든 시간은 **게임초(`Sec`)**, 모든 거리는 **논리 픽셀**(정본 §0.2). 실제ms는 **정확히 4곳**이며 이 섹션은 그중 `render.fxMinRealMs` · `render.degradeOnFrameMs` 둘을 인용한다.
> **틱은 JSON에 등장하지 않는다**(정본 §0.2) — 이 섹션의 v1.0이 제안했던 `visual.telegraph.fireFlashTicks`는 **그 규약 위반이므로 스스로 폐기**한다(§3.3).

## 2. 진영 배타 규격 — "이건 누구 것인가"를 0ms에 판정

### 2.1 집행 규칙 (정본 §7.4 표의 실행형)

정본 §7.4가 7채널(색·코어·엣지·합성·알파·레이어·모양) 전부를 배타로 정의했고, §7.12.6이 **플레이어 텔레그래프**를 그 배타 안으로 끌어들였다. 이 섹션은 **그 배타가 코드에서 깨지지 않게 하는 규칙**을 더한다.

| # | 규칙 |
|---|---|
| **V-1** | **팔레트 접근 경로가 진영을 강제한다.** 플레이어 측 렌더는 `palette.threat.*`를, 적 측 렌더는 `palette.element.*`를 **읽지 않는다.** 예외는 정확히 둘: ① 적 본체의 속성 외곽선·글리프·림(정본 §7.6) — 이것은 "내가 무엇으로 때려야 하는가"의 정보이지 위협 표시가 아니다 ② **적 탄의 호박 테두리**(정본 §7.12.4-⑤) — 이것은 `palette.status.*`이며 `element`도 `threat`도 아니다. |
| **V-2** | **하드 외곽선의 유일성.** 검은 외곽선(`palette.threat.outline`)을 가진 것은 **전부 나를 아프게 한다.** 역도 참이다 — 나를 아프게 하는 것 중 외곽선이 없는 것은 존재하지 않는다(적 탄 / 활성 장판 / 레이저 / 몸통 = 적 본체 실루엣). = I-1의 쌍방향 형태. **정본 §7.12.6이 이것을 플레이어 측에서 확인했다** — 플레이어 텔레그래프는 외곽선 **금지**. |
| **V-3** | **꼬리 = 유도.** 잔상 꼬리는 `turnRateDegSec > 0`인 개체 **전용 표식**이며 진영을 가리지 않는다(적 유도탄에도, 플레이어 `seeker` 탄에도 붙는다). 유도가 아닌 어떤 탄도 꼬리를 갖지 않는다. → **꼬리 = 쫓는다, 색 = 누구 것인가.** 두 축이 직교하므로 조합 학습이 불필요하다. ★ **규격 = `visual.trail`(정본 §9.4.3: `ghostCount` 3 · `ghostAlpha` 0.30), 간격 = 직전 3틱 = 구조.** V-3 자체는 **규칙이지 값이 아니다** — 토글 키를 만들지 않는다(§8.3). |
| **V-4** | **크기 하한이 진영을 보조한다.** 적 탄 `fairness.minBulletRadiusPx = 4`, 플레이어 탄 `render.playerBulletMaxRadiusPx = 10`, 픽업 ≤ 6px(정본 §7.8). 겹치는 구간이 존재하나 **운동·레이어·렌더모드가 이미 분리**하므로 크기는 보조 채널이다. |

> **V-2의 값어치**: 이 한 줄이 `mono` 모드(전 채도 0)에서 **위협 분리를 단독으로 보증**한다(§9.3). 그러므로 I-1은 색맹 모드에서 **협상 불가**다.
> **★ V-3의 강등 하 지위 (정본 §9.4.2 사다리 ④와의 관계)**: 사다리 ④가 잔상을 끄면 꼬리가 사라진다. **정보 손실이 아니다** — 정본 §7.8이 유도의 판독을 **운동**에 걸었기 때문이다("천천히 곡선으로 쫓아오면 100% 탄"). 꼬리는 그 운동의 **리드타임 증폭기**이지 유일 채널이 아니다. 사다리는 하한기에서만 작동하고 정본의 불가침 목록(히트박스 도트·스탠스 링·적 외곽선·글리프·텔레그래프·적 탄)을 건드리지 않는다.

### 2.2 적 탄 실루엣 — 어휘는 둘뿐 (정본이 폐쇄했다)

정본 §7.4: 적 탄 모양 = **원 / 육각(상태이상)**. 정본 §9.7: **`bullets[].shape` 어휘 = `["circle", "hex"]` (폐쇄)** — 02-B4 채택.

| `shape` | 조건 | 부가 |
|---|---|---|
| `circle` | `status === null` | 유도(`turnRateDegSec > 0`)면 **꼬리 3장**(V-3) |
| `hex` | `status !== null` | 크기 ×1.5(slow) / ×1.8(stun), 속도 `visual.statusBulletSpeedMul = 0.6`, 흰 점선 궤적(slow) / 흰 이중 링 4Hz(stun) — 전부 정본 §7.11. **+ 호박 테두리**(정본 §7.12.4-⑤) |

- ★ **3중 동치 (정본 §7.12.4가 02-B4를 확장했다)**: `(status ≠ null) ⟺ (shape == "hex") ⟺ (호박 테두리)`. `check.mjs` **S14**가 `(status === null) === (shape === "circle")`을 강제하고, 어긋나면 **색이 거짓말하는 것**(I-2)이므로 **로드 실패**가 옳다. 02의 v1.0 요청은 2중 동치였고 **정본이 한 겹 더 쌓았다.**
- **호박 테두리의 적층 순서 (I-1과의 정합)**: I-1은 불변식이므로 **검은 하드 외곽선 2px는 삭제 불가**다. → 바깥부터 **검은 2px → 호박 1px → 자홍 채움 → 흰 스페큘러 코어**. 호박이 검은 외곽선을 **대체하지 않고 안쪽에 얹힌다.** 이 순서 외의 어떤 배치도 I-1을 깬다.
- ★ **초안 B의 "유도탄 = 마름모" · "설치 장판 = 링"은 폐기한다.** ◆는 **물 속성의 예약 글리프**다(정본 §7.2). 유도는 **꼬리**로 읽는다(V-3).
- **레이저**: 탄이 아니라 이미터(정본 §8.5, `widthPx`). 실루엣 = 자홍 채움 라인 + 흰 코어 + 검은 외곽선 2px, 레이어 9.
- **활성 장판(`zone`)**: §3.4 — **정본 §12.3이 v1.1에서 알파 모순을 닫았다.**

### 2.3 글리프 예약 — 색과 같은 강도로 예약된다

| 규칙 |
|---|
| **속성 글리프 4종(● ▲ ◆ ✚)은 속성 주장 전용이다.** 아이콘·마커·불릿포인트·장식으로 쓸 수 없다. 자홍 개체·픽업·HUD 크롬·버튼·커서 어디에도 등장하지 않는다. |
| **예외 1 — 무채색 pip.** `●`/`○`/`⊘`는 **수량·상태 표시**로 허용한다(투자 pip, 소지품 pip, 패시브 레벨 pip, **무피격 pip 6칸**). 속성색이 아니므로 주장이 아니다. |
| **예외 2 — 은색 `●` = 노말.** 정본 §7.5의 "나머지 = 은색 ●"은 **정확한 주장**이다: 부여 안 됨 = ×1 = 노말. I-2 준수. |
| **★ 은색의 정의 — 정본이 v1.1에서 직접 답했다.** 정본 §7.12.1이 타이머 정상 색을 **`palette.element.normal` (은색)**이라고 **키와 이름을 나란히** 적었다. → **은색 ≡ `palette.element.normal` (`#DCE6F0`)**. 이 섹션의 v1.0 추론(은색이 등장하는 전 지점에서 의미가 정확히 "×1"이므로 별도 `palette.silver`는 I-2를 깬다)은 **정본이 같은 결론에 도달함으로써 인용으로 강등된다.** |
| **★ 은색 hex의 3중 등장은 충돌이 아니다 (v1.2 확인).** 정본 §9.4의 인쇄본에서 `element.normal` · `hud.textPrimary` · `hud.hpFill`이 **전부 `#DCE6F0`**이다. **I-2 위반이 아니다** — I-2가 금하는 것은 「**속성색**이 그 개체가 적용할 배율을 거짓말하는 것」이고, 패널 **텍스트**와 **HP 바 채움**은 속성을 주장하지 않는다(글리프도 pip도 아니다). 오히려 **한 hex인 것이 옳다**: 은색 = "주장 없음/×1"이 이 게임의 **중립 밝은 무채색**이며, 세 표면이 그 하나를 공유하는 것이 §7.2 대원칙(「hue는 속성 전용 예약 채널」)의 준수 그 자체다. **새 색 0.** |

| **✔ / ✕ / ◈의 지위**: `✔`·`✕`(무피격 pip·퍼펙트·시간 보너스)는 속성 글리프가 아니며 **속성색으로 그리지 않는다**(정본 §7.12.5: 은색 / `palette.neutralGray`). ★ **v1.0의 패널 ASCII에 있던 진화 표식 `◈◈`는 삭제한다** — ◆(물)와 형태가 근접해 예약의 정신을 흐린다. 진화는 **이중선 테두리**가 단독으로 나른다(§7.3-B). |

## 3. 텔레그래프 언어

### 3.1 정본이 이미 닫은 것

2상태(점선 = 미래 / 채움 + 검은 외곽선 = 현재) · 전환 순간 흰 1프레임 플래시 · 거동별 모양 8종 + 상황 4종 · 리드타임 하한(`straight` 0.55 ~ `bossPattern`·`stun` 1.50) · `fairness.minTelegraphSec = 0.55` · `telegraphConcurrentMaxPerEntity = 2` · **`fairness.telegraphConcurrentMaxGlobal = 80`(A층, 파생: 40 × 2)** · **`caps.telegraphs = 96`(B층) + `deferAttack`** · `emphasisBySpeed`(렌더 전용) · 레이어 1(지면) / 8(공중) · 알파 0.50. **전부 정본 §7.4 · §12.1 · §12.3 · §12.4.**

> ★ **v1.0의 이 섹션은 "전역 캡 8"이라고 적었다. 정본 v1.1이 그것을 폐기했다** (04-R4 수정채택): v1.0 정본은 「A 2/개체 < B 8」로 **단위가 다른 두 수를 부등호로 이었다.** 정정 = **A층 80(파생) < B층 96(20% 여유)**. 이 섹션의 §8.1 표도 함께 고쳤다.

### 3.2 ★ 진행률 채널 (정본 §19.1-B3 **채택** → v1.2에서 **값이 정본으로 확정**, §9.4.3)

**문제**: `laser`의 리드타임은 1.20 게임초다. 점선이 1.2초 동안 그냥 있으면 플레이어는 "곧"은 알지만 **"언제"**를 모른다. Normal에서 1.2초는 여유지만 Disaster(×3)에서 실제 0.4초다 — 그때 필요한 것은 강조가 아니라 **카운트다운**이다.

> **정본의 채택 근거 (§19.1-B3)**: *"I-3(「텔레그래프 없이는 발사 없다」)은 **리드타임의 존재**만 보장하고 **남은 시간의 판독**을 보장하지 않았다. `dashTightenAtPct` = 형상 무관 백스톱."*

**규칙**: **모든 텔레그래프는 진행률을 형상으로 인코딩한다.** 채널은 2개이며 항상 둘 다 작동한다.

**① 형상 고유 채널** — 정본이 이미 4종에 넣어 두었고, 나머지 4종을 여기서 지정한다.

| 이미터 | 정본 §7.4가 지정한 모양 | 진행률 = |
|---|---|---|
| `straight` | 발사구 자홍 원 **확대** | 반경 (정본 내장) ✔ |
| `aimed` | 조준 브래킷 4각 **수축** | 수축량 (정본 내장) ✔ |
| `zone` | 지면 점선 원 **수축** | 수축량 (정본 내장) ✔ |
| `laser` | 점선 라인 → **폭 0→최종폭 보간** | 폭 (정본 내장) ✔ |
| **`fan`** | 점선 부채꼴 | ★ **호가 중심에서 바깥으로 자라남** (길이 = 진행률). 각도는 처음부터 실제 각도 고정 |
| **`ring`** | 발사구 점선 원 | ★ **반경 0 → 최종 반경 확대** (`straight`와 동일 규칙 = 학습 이월) |
| **`spiral`** | 점선 원 + 회전 화살 | ★ **화살이 1회전을 완주하면 발사** (진행률 = 각도) |
| **`wall`** | 점선 수평선 + 틈이 미리 비어 있음 | ★ **좌우 양끝에서 중앙으로 선이 그려지고, 완성되면 발사**. 틈은 **처음부터** 비어 있다(정본 규정 유지) |

**② 전역 dash 백스톱** — 형상과 무관하게 항상 작동한다. ★ **값은 정본 §9.4.3이 소유한다 (v1.2, 02의 제안값은 수정채택)**.

```
리드타임의 0 ~ 70% 구간:    dash = visual.telegraph.dashPx = 6          (dash 6 / gap 6)
리드타임의 70 ~ 100% 구간:  간격이 dashTightenMul(0.5)까지 선형 수축  → gap 6 → 3
발사 순간:                  흰색 플래시 → 채움 (정본 §7.4의 2상태)
```

> **정본의 문장 (인용, §9.4.3)**: *"`dashTightenAtPct`(**0.70**) = 형상 무관 백스톱 — **진행률 70%부터 dash 간격이 ×0.5로 조여든다.**"*

★ **v1.2에서 이 섹션이 승복한 3건** — 02는 `dashPx: [6, 5]` · `dashTightenAtPct: 0.85` · `dashTightenMul: 0.35`를 제안했고, **정본은 `6` · `0.70` · `0.5`로 확정했다.** 승복하고 본문 산술을 다시 계산한다.

| 축 | 02의 제안 | **정본 확정** | 이 섹션의 재검산 |
|---|---|---|---|
| `dashPx` | `[6, 5]` (dash/gap 쌍) | ★ **`6`** (스칼라) | **정본이 옳다.** 02는 쌍으로 dash≠gap을 만들려 했으나, **gap이 조여드는 것이 이 규칙의 전부**이므로 기준 gap은 dash와 같아야 「조임」이 **비율로** 읽힌다. 스칼라 하나가 `setLineDash([6, 6 × k])`를 낳고 `k`가 곧 진행률의 함수다 — **자유 숫자가 2개에서 1개로 줄었다.** |
| `dashTightenAtPct` | 0.85 | ★ **0.70** | **정본이 옳다.** 02의 0.85는 **최악 케이스에서 소실된다**: 하한 리드타임 `straight` 0.55 게임초 × Disaster(×3) = **실제 0.183초**. 그중 15% = **27.5ms = 1.65 프레임** → 조임이 **2프레임 안에 끝난다 = 안 보인다.** 0.70이면 **55ms = 3.3 프레임**이고, 이것은 `render.fxMinRealMs`(50)가 「순수 장식의 소실 하한」으로 이미 못박은 **바로 그 50ms 스케일**이다 → **정본의 두 수가 서로를 지지한다.** ★ **02는 0.85를 「끝에 가까울수록 긴박」이라는 감각으로 골랐고 배속을 곱해보지 않았다.** |
| `dashTightenMul` | 0.35 | ★ **0.5** | **정본이 옳다.** 0.35면 gap 6 → **2.1px**이고, `visual.telegraph.strokePx = 2`이므로 **간격(2.1)이 선 두께(2)와 같아진다 = 점선이 실선으로 보인다** → 정본 §7.4의 2상태 규칙(「점선 = 미래 / **채움 = 현재**」)이 **발사 전에 거짓말한다**(I-2). 0.5면 gap 3 > strokePx 2 → **끝까지 점선으로 읽힌다.** ★ **02가 자기 §3.2에서 「채워지지 않으므로 2상태가 유지된다」고 논증해 놓고, 자기 값이 그 논증을 깨고 있었다.** |

> **왜 이 규칙이 옳은가 (근거는 유지)**: 점선이 실선에 **근접**할 뿐 채워지지 않으므로 정본의 2상태 규칙("채움 금지")이 그대로 유지되고, 동시에 **"점선이 조여들면 발사 직전"**이라는 연속 판독이 8종 전부에 공짜로 붙는다. 규칙 1개, 새 색 0, 새 형태 0, 색맹 안전(형태 채널). `emphasisBySpeed`(굵기·알파)와 직교하므로 고배속에서 두 채널이 **겹쳐서 강해진다.**

### 3.3 텔레그래프 규격 — 인쇄 자리는 정본 §9.4.3 (★ `fireFlashTicks` 자진 폐기)

**`visual.telegraph`의 필드 집합 = 정본 §9.4.3이 확정한다 (인용, 재정의 금지)**

| 키 | 값 | 이 섹션과의 관계 |
|---|---|---|
| `strokePx` | **2** | 인용 |
| `dashPx` | **6** | §3.2 (02 제안 `[6,5]` → 수정채택) |
| `dashTightenAtPct` | **0.70** | §3.2 (02 제안 0.85 → 수정채택) |
| `dashTightenMul` | **0.5** | §3.2 (02 제안 0.35 → 수정채택) |
| `airAlpha` | **0.50** | 공중 텔레그래프(레이어 8) — 정본 §12.3. **위기 예고 자홍 점선도 이 값**(§7.12.2) |
| `fillAlpha` | **0.50** | §7.4의 「내부 알파 0.5」의 거처 |
| `emphasisBySpeed` | **{1.0:1.0, 1.5:1.15, 2.0:1.30, 3.0:1.50}** | 렌더 전용, 인용 |

> ★ **v1.2 정정**: v1.1의 이 섹션은 *"alpha 0.50은 정본 §7.4·§12.3 소유 — 여기서 재인쇄하지 않는다"*고 적으며 **알파를 키 없는 산문으로 남겨 두었다.** 정본 §21-A7의 감사가 그것을 **`visual.telegraph.airAlpha`·`fillAlpha` 두 키로** 인쇄했다 — **「정본이 소유한다」는 「거처가 있다」가 아니었다.** 이 섹션이 놓친 것이며, 정본의 감사가 옳았다.

- ★ **v1.0의 `fireFlashTicks: 1`은 폐기한다.** 정본 §0.2: **"틱은 JSON에 등장하지 않는다. 틱(1/60 게임초)은 엔진 내부 단위이며 데이터 어휘가 아니다. `*_ticks` 키는 전부 폐기한다."** → **`fireFlashTicks`는 그 규약의 정면 위반이었고, 정본이 그것을 지적하지 않은 것은 이 키가 요청 대장에 `visual.telegraph.*` 묶음으로만 올라갔기 때문이다. 이 섹션이 스스로 거둔다.**
- **대체 = 구조 (새 키 0)**: **발사 플래시는 판정 1틱**(엔진 내부 단위, JSON 키 없음)이고, **표시는 `render.fxMinRealMs = 50`(정본 §9.4.2) 하한을 탄다** → **Disaster(×3)에서도 발사 순간이 소실되지 않는다.** 정본 §7.4의 "흰색 1프레임 플래시"가 판정 1틱인지 렌더 1프레임인지의 모호함은 **키가 아니라 이 한 줄이 닫는다.** (`fxMinRealMs`의 정본 근거 문장이 정확히 이것이다: *"순수 장식이 고배속에서 소실되지 않게(판정 무관)"* — §6.2.)

### 3.4 활성 장판 — ★ 정본 §12.3이 닫았다 (02-A2 **채택**, blocker 해소)

v1.0의 모순: 정본 §7.4("지금 아픈 것 = 자홍 채움 + **불투명** + 검은 외곽선") vs 정본 §12.3(레이어 1 = "**알파 ≤0.35**") → **평가 불가.**

**정본 v1.1 §12.3의 확정 (인용)**:

| 레이어 1의 요소 | 확정 알파 |
|---|---|
| 띠의 **배경 판** | `visual.band.plateAlpha` = **0.30** |
| 띠의 **내용** (타이머·HP 세그먼트·XP 바·라벨·코어 HP 바) | **불투명** (`visual.band.contentOpaque = true`) |
| 지면 **텔레그래프** | 점선 외곽선 알파 1.0 + 내부 채움 `visual.zone.fillAlpha` **0.30** |
| **활성 장판** | **외곽선 불투명 + 검은 외곽선 + 내부 채움 0.30 + `visual.zone.pulseHz` 1.0 맥동** |

> **정본의 근거 (인용)**: *"「알파 ≤0.35」의 진짜 목적은 「지면이 게임플레이 오브젝트와 경쟁하지 않게」였다. 레이어 1은 모든 게임플레이 오브젝트 아래이므로 **불투명해도 아무것도 가릴 수 없다** — **I-4는 알파가 아니라 레이어 구조가 보장한다.** v1.0은 같은 목적을 두 장치(레이어 + 알파)로 중복 집행하다가 §7.4와 충돌했다."*

**이 섹션의 처리**: v1.0이 제안했던 `visual.zone.outlinePx: 2` · `outlineAlpha: 1.0`은 **삭제한다** — 외곽선 2px는 `palette.threat.outline`의 정의(정본 §7.2: "하드 외곽선 `#0A0C10` **(2px)**")가 이미 소유하고, 불투명은 정본 §12.3이 확정했다. **파생 가능한 값에 키를 만들면 두 값이 갈라진다.** 남는 것은 정본이 소유한 `fillAlpha` · `pulseHz` 둘뿐이다.

## 6. 스탠스 판독

### 6.1 정본이 이미 닫은 것 (기체 측)

### 6.2 ★ 투자 pip = 슬롯 지도 (패널 측의 핵심)

정본 §4.2: `elementCapPerElement = 4` = **무기 슬롯 수**(`player.weaponSlots = 4`) (LOCKED).
정본 §11.1 유효성 필터: **"속성 레벨 ≥ 현재 무기 수 → 그 속성 카드 제외 — 죽은 투자 차단"** → `투자 레벨 N ≤ 무기 수`가 **항상 참**이다(무기는 절대 잃지 않으므로 단조 — 정본 §11.1: "무기 교체를 거부하는 이유").

**귀결**: 좌 패널의 4칸 투자 pip은 **4칸 무기 슬롯과 1:1이며, pip을 그대로 슬롯 지도로 읽을 수 있다.**

```
W 불  ▲  ●●○○      ← "불 스탠스를 켜면 슬롯 1·2가 불로 발사된다"를 그대로 읽는다
E 물  ◆  ●○○○      ← "물 스탠스를 켜면 슬롯 1만 물"
R 풀  ✚  ○○○○      ← "풀 스탠스는 전부 무속성" (= 탈색 링이 뜬다)
```

- **부여 미리보기를 따로 그리지 않는다.** 투자 pip과 정보가 **동형**이기 때문이다. 4행 × 4pip = **부여 행렬 전체가 상시 화면에 있다** — 키를 눌러보지 않고도 4개 스탠스의 결과를 전부 안다.
- **3상태 pip**: `●` 채움 = 부여됨 / `○` 빈 원 = 슬롯은 있으나 투자 없음 / **`⊘` 점선 원 = 그 슬롯 자체가 아직 없음**. 세 번째 상태가 초반(무기 1~2개)에 "칸이 왜 4개인가"를 말없이 가르친다.
- **혼동 방지**: 투자 = **무채색 도트**(●/○/⊘), 부여 = **속성 글리프 칩**(▲/◆/✚) 또는 은색 `무`. 같은 화면에 두 종류의 "칸"이 있지만 **형태 어휘가 다르다.**

### 6.3 1:1 대응 강제

| 표면 | 규칙 |
|---|---|
| 기체 스트립 pip *i* (레이어 6) | ↔ 좌 패널 무기 슬롯 *i*의 부여 배지 |
| 좌 패널 스탠스 행의 투자 pip | ↔ 그 스탠스를 켰을 때의 스트립 |
| 히트박스 링 색 (레이어 10) | ↔ 좌 패널 `▶` 마커가 가리키는 행 |
| **어긋나면** | **구현 버그**(I-2). 세 표면은 같은 하나의 상태를 읽는다. |

### 6.4 온보딩 훅의 구체화 (정본 §9.9의 3키)

| 정본 키 | 이 섹션의 규격 |
|---|---|
| `autoEquipFirstElement` | 속성 카드 확정 순간 → 그 스탠스 자동 장착 + **좌 패널 해당 행의 키캡 박스가 은색 → 속성색으로 0.4초 전환** + 행 하이라이트 이동. 투자 0→1 최초 전이에서만. **은색에서 출발하는 것이 아래 §7.12.7과 이어지는 지점이다.** |
| `stanceHintPulse` + `stanceHintPulseStageMax: 1` | **좌 패널 키캡 박스**가 맥동한다 — **자리는 이 섹션(§7.3-A)** / ★ **색·주파수·알파는 정본 §7.12.7** (아래). |
| **타겟의 정의** | ★ **정본 §9.9.1이 소유한다** (02-B8 **채택**). 인용: **아레나 안에 살아있는 적 개체 수 기준 최빈(mode) 속성의 `counter` 키.** 세는 대상 = 잡몹·엘리트·새떼 (**중간보스·보스·부위 제외** — 이들은 개체 위 HP 바 + 부위 글리프가 이미 답을 말한다). 타이 → `Q W E R` 순. 최빈이 노말이거나 개체 0 → **맥동 없음.** 키 = `hud.stanceHintTargetsMajorityElement`(§9.4.1). |

> ★ **정본이 v1.0의 이 섹션보다 정밀했다**: 02는 "아레나 내 살아있는 적 개체 수 기준 최빈"이라고만 했고, **무엇을 세는지**(중간보스·보스·부위의 포함 여부)를 정하지 않았다. 정본이 그 구멍을 닫았다.

#### ★ `stanceHintPulse`의 색 — 정본 §7.12.7 소유 (7번째 경계면, **02 기각**). 승복한다.

**v1.1의 이 섹션은 「그 속성색으로 1Hz 맥동」이라고 적었다. 정본 v1.2가 기각했다.**

| 채널 | **정본 §7.12.7의 확정 (인용)** |
|---|---|
| 색 — **투자 > 0** | 그 **속성색** (`palette.element[e]`) |
| ★ 색 — **투자 0** | ★ **은색(탈색)** = `palette.element.normal` |
| 주파수 | `visual.stance.hintPulseHz` = **1.0** ← **02의 값이 채택** |
| 알파 | `visual.stance.hintPulseAlpha` = **[0.40, 1.0]** ← **02의 값이 채택** |
| 범위 | `stanceHintPulseStageMax = 1` — 스테이지 1 전용 |

> **정본의 기각 근거 (인용)**: *"**스테이지 1 전용 기능이고 런 시작 투자는 0-0-0이다**(§2.6) → **힌트가 처음 뜨는 순간은 거의 항상 투자 0인 키**다. 02의 스펙대로면 **온보딩 첫 순간부터 화면이 거짓말한다** — I-2: 「화면에 그려진 속성색 = 그 개체가 **지금** 적에게 적용할 배율」. **투자 0인 키는 아무것도 적용하지 않는다.**"*

**이 섹션이 승복하는 이유 (자기 검산)** — 정본의 근거는 **이 문서 자신의 두 문장**에서 곧바로 유도된다.

| 이 문서의 문장 | 그것이 02의 v1.1 스펙에 대해 말하는 것 |
|---|---|
| §2.3 예외 2: *"**은색 `●` = 노말**은 **정확한 주장**이다: 부여 안 됨 = ×1 = 노말. I-2 준수."* | 그렇다면 **투자 0인 키를 속성색으로 칠하는 것은 정확히 그 반대**다 = **부정확한 주장.** 02는 같은 문서 안에서 은색의 정당성을 논증하고 4장 뒤에서 그것을 깼다. |
| §6.3: *"히트박스 링 색 ↔ 좌 패널 `▶` 마커 (…) **어긋나면 구현 버그**(I-2). 세 표면은 같은 하나의 상태를 읽는다."* | 정본 §7.5는 **투자 0 스탠스 = 은색 링**이다. 02의 스펙이면 **같은 키가 링에서는 은색인데 키캡에서는 속성색** = 02 자신이 「구현 버그」라 부른 그것. |

- ★ **02의 v1.1은 「경계면 6건」을 인용하면서 이 항목이 7번째라는 것을 보지 못했다.** 근거: 05도 같은 픽셀(좌 패널 키캡)에 스펙을 냈고, **양쪽 다 「새 색 0」을 표방했다.** 02의 「새 색 0」은 참이었으나(속성색은 이미 있다) **「새 거짓말 0」이 아니었다.**
- ★ **살아남은 것 = 값 2개.** `hintPulseHz` 1.0 · `hintPulseAlpha` [0.40, 1.0]은 **02의 값이 그대로 정본이 되었다**(거처 = `visual.stance`, §9.4.3). 정본 인용 — *"02는 1Hz/0.40↔1.0을, 05는 「`visual.stance` 어휘 재사용」을 말해 **둘 다 값을 확정하지 않았다** → 정본이 02의 값을 채택해 `visual.stance`에 편입한다."*
- ★ **서사가 좋아졌다 (정본이 옳은 세 번째 이유)**: **은색 맥동("이 키를 봐라, 그런데 아직 아무것도 없다") → 속성 카드 획득 → `autoEquipFirstElement`의 키캡 점등(은색 → 속성색)**이 **하나의 인과**가 된다. 02의 스펙이면 카드를 먹기 **전에 이미 속성색**이라 그 점등이 **아무것도 바꾸지 않는다** — 02는 자기 온보딩 훅 두 개를 서로 무의미하게 만들고 있었다.

**남은 것 3가지를 이 섹션이 닫는다.**

### 8.2 ★ 글리프 LOD의 정의 (02-A7/B2 **채택** → v1.2에서 **정본 §9.4.3이 인쇄**)

정본 §7.3은 `off`에서 "**글리프 렌더: LOD 적용**", `cvd`/`mono`에서 "**항상 렌더**"라고 대비시킨다 → **LOD는 실재하는 규칙인데 v1.0 어디에도 정의가 없었다.**

**정본의 채택 근거 (§19.1-A7 인용)**: *"크기 기준 → **팝핑 0**, 외곽선이 속성을 나르므로 **정보 손실 0**."*

**값 = `visual.glyph.lodMinBodyPx` = 16 (정본 §9.4.3 인쇄 — 02의 제안값이 그대로 채택됐다)**
**판정식 = 이 섹션 (§17 위임 = 「글리프 LOD」)**

```
글리프를 그린다  ⟺  ( cbMode !== "off"                        // cvd·mono는 무조건 (정본 §7.3)
                    ∨ 개체가 엘리트·중간보스·보스·부위        // 정본 §7.6 "상시 글리프"
                    ∨ bodyPx >= visual.glyph.lodMinBodyPx )   // 값 = 정본 / 이 식 = 이 섹션
                 ∧ !occluded                                   // 정본 visual.glyph.occludedSkip
```

> ★ **v1.2 정정 — 괄호가 없었다.** v1.1의 이 식은 `A ∨ B ∨ C ∧ D`로 적혀 있었고, `∧`가 `∨`보다 **결합력이 강하므로** 문면대로 읽으면 `A ∨ B ∨ (C ∧ D)` = **`cvd`·엘리트는 가려져도 글리프를 그린다**로 읽힌다. 정본 `occludedSkip = true`는 **모드·밴드 무관 전역 규칙**(§7.6: "글리프는 최상위 개체만 렌더")이므로 **`∧ !occluded`가 세 항 전부에 걸려야 한다.** 괄호를 넣어 닫는다 — **규칙 변경 0, 값 변경 0, 오타 정정 1.**

| | |
|---|---|
| **무엇이 걸리나** | `chaff` 밴드(정본 §9.7 저작 가이드: `radius` 12~16) · 새떼 — **가장 많고 가장 작은 것들** |
| **정보 손실 0인 이유** | 정본 §7.6: **속성색 외곽선은 "본체 크기 무관 항상"** 그려진다. `off` 모드에서 속성은 **외곽선 색이 이미 100% 나른다.** 글리프는 색맹 이중화 채널이고, 색맹 모드에서는 LOD가 꺼진다. → **어떤 모드에서도 정보가 사라지지 않는다.** |
| **왜 이 값인가** | `chaff` 밴드 상한 16 = 경계. `line`(14~18)·`turret`(18~24)·`bruiser`(24~32)는 대부분 넘는다. **`elite.sizeMult = 1.4`는 `radius`에 곱해지고**(정본 §9.7, 04-R17: *"`radius`가 유일 진실"*) 엘리트는 어차피 규칙이 상시 렌더한다. |

| **위기 세션에서 특히 옳은 이유** | `crisisElementRule = "themePure"` → **새떼 전량이 같은 속성**이다(스테이지 4~6 = 60기, `crisisTotal` = 60 × `swarmTotalScale` = 30/42/51/60/60/60 — 정본 §8.3·§8.10). 개체별 속성 판독이 **애초에 불필요**하다. 판독해야 하는 것은 ① 덩어리의 가장자리 ② 적 탄 — 둘 다 글리프와 무관하다. **위기 세션의 가독성 예산은 전액 적 탄으로 간다.** |

### 8.4 패널 캐시

`hud.panelCacheDirtyOnly = true`(정본 §9.4.1) — 패널은 **값이 바뀔 때만** 오프스크린 캔버스에 갱신하고 매 프레임 `drawImage` 2회(좌·우)로 blit한다.

- 좌 패널이 갱신되는 순간: 스탠스 전환 · 투자/무기/패시브 획득 · 슬롯 재정렬 · `stanceHintPulse` 프레임(스테이지 1 한정 — 이때만 매 프레임 갱신).
- 우 패널: 점수 · 코인 · 스톡 · 스테이지 · **무피격 pip 상태** 변화.
- ×3 배속에서 한 프레임에 최대 `loop.maxStepsPerFrame`(6)틱을 돌리므로(정본 §6.1) **렌더 예산이 가장 빡빡한 지점**이며, 패널 2장을 매 프레임 새로 그리는 것은 그 예산의 순손실이다.
- `cbMode`·`reduceFlash` 변경 시 **양쪽 캐시 즉시 무효화** (OPTIONS는 TITLE·PAUSE 양쪽에서 열린다 — 정본 §5.5 · §14).

### 8.5 겹침

- **적 겹침**: 속성색 외곽선이 각 개체를 분리(정본 §7.6) + `glyph.occludedSkip`. 새떼는 §8.2의 근거로 개체 판독이 비-목표다.
- **적 탄 겹침**: 검은 하드 외곽선 2px(cvd 3px) + 흰 스페큘러 코어 + 불투명 → **탄 20개가 뭉쳐도 각 탄의 경계가 남는다.** 이것이 `additive 금지`의 이유다(additive면 뭉치는 순간 한 덩어리가 된다).
- **픽업 누적**: 정본 §7.8 — 아레나 하단 경계에서 정지·누적. `caps.pickups = 256` 초과 시 `merge`(손실 0 = 무-노가다 기둥 보존). **누적 더미는 무광·≤6px·레이어 2**이므로 그 위 모든 것이 읽힌다.
- **HUD ↔ 아레나**: 물리적으로 겹치지 않는다(1280 = 350 + 580 + 350).

### 9.1 정본이 이미 닫은 것

### 9.2 ★ `mono`의 정의역 — **정본 §7.3이 소유한다** (판정 11, **02-R3 채택**). 인용만 한다.

**v1.1의 이 섹션은 「전 팔레트 무채화」를 *읽기*로 제시하고 정본 한 줄 명문화를 요청했다(R3, minor). 정본 v1.2가 채택하고 소유했다 — R3는 닫혔다.**

> **정본 §7.3의 확정 (인용)**: *"`mono` = **`elementCvd`를 기저로 `palette` 전체의 채도를 0으로 만든다** — `element`뿐 아니라 **`threat` · `status` · `pickup` · `hud` · `enemyBody` · `partDestroyed` · `bg` 전부.** 새 키 0, 새 색 0."*
> **정본이 02의 읽기를 채택한 근거 (인용)**: *"**02 §9.2는 전 팔레트로 읽었고 그 위에 §9.3의 위협 분리 증명을 세웠다.** (…) 그 증명은 **둘 다 무채색일 때만 필요하고 또 성립한다** — 자홍이 색을 유지하면 애초에 분리 문제가 없고, **증명도 무의미해진다.**"*

- ★ **정본이 02의 논거를 한 걸음 더 밀었다.** 02는 *"부분 무채화는 개발자에게만 거짓 안심을 준다"*고 썼다. 정본은 거기에 **자기모순 논증**을 더했다 — 부분 무채화를 택하면 **§9.3의 증명이 필요 없어지는 동시에 무의미해진다**(자홍이 색을 유지하면 물 탄과 애초에 안 겹친다). 즉 **02의 §9.3이 존재한다는 사실 자체가 「전 팔레트」의 증거**였다. **한 섹션의 증명이 다른 절의 정의역을 결정한 사례**이며, 그래서 이것은 「읽기」가 아니라 처음부터 **정본의 문장이어야 했다.**
- **`bg`가 목록에 있는 것에 유의한다**: 배경 상한은 `palette.bg`가 소유하므로(정본 §9.4.3: *"`visual.bg`는 존재하지 않는다"*) **`mono`의 무채화 대상에 배경도 든다** → §9.4-7(배경 L\*24 vs 위협 L\*57 = 33 차이)은 **두 색이 모두 회색이 된 뒤에도** 성립한다. 이 섹션의 검산은 애초에 그렇게 되어 있었다.

### 9.3 ★ `mono`의 위협 분리 — I-1이 단독 보증한다

**정직한 산술** (L\*는 sRGB → CIE L\* 실계산):

```
elementCvd 사다리:  normal L*96 │ fire L*76 │ water L*55 │ grass L*40   (정본 §7.3)
위협(자홍 #FF2E88):                              L*57  ← water L*55와 2 차이
배경 상한 (HSL L 0.22):                          ≈ L*24
```

> ★ **v1.0의 이 섹션은 자홍을 L\*54로 적었다. 재계산 결과 L\*57이다** (Y = 0.2126×1.0 + 0.7152×0.0273 + 0.0722×0.2463 = 0.2499 → L\* = 116·0.2499^⅓ − 16 = **57.1**). **결론은 강화된다** — 차이가 1이 아니라 2여도 `elementCvd` 자신의 기준(인접 ≥15)에 **13 미달**이다.
> ★ **정본의 사다리 4색도 전부 재검산했다** (sRGB → CIE L\*, D65): `normal #F2F6FA` = **96.7** / `fire #FFA83A` = **75.8** / `water #2E86E0` = **55.1** / `grass #1F6B3A` = **39.8**. 정본 표기(96 / 76 / 55 / 40)와 **fire·water·grass는 일치**하고 normal만 **반올림 차 0.7**이다(96.7 → 정본 96). **정본의 계약(인접 ≥15 L\*)은 전 구간에서 성립한다**: 96.7−75.8 = **20.9** / 75.8−55.1 = **20.7** / 55.1−39.8 = **15.3**. ★ **마지막 구간이 15.3으로 계약에 0.3만 남기고 붙어 있다** — `elementCvd`의 4색 중 **어느 하나라도 값을 바꾸면 이 계약이 먼저 깨진다.** 이 섹션은 그 사실을 기록만 하고 값을 만지지 않는다(팔레트는 정본 §7.3 소유).

**즉, `mono`에서 적 탄과 물 속성 플레이어 탄은 명도로 구분되지 않는다.**

**자유 슬롯이 없다는 증명**: 사다리가 40~96을 점유하고 배경이 ≤24를 점유한다. 남은 구간은 24~40뿐이며 그 중앙(L\*32)은 `grass`(40)와 **8 차이**, 배경(24)과도 **8 차이** — 사다리 자신의 기준(≥15)에 **양쪽 다 미달**이다. **`mono`에서 위협에 줄 수 있는 명도는 존재하지 않는다.**

**그러므로 분리는 명도가 아니라 다음이 한다** (정본 §7.4의 7채널 중 6개가 살아있다):

| 채널 | 적 탄 | 플레이어 탄 |
|---|---|---|
| **엣지** | **검은 하드 외곽선 3px** (cvd/mono) | **없음** (I-1) |
| 코어 | 흰 스페큘러 점 | 같은 회색의 밝은 코어 |
| 합성 | 불투명 | additive |
| 알파 | 1.0 | ≤ 0.80 (`render.playerBulletMaxAlpha`) |
| 레이어 | 9 | 4 |
| 모양 | 원 / 육각 | **글리프만** (정본 §7.3의 `mono` 행) |

> **결론: `mono`에서 위협 분리를 하는 것은 I-1이다.** 어두운 배경(L\*24) 위의 **검은 3px 테두리**는 그 자체로 L\*≈4의 국소 후광이며, 플레이어 측 어디에도 존재하지 않는다. **그러므로 I-1은 색맹 모드에서 협상 불가다** — "플레이어 탄에 외곽선을 조금 넣으면 예쁘겠는데"는 **접근성 기능 하나를 통째로 삭제하는 변경**이다. §2.1-V-2가 이것을 쌍방향으로 못 박고, **정본 §7.12.6이 플레이어 텔레그래프에 대해 같은 결론을 독립적으로 냈다**(*"외곽선 금지 (I-1)"*).

### 9.5 ★ `cvd`의 기체 스트립 오버플로 (02-A8/B2 **채택**)

정본 §7.3: `cvd` → **글리프 크기 ×1.5**
정본 §7.5: 슬롯 스트립 pip = **`visual.stance.pipPx = 7`**

→ 7px pip 안에 ×1.5 배율의 글리프를 그릴 수 없다. **정본의 두 값이 서로를 초과했다.**

**정본의 채택 근거 (§19.1-A8 인용)**: *"7px pip에 `cvd` ×1.5 글리프가 안 들어간다 → **색맹 사용자에게 슬롯 스트립(= 시그니처 시스템의 유일한 판독 채널)이 사라진다.**"*

**해소: `visual.stance.pipPxCvd = 10`** (`cbMode !== "off"`일 때 `pipPx` 대신 사용) — ★ **v1.2에서 정본 §9.4.3이 인쇄했다. 02의 값 그대로이며, 이제 02는 인용한다.**

> ★ **v1.1의 채택은 로드 실패였다 (정본 §21-A7 인용)**: *"§19.1이 이미 「채택」이라 심사했으나 **인쇄 자리가 없었다.**"* — `visual`이 `"..."`로 축약되어 있었으므로 C-7의 확정력이 미치지 않았고, 02가 이 키를 선언하면 **S2가 미지 키로 죽였다.** **「채택」은 인쇄가 아니다**(C-9). 이 섹션이 v1.1에서 R2로 올린 것이 정확히 그것이고, 정본이 그 클래스 전체(17건)를 닫았다.

```
off:  pip 7  × 4 + 간격 2 × 3 = 34px   (정본 §7.5 그대로)
cvd:  pip 10 × 4 + 간격 2 × 3 = 46px
```
- 기체 스프라이트 반경 `player.spriteRadius = 14` → 지름 28px. 46px 스트립은 기체보다 넓지만 **기체 하단 +14px, 레이어 6**이므로 적 탄을 가리지 않는다(I-4 안전). 이동 영역 좌우 인셋 20px(정본 §1.1) → 스트립 좌우 23px가 아레나 밖으로 3px 나가는 경우가 있다 → **스트립은 아레나 rect로 클리핑**한다(pip이 잘려도 순서·색은 읽힌다).
- **`cvd`에서 스트립이 안 읽히면 정본 §7.5의 3중 표시 중 하나가 죽고, 그 순간 "슬롯 순서 부여"라는 이 게임의 시그니처 시스템이 색맹 사용자에게 존재하지 않게 된다.** 3px는 그 대가로 싸다.

### 9.6 텍스트 라벨 강제의 적용 지점 (`cbMode !== "off"`)

| 표면 | `off`에서도 텍스트 있음? | `cvd`에서 추가 |
|---|---|---|
| 좌 패널 스탠스 행 | ✔ (`노말`/`불`/`물`/`풀`) | — |

| 우 패널 혼합 바 | ✔ (라벨 행 `불~70` 등) | — |
| 코어 HP 바 | ✖ | — (상단 띠의 **개체 이름**이 이미 텍스트다 — §9.4-10) |
