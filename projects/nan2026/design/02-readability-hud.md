# 가독성 · 비주얼 언어 · HUD — 섹션 v1.0

> **지위**: 이 문서는 `design/CANON.md`(정본)의 **하위 섹션**이다. 정본의 값·공식·어휘·스키마는 **인용만** 하며 재정의하지 않는다(C-1·C-2). 이 문서가 정본과 충돌하면 **이 문서가 틀린 것이다.**
> **위임 범위** (정본 §17): 「HUD」 = 패널 내부 레이아웃 · 폰트 · 아이콘. 정본의 요소 목록 · 순서 `Q W E R` · 기체 부착 규격은 **고정**이다.
> **이 문서가 새로 정한 값**은 전부 §11에 JSON으로 모아 두었고, 정본 편입 요청은 §12에 있다.
> **단위**: 모든 시간은 **게임초(`Sec`)**, 모든 거리는 **논리 픽셀**(정본 §0.2). 예외는 정본이 명시한 `Ms` 3곳뿐이며 이 섹션은 그중 `render.fxMinRealMs` 하나만 인용한다.

## 2. 진영 배타 규격 — "이건 누구 것인가"를 0ms에 판정

### 2.1 집행 규칙 (정본 §7.4 표의 실행형)

정본 §7.4가 7채널(색·코어·엣지·합성·알파·레이어·모양) 전부를 배타로 정의했다. 이 섹션은 **그 배타가 코드에서 깨지지 않게 하는 규칙**을 더한다.

| # | 규칙 |
|---|---|
| **V-1** | **팔레트 접근 경로가 진영을 강제한다.** 플레이어 측 렌더는 `palette.threat.*`를, 적 측 렌더는 `palette.element.*`를 **읽지 않는다.** 유일한 예외 = 적 본체의 속성 외곽선·글리프·림(정본 §7.6) — 이것은 "내가 무엇으로 때려야 하는가"의 정보이지 위협 표시가 아니다. |
| **V-2** | **하드 외곽선의 유일성.** 검은 외곽선(`palette.threat.outline`)을 가진 것은 **전부 나를 아프게 한다.** 역도 참이다 — 나를 아프게 하는 것 중 외곽선이 없는 것은 존재하지 않는다(적 탄 / 활성 장판 / 레이저 / 몸통 = 적 본체 실루엣). = I-1의 쌍방향 형태. |
| **V-3** | **꼬리 = 유도.** 잔상 꼬리는 `turnRateDegSec > 0`인 개체 **전용 표식**이며 진영을 가리지 않는다(적 유도탄에도, 플레이어 `seeker` 탄에도 붙는다). 유도가 아닌 어떤 탄도 꼬리를 갖지 않는다. → **꼬리 = 쫓는다, 색 = 누구 것인가.** 두 축이 직교하므로 조합 학습이 불필요하다. |
| **V-4** | **크기 하한이 진영을 보조한다.** 적 탄 `fairness.minBulletRadiusPx = 4`(정본), 플레이어 탄 `render.playerBulletMaxRadiusPx = 10`(정본), 픽업 ≤ 6px(정본 §7.8). 겹치는 구간이 존재하나 **운동·레이어·렌더모드가 이미 분리**하므로 크기는 보조 채널이다. |

> **V-2의 값어치**: 이 한 줄이 `mono` 모드(전 채도 0)에서 **위협 분리를 단독으로 보증**한다(§9.3 참조). 그러므로 I-1은 색맹 모드에서 **협상 불가**다.

### 2.2 적 탄 실루엣 — 어휘는 둘뿐

정본 §7.4: 적 탄 모양 = **원 / 육각(상태이상)**. `bullets[].shape` 필드가 스키마에 존재한다(정본 §9.7).

| `shape` | 조건 | 부가 |
|---|---|---|
| `circle` | `status === null` | 유도(`turnRateDegSec > 0`)면 **꼬리 3장**(V-3) |
| `hex` | `status !== null` | 크기 ×1.5(slow) / ×1.8(stun), 속도 `visual.statusBulletSpeedMul = 0.6`(정본), 흰 점선 궤적(slow) / 흰 이중 링 4Hz(stun) — 전부 정본 §7.11 |

- **`shape`는 `status`의 함수다** → `check.mjs`가 `(status === null) === (shape === "circle")`를 강제한다(§12-B4 요청). 어긋나면 **색이 거짓말하는 것**(I-2)이며 로드 실패시킨다.
- ★ **초안 B의 "유도탄 = 마름모" · "설치 장판 = 링"은 폐기한다.** ◆는 **물 속성의 예약 글리프**다(정본 §7.2) — 적 탄에 쓰면 4속성 색 금지 규칙의 정신을 형태 채널에서 위반하고, 자홍 마름모와 물색 마름모가 같은 화면에 공존한다. 유도는 **꼬리**로 읽는다(V-3).
- **레이저**: 탄이 아니라 이미터(정본 §8.5, `widthPx`). 실루엣 = 자홍 채움 라인 + 흰 코어 + 검은 외곽선 2px, 레이어 9.
- **활성 장판(`zone`)**: §3.4 참조 — 레이어 1의 유일한 피해원이며 정본 내부 모순 1건이 여기 있다(§12-A2).

### 2.3 글리프 예약 — 색과 같은 강도로 예약된다

| 규칙 |
|---|
| **속성 글리프 4종(● ▲ ◆ ✚)은 속성 주장 전용이다.** 아이콘·마커·불릿포인트·장식으로 쓸 수 없다. 자홍 개체·픽업·HUD 크롬·버튼·커서 어디에도 등장하지 않는다. |
| **예외 1 — 무채색 pip.** `●`/`○`는 **수량 표시**로 허용한다(투자 pip, 소지품 pip, 패시브 레벨 pip). 속성색이 아니므로 주장이 아니다. |
| **예외 2 — 은색 `●` = 노말.** 정본 §7.5의 "나머지 = 은색 ●"은 **정확한 주장**이다: 부여 안 됨 = ×1 = 노말. I-2 준수. |
| **★ 은색의 정의**: 정본이 "은색"을 6곳에서 쓰면서 값을 주지 않았다. **은색 ≡ `palette.element.normal` (`#DCE6F0`)로 확정한다.** 근거: 은색이 등장하는 전 지점(투자 0 스탠스 링 / 스트립 미부여 pip / 코어 외곽선 / ×1 숫자 / 자석 링 / `무` 배지)에서 그 의미가 **정확히 "×1"**이다 → I-2가 새 키를 금지한다. 별도 `palette.silver`를 만들면 "×1인데 노말색이 아닌 것"이 생겨 I-2가 깨진다. |
| **XP 마름모의 해명**: 정본 §7.2가 XP 픽업을 `작은 마름모`로 지정했다(◆ = 물). **충돌이 아니다** — 픽업/유도탄 분리는 **운동**이 보증하고(정본 §7.8: "순식간에 직선으로 빨려오면 100% 보상"), 픽업은 무광·무외곽선·레이어 2·≤6px이며 플레이어 탄은 글로우·additive·레이어 4다. **7채널 중 6채널이 다르다.** 그러나 이 좁음은 실재하므로 §9.4의 색맹 검증 항목에 명시적으로 넣는다. |

## 3. 텔레그래프 언어

### 3.1 정본이 이미 닫은 것

2상태(점선 = 미래 / 채움 + 검은 외곽선 = 현재) · 전환 순간 흰 1프레임 플래시 · 거동별 모양 8종 + 상황 4종 · 리드타임 하한(`straight` 0.55 ~ `bossPattern` 1.50) · `fairness.minTelegraphSec = 0.55` · `telegraphConcurrentMaxPerEntity = 2` · 전역 캡 8 + `deferAttack` · `emphasisBySpeed`(렌더 전용) · 레이어 1(지면) / 8(공중) · 알파 0.50. **전부 정본 §7.4 · §12.3 · §12.4.**

### 3.2 ★ 진행률 채널 (정본이 안 닫은 것)

**문제**: `laser`의 리드타임은 1.20 게임초다. 점선이 1.2초 동안 그냥 있으면 플레이어는 "곧"은 알지만 **"언제"**를 모른다. Normal에서 1.2초는 여유지만 Disaster(×3)에서 실제 0.4초다 — 그때 필요한 것은 강조가 아니라 **카운트다운**이다.

**규칙 (신규)**: **모든 텔레그래프는 진행률을 형상으로 인코딩한다.** 채널은 2개이며 항상 둘 다 작동한다.

**① 형상 고유 채널** — 정본이 이미 4종에 넣어 두었고, 나머지 4종을 여기서 지정한다.

| 이미터 | 정본이 지정한 모양 | 진행률 = |
|---|---|---|
| `straight` | 발사구 자홍 원 **확대** | 반경 (정본 내장) ✔ |
| `aimed` | 조준 브래킷 4각 **수축** | 수축량 (정본 내장) ✔ |
| `zone` | 지면 점선 원 **수축** | 수축량 (정본 내장) ✔ |
| `laser` | 점선 라인 → **폭 0→최종폭 보간** | 폭 (정본 내장) ✔ |
| **`fan`** | 점선 부채꼴 | ★ **호가 중심에서 바깥으로 자라남** (길이 = 진행률). 각도는 처음부터 실제 각도 고정 |
| **`ring`** | 발사구 점선 원 | ★ **반경 0 → 최종 반경 확대** (`straight`와 동일 규칙 = 학습 이월) |
| **`spiral`** | 점선 원 + 회전 화살 | ★ **화살이 1회전을 완주하면 발사** (진행률 = 각도) |
| **`wall`** | 점선 수평선 + 틈이 미리 비어 있음 | ★ **좌우 양끝에서 중앙으로 선이 그려지고, 완성되면 발사**. 틈은 **처음부터** 비어 있다(정본 규정 유지) |

**② 전역 dash 백스톱** — 형상과 무관하게 항상 작동한다.

```
리드타임의 0 ~ 85% 구간:  dash 간격 = visual.telegraph.dashPx = [6, 5]
리드타임의 85 ~ 100% 구간: 간격이 dashTightenMul(0.35)까지 선형 수축 → [6, 1.75]에 근접
발사 순간:                 흰색 1틱 플래시 → 채움 (정본 2상태)
```

> **왜 이게 옳은가**: 점선이 실선에 **근접**할 뿐 채워지지 않으므로 정본의 2상태 규칙("채움 금지")이 그대로 유지되고, 동시에 **"점선이 조여들면 발사 직전"**이라는 연속 판독이 8종 전부에 공짜로 붙는다. 규칙 1개, 새 색 0, 새 형태 0, 색맹 안전(형태 채널). `emphasisBySpeed`(굵기·알파)와 직교하므로 고배속에서 두 채널이 **겹쳐서 강해진다.**

### 3.3 텔레그래프 규격

```
visual.telegraph = { strokePx: 2, dashPx: [6, 5], alpha: 0.50,      // alpha는 정본 §12.3 인용
                     dashTightenAtPct: 0.85, dashTightenMul: 0.35,
                     fireFlashTicks: 1 }
```
- `fireFlashTicks: 1` = **게임 로직 1틱**. 렌더는 `render.fxMinRealMs = 50`(정본)에 의해 최소 50 실제ms 유지된다 → **Disaster(×3)에서도 발사 순간이 소실되지 않는다.** ("1프레임 플래시"가 판정 1틱인지 렌더 1프레임인지 정본이 모호했다 — 판정은 1틱, 표시는 50ms로 확정.)
- **텔레그래프는 발사 주체에 귀속된다** — 개체가 죽으면 그 개체의 텔레그래프도 즉시 소멸한다(발사되지 않으므로 그려질 이유가 없다 = I-2).

### 3.4 활성 장판 — ★ 정본 내부 모순 1건 (§12-A2)

정본 §7.4: **"지금 아픈 것 = 자홍 채움 + 불투명 + 검은 외곽선"**
정본 §12.3: 레이어 1(**지면 장판** 포함) = **"알파 ≤0.35"**

→ 활성 장판은 "불투명이어야 하고 동시에 알파 0.35 이하여야" 한다. **평가 불가.**

**이 섹션의 해소 (정본 개정 요청 A2)**: 레이어 1의 알파 상한은 **배경성 요소**(오버레이 띠의 배경 판 · 지면 텔레그래프)에만 적용하고, **피해원의 외곽선은 불투명**으로 한다.

| 장판 상태 | 규격 |
|---|---|
| 텔레그래프 (0.90 리드) | 자홍 **점선** 원 수축, 알파 0.50, 채움 금지 — 정본 §7.4 그대로 |
| **활성** | 자홍 **실선 외곽선 2px, 알파 1.0** + 자홍 채움 **알파 0.30** + 1Hz 맥동 |

> **근거**: 알파 상한 0.35의 목적은 "지면이 게임플레이 오브젝트와 밝기 경쟁을 하지 않게" 하는 것이다. 그런데 활성 장판은 **게임플레이 오브젝트**(피해원)다. 외곽선을 불투명으로 해도 **레이어 1이므로 적 탄(9)·플레이어 탄(4)·기체(5·6)를 물리적으로 가릴 수 없다** → I-4는 구조적으로 안전하다. 반대로 알파 0.35로 그리면 **나를 아프게 하는 것이 안 보인다** = I-1·I-3의 정신을 정면으로 위반한다. **채움은 0.30으로 낮춰 "지면"의 정체성을 지키고, 외곽선이 정보를 전부 나른다** — 이것은 I-1(테두리 = 나를 아프게 한다)의 정확한 적용이다.

## 4. 상태이상 표시

### 4.1 정본이 이미 닫은 것

호박 `#FFB000` 단일 채널 · 3층 표(적 tell / 온-플레이어 / 화면 가장자리) + 전용층(둔화 잔상 3장 / 스턴 링 점멸) · 리드 0.80(slow) / 1.50(stun) + 가장자리 사전 펄스 · `visual.statusBulletSpeedMul = 0.6` · 저항 방패 pip · 저체력 1Hz/2Hz · 피격 vs 실드 흡수 분리. **전부 정본 §7.11.**

### 4.2 적 측 — 상시 표식을 **두지 않는다** (확정)

`stage.statusStunMaxPerStage = 2`(정본)이므로 "스턴을 쏘는 2기"를 미리 표시할 유인이 있다. **그러나 두지 않는다.**

| | |
|---|---|
| **왜** | 적 본체에 호박을 올리면 정본 §7.2의 예약("호박은 **탄·적·배경 금지**")이 깨진다. 호박이 적 본체에 한 번이라도 나오면 "내 상태가 나쁘다"는 단일 의미가 소멸한다. |
| **왜 2** | 회전하는 이중 외곽선은 **엘리트가 이미 점유**했다(정본 §7.6). 재사용하면 두 개념이 같은 형태를 갖는다. |
| **대신 무엇이 보증하나** | `minStunTelegraphSec = 1.5` + **화면 가장자리 호박 1회 사전 펄스** + 육각 ×1.8 + 흰 이중 링 4Hz + 속도 ×0.6 + `stunMinDifficulty = "hard"`. **채널 5개**가 이미 붙어 있다. 6번째는 예약을 깨는 값만큼 벌지 못한다. |

### 4.3 ★ 호박 예약의 텔레그래프 예외 (정본 개정 요청 A4)

정본 §7.2: 호박은 "**플레이필드 밖 + 온-플레이어 아이콘에만**. 탄·적·배경 금지."
정본 §7.4: slow 탄 텔레그래프 = "발사구 원 + **호박 테두리 링**" ← **발사구는 적 위에 있다.**

문언상 "텔레그래프"는 금지 목록(탄·적·배경)에 없으므로 위반은 아니지만, **예약이 문언 해석에 의존하는 상태**다. 정본이 다음 한 줄을 명문화할 것을 요청한다:

> **호박은 ① 화면 가장자리 밴드 ② 온-플레이어 아이콘 ③ 상태이상 탄의 텔레그래프 링 ④ 상태이상 탄 자체의 테두리 — 정확히 이 4곳에만 등장한다. 적 본체·일반 탄·배경·픽업·HUD 크롬 금지.**

④를 함께 명문화하는 이유: 육각탄이 날아오는 동안 그것이 상태이상 탄임을 알리는 채널이 형태(육각)뿐이면 밀도 하에서 약하다. **자홍 육각 + 호박 테두리 링 1px**이면 "위협(자홍) × 내 상태를 나쁘게 함(호박)"이 색으로 정확히 합성된다. 그리고 이것은 §2.2의 `shape ↔ status` 동치를 색 채널로 이중화한다.

### 4.4 저항 유틸의 상시 가시성

정본 §7.11: "저항 유틸 보유·발동 → **온-플레이어 아이콘 옆** 호박 방패 pip 상시 표시". 문언상 **상태이상이 걸린 동안에만** 보인다(아이콘이 없으면 옆도 없다).

→ 평시에는 "내가 저항을 샀다"가 화면에 없다. **이 섹션의 처리**: 우 패널 `유틸 요약` 행(§7.4-E)이 `저항 40%`를 상시 표시한다. **온-플레이어는 정본 규격 그대로 두고, 평시 가시성은 패널이 맡는다** — 정본의 "온-플레이어 아이콘 옆"과 충돌하지 않으며 영역이 다르다(호박은 패널에 쓰지 않는다; 패널의 유틸 요약은 `palette.hud.textDim` 텍스트다).

## 6. 스탠스 판독

### 6.1 정본이 이미 닫은 것 (기체 측)

① 히트박스 코어 링(흰 r=4 + 스탠스색 링 2px, **레이어 10**) ② 슬롯 스트립(pip 7px × 4, 기체 하단 +14px, **레이어 6**) ③ 상태이상 배지(기체 위 −16px, 레이어 6) ④ 림 오라(알파 0.40) · 전환 3케이스 연출(정상 / 탈색 / 노말) · `visual.stance` 전 키. **전부 정본 §7.5.**

### 6.2 ★ 투자 pip = 슬롯 지도 (패널 측의 핵심)

정본 §4.2: `elementCapPerElement = 4` = **무기 슬롯 수** (LOCKED).
정본 §11.1 유효성 필터: **"속성 레벨 ≥ 현재 무기 수 → 그 속성 카드 제외"** → `투자 레벨 N ≤ 무기 수`가 **항상 참**이다(무기는 절대 잃지 않으므로 단조).

**귀결**: 좌 패널의 4칸 투자 pip은 **4칸 무기 슬롯과 1:1이며, pip을 그대로 슬롯 지도로 읽을 수 있다.**

```
W 불  ▲  ●●○○      ← "불 스탠스를 켜면 슬롯 1·2가 불로 발사된다"를 그대로 읽는다
E 물  ◆  ●○○○      ← "물 스탠스를 켜면 슬롯 1만 물"
R 풀  ✚  ○○○○      ← "풀 스탠스는 전부 무속성" (= 탈색 링이 뜬다)
```

- **부여 미리보기를 따로 그리지 않는다.** 투자 pip과 정보가 **동형**이기 때문이다. 4행 × 4pip = **부여 행렬 전체가 상시 화면에 있다** — 키를 눌러보지 않고도 4개 스탠스의 결과를 전부 안다.
- **3상태 pip**: `●` 채움 = 부여됨 / `○` 빈 원 = 슬롯은 있으나 투자 없음 / **`⊘` 점선 원 = 그 슬롯 자체가 아직 없음**. 세 번째 상태가 초반(무기 1~2개)에 "칸이 왜 4개인가"를 말없이 가르친다.
- **혼동 방지**: 투자 = **무채색 도트**(●/○), 부여 = **속성 글리프 칩**(▲/◆/✚) 또는 회색 `무`. 같은 화면에 두 종류의 "칸"이 있지만 **형태 어휘가 다르다.**

### 6.3 1:1 대응 강제

| 표면 | 규칙 |
|---|---|
| 기체 스트립 pip *i* (레이어 6) | ↔ 좌 패널 무기 슬롯 *i*의 부여 배지 |
| 좌 패널 스탠스 행의 투자 pip | ↔ 그 스탠스를 켰을 때의 스트립 |
| 히트박스 링 색 (레이어 10) | ↔ 좌 패널 `▶` 마커가 가리키는 행 |
| **어긋나면** | **구현 버그**(I-2). 세 표면은 같은 하나의 상태를 읽는다. |

### 6.4 온보딩 훅의 구체화 (정본 §9.9의 3키)

| 정본 키 | 이 섹션의 규격 |
|---|---|
| `autoEquipFirstElement` | 속성 카드 확정 순간 → 그 스탠스 자동 장착 + **좌 패널 해당 행의 키캡 박스가 흰색 → 속성색으로 0.4초 전환** + 행 하이라이트 이동. 투자 0→1 최초 전이에서만. |
| `stanceHintPulse` + `stanceHintPulseStageMax: 1` | **좌 패널 키캡 박스**가 그 속성색으로 **1Hz 맥동**(알파 0.40 ↔ 1.0). 스테이지 1 전용. |
| ★ **"지금 화면의 적에게 ×2인 키"의 정의** | 정본이 혼합 화면(70/10/10/10)에서 이것을 정의하지 않았다. **확정: 아레나 내 살아있는 적 개체 수 기준 최빈 속성의 `counter` 키.** 동수 타이 → `Q W E R` 순서 우선(결정적). 최빈이 노말이면 **맥동 없음**(노말에 counter가 없다). 새 데이터 없음 — 개체 수는 이미 세고 있다. |

**★ 토큰 게이팅의 가시화 (신규)**: 정본 §11.4는 "빨간불일 때만 유효, 그 외엔 **입력 무시**"라고만 한다. **입력이 무시된다는 것을 플레이어에게 알리는 채널이 없었다.** → 빨간불에서만 타이머 아래에 `[Shift] +30초` 힌트가 뜨고, 우 패널의 토큰 키캡이 그때만 활성화된다(§7.4-C). **"왜 Shift가 안 먹지"가 발생 불가.**

##### 7.2.1 ★ 코어 HP 바 — 정본에 자리가 없던 요소 (§12-A6)

### 8.2 ★ 글리프 LOD의 정의 (정본 §7.3의 미정 키)

정본 §7.3은 `off`에서 "**글리프 렌더: LOD 적용**", `cvd`/`mono`에서 "**항상 렌더**"라고만 했다. **LOD가 무엇인지 정본 어디에도 없다.**

**확정: `visual.glyph.lodMinBodyPx = 16`**

```
글리프를 그린다  ⟺  cbMode !== "off"                          // cvd·mono는 무조건
                  ∨ 개체가 엘리트·중간보스·보스·부위          // 정본 §7.6 "상시 글리프"
                  ∨ bodyPx >= visual.glyph.lodMinBodyPx       // ← 신규
                 ∧ !occluded                                   // 정본 glyph.occludedSkip
```

| | |
|---|---|
| **무엇이 걸리나** | `chaff`(sizePx 14) · `swarmChaff` — **가장 많고 가장 작은 것들** |
| **정보 손실 0인 이유** | 정본 §7.6: **속성색 외곽선은 "본체 크기 무관 항상"** 그려진다. `off` 모드에서 속성은 **외곽선 색이 이미 100% 나른다.** 글리프는 색맹 이중화 채널이고, 색맹 모드에서는 LOD가 꺼진다. → **어떤 모드에서도 정보가 사라지지 않는다.** |
| **왜 이 값인가** | `chaff` 14px < 16 ≤ `line`·`turret`·`bruiser`. `elite.sizeMult = 1.4`(정본)로 커진 chaff도 엘리트 규칙이 이미 상시 렌더한다. |
| **왜 "개수 기준"이 아닌가** | 개수 기준 LOD는 적이 하나 더 스폰되는 순간 **화면 전체의 글리프가 사라진다**(팝핑). 크기 기준은 개체마다 결정이 고정이므로 **팝핑이 물리적으로 발생하지 않는다.** |
| **위기 세션에서 특히 옳은 이유** | `crisisElementRule = "themePure"` → **60~70기가 전부 같은 속성**이다. 개체별 속성 판독이 **애초에 불필요**하다. 판독해야 하는 것은 ① 덩어리의 가장자리 ② 적 탄 — 둘 다 글리프와 무관하다. **위기 세션의 가독성 예산은 전액 적 탄으로 간다.** |

### 8.3 ★ 강등 사다리 (신규 — 하한기 정책 부재)

정본 §12.5는 기준기를 **M1 MacBook Air**로 잡고 "Disaster 최악 프레임 10ms / 예산 16.6ms"를 계산했다. 하한기는 **2019 Intel i5**로 선언했으나 **그 기계에서 예산을 넘겼을 때의 정책이 없다.**

**확정**: 최근 `render.degradeRecoverFrames`(120) 프레임의 렌더 시간 중앙값이 `render.degradeOnFrameMs`(14.0)를 넘으면 **아래 순서대로 단계를 내린다.** 중앙값이 다시 하한 아래로 내려가면 역순으로 복구한다.

| 단계 | 버리는 것 |
|---|---|
| 1 | 파티클 수 ×0.5 (`particles.super 3 → 1`) |
| 2 | 데미지 숫자 (`numberTargets` → `["boss"]`만) |
| 3 | 배율 마커 (×0.5 방패 호 제외 — **정보이지 장식이 아니다**) |
| 4 | 배경 패럴랙스 2층 → 1층 |
| 5 | 글리프 LOD 강화 (`lodMinBodyPx` → 24) — **`cbMode !== "off"`이면 이 단계를 건너뛴다** |

**절대 버리지 않는 것 (협상 불가)**: 적 탄 · 텔레그래프 · 히트박스 도트 · 스탠스 링 · 슬롯 스트립 · 상태이상 배지 · 화면 가장자리 호박 밴드 · 오버레이 띠 3개 · 패널.

| | |
|---|---|
| **왜 안전한가** | 사다리는 **레이어 7·11·0과 색맹 이중화 채널만** 건드린다. **판정에 관여하는 채널은 하나도 없다.** 그리고 전부 core 바깥(정본 §10.1)이므로 **시뮬 결정성·인증에 영향이 0**이다. |
| **왜 `cbMode`에서 5단계를 건너뛰나** | 색맹 모드에서 글리프는 **장식이 아니라 유일한 속성 채널**이다. 성능을 위해 접근성을 끄는 것은 사다리의 원칙(판정·정보 채널 불가침) 위반이다. |
| **왜 `evictOldest`로 안 되나** | 캡 초과 정책(정본 §12.1)은 **개체 수**의 문제를 풀고, 사다리는 **프레임 시간**의 문제를 푼다. `capHits == 0`이 인증된 콘텐츠도 느린 기계에서는 프레임을 넘길 수 있다 — 다른 문제, 다른 장치. |

### 8.4 패널 캐시

`hud.panelCacheDirtyOnly = true` — 패널은 **값이 바뀔 때만** 오프스크린 캔버스에 갱신하고 매 프레임 `drawImage` 2회(좌·우)로 blit한다.

- 좌 패널이 갱신되는 순간: 스탠스 전환 · 투자/무기/패시브 획득 · 슬롯 재정렬 · `stanceHintPulse` 프레임(스테이지 1 한정 — 이때만 매 프레임 갱신).
- 우 패널: 점수 · 코인 · 스톡 · 스테이지 · 무피격 상태 변화.
- ×3 배속에서 한 프레임에 3틱을 돌리므로(정본 §6.1) **렌더 예산이 가장 빡빡한 지점**이며, 패널 2장을 매 프레임 새로 그리는 것은 그 예산의 순손실이다.
- `cbMode`·`reduceFlash` 변경 시 **양쪽 캐시 즉시 무효화** (OPTIONS는 PAUSE에서 열린다 — 정본 §14).

### 8.5 겹침

- **적 겹침**: 속성색 외곽선이 각 개체를 분리(정본 §7.6) + `glyph.occludedSkip`(정본). 새떼는 §8.2의 근거로 개체 판독이 비-목표다.
- **적 탄 겹침**: 검은 하드 외곽선 2px(cvd 3px) + 흰 스페큘러 코어 + 불투명 → **탄 20개가 뭉쳐도 각 탄의 경계가 남는다.** 이것이 `additive 금지`의 이유다(additive면 뭉치는 순간 한 덩어리가 된다).
- **픽업 누적**: 정본 §7.8 — 아레나 하단 경계에서 정지·누적. `caps.pickups = 256` 초과 시 `merge`(손실 0). **누적 더미는 무광·≤6px·레이어 2**이므로 그 위 모든 것이 읽힌다.
- **HUD ↔ 아레나**: 물리적으로 겹치지 않는다(1280 = 350 + 580 + 350).

## 9. 색맹 모드

### 9.1 정본이 이미 닫은 것

`cbMode ∈ {off, cvd, mono}` · `palette.elementCvd` 명도 사다리(96/76/55/40, 인접 ≥15 L\*) · 글리프 ×1.5 · 항상 렌더 · 적 외곽선 3px · 배경 명도 상한 0.22 · HUD/카드/고지 텍스트 라벨 강제 · OPTIONS에서 TITLE·PAUSE 양쪽 진입. **전부 정본 §7.3 · §14.**

> 정본의 철학 인용: "4개 카테고리를 어두운 배경 위에서 **색만으로** 이색각자에게 분리시키는 것은 이론적으로 불가능하다. 목표는 **색이 완전히 실패해도 게임이 100% 성립하는 것**이고, 그 보증은 **글리프**다."

### 9.2 ★ `mono`의 범위 (정본 미정)

정본 §7.3의 `mono` 열은 팔레트 행에 "`elementCvd` + 채도 0"이라고만 적었다. **`threat`·`status`·`pickup`·`hud`가 무채화되는지 미정이다.**

**확정: `mono`는 전 팔레트를 무채화한다** (`element` · `threat` · `status` · `pickup` · `hud` · `bg` 전부). 각 색은 자신의 L\*를 유지한 회색이 된다.

| | |
|---|---|
| **왜 전부인가** | ① `mono`의 대상은 **전색맹(achromatopsia)**이며 그들에게 색은 이미 존재하지 않는다 — 부분 무채화는 그들에게 아무 의미가 없고 개발자에게만 거짓 안심을 준다. ② `mono`의 두 번째 목적은 **"색 채널이 완전히 죽어도 성립한다"의 증명 스위치**다. 일부를 남기면 **그 증명을 하지 못한다.** ③ 하드웨어가 임의로 hue→휘도를 사상하게 두는 것보다, **우리가 L\*를 통제**하는 편이 전색맹 사용자에게 실제로 더 낫다. |

### 9.3 ★ `mono`의 위협 분리 — I-1이 단독 보증한다

**정직한 산술**:

```
elementCvd 사다리:  normal L*96 │ fire L*76 │ water L*55 │ grass L*40
위협(자홍 #FF2E88):                              L*54  ← water L*55와 1 차이
배경 상한 (HSL L 0.22):                          ≈ L*24
```

**즉, `mono`에서 적 탄과 물 속성 플레이어 탄은 명도로 구분되지 않는다.**

**자유 슬롯이 없다는 증명**: 사다리가 40~96을 점유하고 배경이 ≤24를 점유한다. 남은 구간은 24~40뿐이며 그 중앙(L\*32)은 `grass`(40)와 8 차이 — 사다리 자신의 기준(≥15)에 미달한다. **`mono`에서 위협에 줄 수 있는 명도는 존재하지 않는다.**

**그러므로 분리는 명도가 아니라 다음이 한다** (정본 §7.4의 7채널 중 6개가 살아있다):

| 채널 | 적 탄 | 플레이어 탄 |
|---|---|---|
| **엣지** | **검은 하드 외곽선 3px** (cvd/mono) | **없음** (I-1) |
| 코어 | 흰 스페큘러 점 | 같은 회색의 밝은 코어 |
| 합성 | 불투명 | additive |
| 알파 | 1.0 | ≤ 0.80 |
| 레이어 | 9 | 4 |
| 모양 | 원 / 육각 | **글리프만** (정본 §7.3의 `mono` 행) |

> **결론: `mono`에서 위협 분리를 하는 것은 I-1이다.** 어두운 배경(L\*24) 위의 **검은 3px 테두리**는 그 자체로 L\*4의 국소 후광이며, 플레이어 측 어디에도 존재하지 않는다. **그러므로 I-1은 색맹 모드에서 협상 불가다** — "플레이어 탄에 외곽선을 조금 넣으면 예쁘겠는데"는 접근성 기능 하나를 통째로 삭제하는 변경이다. §2.1-V-2가 이것을 쌍방향으로 못 박는다.

### 9.4 검증 항목 (구현 체크리스트 — 시뮬레이터로 확인)

| # | 확인 | 답 |
|---|---|---|
| 1 | deuteranopia에서 자홍(적 탄) vs 호박(내 상태) | ✔ 자홍 → 회청색 / 호박 → 황색. **그리고 영역이 배타**다(호박은 플레이필드에 없다 — §4.3). |
| 2 | deuteranopia에서 **호박(밴드) vs 코인 금** | ⚠ **거의 동일한 황색.** 분리 = **영역**(가장자리 밴드 vs 플레이필드 픽업) + **형태**(밴드 vs 원반+구멍) + **레이어**(13 vs 2) + **운동**(정지 vs 스트릭). **색은 분리에 기여하지 않으며, 기여할 필요가 없다.** |
| 3 | deuteranopia에서 fire(L\*76) vs grass(L\*40) | ✔ `elementCvd` 36 L\* 차이 + 글리프 ▲/✚. |
| 4 | **XP 마름모(백) vs 물 탄(◆)** | ⚠ **가장 좁은 케이스.** 분리 = 글로우 유무 · 외곽선 유무 · 레이어(2 vs 4) · 크기(≤6 vs ≥8 상당) · **운동**(정본 §7.8: "순식간에 직선으로 빨려오면 100% 보상"). **7채널 중 6채널.** 통과하나 **좁으므로 이 항목은 회귀 검사로 남긴다.** |
| 5 | `mono`에서 적 탄 vs 물 탄 | ⚠ **L\* 1 차이.** §9.3 — **I-1 단독 보증.** 통과하되 **I-1 변경 시 이 항목이 즉시 깨진다**는 사실을 명시적으로 기록한다. |
| 6 | `mono`에서 ×0.5 회색 방패 호 | ✔ 채널은 **형태(호) + 파티클 0 + 금속 "틴"**이며 애초에 색이 아니다(정본 §7.7). |
| 7 | `mono`에서 배경 vs 위협 | ✔ L\*24 vs L\*54 = 30 차이 + 소프트 엣지 vs 하드 엣지. |
| 8 | `cvd`에서 기체 슬롯 스트립 | ✖ → **§9.5에서 수정** |

### 9.5 ★ `cvd`의 기체 스트립 오버플로 (정본 규격 충돌)

정본 §7.3: `cvd` → **글리프 크기 ×1.5**
정본 §7.5: 슬롯 스트립 pip = **`visual.stance.pipPx = 7`**

→ 7px pip 안에 ×1.5 배율의 글리프를 그릴 수 없다. **정본의 두 값이 서로를 초과한다.**

**해소 (정본 편입 요청 B2): `visual.stance.pipPxCvd = 10`** (`cbMode !== "off"`일 때 `pipPx` 대신 사용)

```
off:  pip 7  × 4 + 간격 2 × 3 = 34px   (정본 §7.5 그대로)
cvd:  pip 10 × 4 + 간격 2 × 3 = 46px
```
- 기체 스프라이트 반경 14 → 지름 28px. 46px 스트립은 기체보다 넓지만 **기체 하단 +14px, 레이어 6**이므로 적 탄을 가리지 않는다(I-4 안전). 이동 영역 좌우 인셋 20px(정본 §1.1) → 스트립 좌우 23px가 아레나 밖으로 3px 나가는 경우가 있다 → **스트립은 아레나 rect로 클리핑**한다(pip이 잘려도 순서·색은 읽힌다).
- **`cvd`에서 스트립이 안 읽히면 §7.5의 3중 표시 중 하나가 죽고, 그 순간 "슬롯 순서 부여"라는 이 게임의 시그니처 시스템이 색맹 사용자에게 존재하지 않게 된다.** 3px는 그 대가로 싸다.

### 9.6 텍스트 라벨 강제의 적용 지점 (`cbMode !== "off"`)

| 표면 | `off`에서도 텍스트 있음? | `cvd`에서 추가 |
|---|---|---|
| 좌 패널 스탠스 행 | ✔ (`노말`/`불`/`물`/`풀`) | — |
| 좌 패널 상성표 | ✔ (열 헤더 이름) | — |
| 무기 부여 배지 | ✖ (글리프만) | **속성명 1자** (`불`/`물`/`풀`/`무`) |
| 우 패널 혼합 바 | ✔ (라벨 행) | — |
| 드래프트 `elementLevel` 카드 | ✔ (정본 §11.1 필수 표기) | — |
| 테마 배너 | ✔ | — |
| **기체 슬롯 스트립** | ✖ | **불가**(7~10px) → **글리프 ×1.5가 유일한 채널** → §9.5가 필수인 이유 |

- `off`에서 이미 텍스트를 가진 표면이 대부분이다 = **`cbMode` 분기가 2곳뿐**이다(부여 배지 · pip 크기). 접근성 비용이 구조적으로 낮다.
- OPTIONS에서 `cbMode` 변경 → **즉시 반영**(재시작·재로드 없음). 패널 캐시 무효화(§8.4).
