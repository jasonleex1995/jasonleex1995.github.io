# NAN 2026 — 정본 (CANON) v1.0

> **제목: `TETRA` (확정).** 가제 종료.
>
> 이 문서는 이 게임의 **단일 설계 권위**다. 초안 6종(`_drafts/01`~`06`)은 **재료이며 정본과 충돌하는 부분은 전부 무효**다.
> 이후 작성되는 모든 섹션 문서는 정본을 **참조만** 하고 **재정의는 금지**한다.
>
> 목표: **기획 확정 → 개발 중 설계 결정 0 → 이후 밸런싱은 오직 숫자.**

## 0. 정본 규약

### 0.1 재정의 금지 규칙

| # | 규칙 |
|---|---|
| **C-1** | **한 항목에 답은 하나.** 정본이 정한 값·모델·어휘를 다른 문서가 다시 정의하면 그 문서가 틀린 것이다. |
| **C-2** | **참조만 가능.** 섹션 문서는 정본의 키 이름을 인용할 수는 있으나 **값을 적을 수 없다.** 값의 유일한 거처는 `data/*.json`이다. |
| **C-3** | **어휘는 닫혀 있다.** §9의 모든 enum(이동·이미터·패밀리·패시브 스탯·부위 타입·편대·도형)은 **동결**이다. 추가·개명 = 설계 변경 = 정본 개정. |
| **C-4** | **구조 동결 / 값 튜너블.** 데미지 공식의 항, 스키마의 필드 집합, 상태 기계, 레이어 스택은 구조다. 숫자·확률·곡선은 값이다. **값을 바꾸려고 `.js`를 여는 순간 = 정본 실패.** |
| **C-5** | **"동결" 선언은 정본만 할 수 있다.** 섹션 문서의 동결 선언은 무효. |
| **C-6** | 정본에 없는 항목이 개발 중 발견되면 → **정본을 개정**하고 개정 이력을 남긴다. 섹션에서 즉석 결정 금지. |

### 0.2 값의 단위 (예외 없음)

| 접미 | 단위 | 사용처 |
|---|---|---|
| **`Sec`** | **게임초 (game second)** | **JSON의 모든 시간.** 배속과 무관한 불변 단위. |
| `Ms` | **실제 밀리초 (real ms)** | **정확히 3곳에서만 등장한다** — ① `loop.maxFrameGapMs` ② `render.fxMinRealMs` ③ `bot.reactionMs`·`bot.reactionJitterMs`·`bot.stanceSwitchMs` |
| `Px` / (무접미 거리) | 논리 픽셀 | 좌표·반경·폭 |
| `PxSec` | 논리 픽셀 / 게임초 | 속도 |
| `DegSec` | 도 / 게임초 | 각속도 |
| `Pct` | 0.0~1.0 실수 | 비율 |
| `Hz` | 게임초당 횟수 | 맥동·점멸 주기 |

- **틱은 JSON에 등장하지 않는다.** 틱(1/60 게임초)은 엔진 내부 단위이며, 데이터 어휘가 아니다. 초안 C의 `*_ticks` 키는 **전부 폐기**한다.
- **실제시간은 게임플레이에 존재하지 않는다.** 회피 예산(리드타임·탄속·스폰 거리·i-frame·타이머)은 전부 게임초이며 배속을 그대로 탄다. `Ms` 3곳은 각각 루프 안전장치 / 순수 장식 / 시뮬 봇 모델이며 **판정에 관여하지 않는다.**

### 0.3 명명 규칙

| 대상 | 규칙 | 예 |
|---|---|---|
| JSON 키 | **camelCase** | `hitboxRadius`, `telegraphSec`, `elementCapTotal` |
| JSON 파일 | 소문자 단수/복수 | `rules.json`, `enemies.json` |
| enum 값 (id) | **camelCase** | `orbitDrift`, `randomInArena`, `mbNest` |
| 콘텐츠 id | **snake 금지, camelCase** | `columnAnt`, `frostLance`, `seaTheme` |
| 버전 키 | `schemaVersion` | 모든 파일 루트에 필수 |
| 표시 이름 | `name` (한국어 인라인) | `"name": "추격체"` |
| 설명 1줄 | `desc` (한국어 인라인) | `"desc": "최근접 적을 추적"` |

- **로컬라이제이션 없음.** 한국어 단일. 초안 F의 `descKey` / 문자열 테이블 모델은 폐기 — 이름·설명은 각 엔티티에 인라인한다. (파일 매니페스트가 9개로 닫힌다.)
- `snake_case`(C·D)는 전부 폐기. `schema_version` → `schemaVersion`.

## 1. 좌표 · 화면

### 1.1 값 (전부 하나씩)

| 항목 | 키 | 값 |
|---|---|---|
| 논리 캔버스 | `view.logicalW` / `view.logicalH` | **1280 × 720** (16:9 고정) |
| 아레나 rect | `view.arena` | **`{x: 350, y: 0, w: 580, h: 720}`** |
| 좌 패널 | `view.panelLeftW` | **350** (x = 0~350) |
| 우 패널 | `view.panelRightW` | **350** (x = 930~1280) |
| 최소 뷰포트 | `view.minViewportW/H` | 1024 × 576 |
| DPR 상한 | `view.maxDpr` | 2 |
| 아레나 상단 띠 | `view.bandTopH` | 48 |
| 아레나 HP 띠 | `view.bandHpH` | 24 |
| 아레나 XP 띠 | `view.bandXpH` | 24 |
| 플레이어 이동 인셋 | `view.playerBoundsInset` | `{top: 56, bottom: 56, left: 20, right: 20}` |
| → 이동 가능 영역 | (파생) | **540 × 608 @ (370, 56)** |
| 충돌 그리드 셀 | `collide.gridCellPx` | 64 |
| 적 스폰 라인 | `view.spawnLineY` | −40 (아레나 좌표) |

- **1280 = 350 + 580 + 350.** 세로 슈팅을 16:9에 올릴 때 남는 좌우를 패널이 정확히 채운다 → "HUD 자리가 없다"가 구조적으로 발생 불가.
- **HUD는 아레나 rect를 침범하지 않는다.** 예외는 정확히 둘: ① §1.2의 아레나 오버레이 띠 3개 ② §7.5의 기체 부착 요소.
- 스케일: `s = min(vw/1280, vh/720)`, CSS 중앙 정렬 레터박스. 백킹스토어 = `1280·min(dpr,2) × 720·min(dpr,2)`, `ctx.setTransform(s,0,0,s,0,0)`. **모든 게임 좌표는 논리 픽셀이며 스케일은 렌더에만 존재한다.**
- 아레나는 **고정 화면**(카메라 없음). 배경만 스크롤.
- 뷰포트 < 최소 → 플레이 차단 + 안내(입력 무시). **데스크톱 키보드 전용, 터치·모바일 미지원(확정).**

### 1.2 아레나 오버레이 띠 3개 (유일한 예외)

| 띠 | y 범위 | 내용 |
|---|---|---|
| 상단 | 0~48 | **보스 타이머**(중앙, 큰 숫자) + 배속 배지(우) / 잡몹 페이즈 중엔 페이즈 잔여 |
| 하단 A | 672~696 | **HP 세그먼트 바** 5칸(칸당 20 HP) + 우측 `86/100` |
| 하단 B | 696~720 | **XP 바** + 좌측 `Lv.23` |

- **띠는 레이어 1(지면)에 그린다.** 배경 위, 모든 게임플레이 오브젝트 아래. → 적 탄이 띠 위를 지나가므로 **I-4(적 탄은 절대 가려지지 않는다)가 유지된다.**
- 띠 영역은 플레이어 이동 인셋(상 56 / 하 56)으로 **플레이어가 들어갈 수 없다.**
- 적은 `spawnLineY = −40`에서 스폰해 반투명 상단 띠를 통과해 내려온다(진입이 미리 보임).
- 타이머는 **게임시간을 표시한다**(항상 3:00 → 0:00). 옆의 `×3.0` 배지가 실제 시간이 압축돼 있음을 알린다.

### 1.3 밀도 예산의 기준 좌표

- 초안 B의 `render.refPlayfieldPx: [640, 960]`과 `caps.areaScale`은 **폐기**한다.
- **모든 캡·밀도·통과 틈·탄속은 위 아레나 580 × 720 기준으로 직접 정의된다.** 기준 변환 규칙이 존재하지 않으므로 드리프트가 발생할 수 없다.

## 2. 플레이어

### 2.1 체력 · 방어

| 항목 | 키 | 값 |
|---|---|---|
| 최대 HP | `player.hpMax` | **100** |
| HP 세그먼트 | `player.hpSegment` | 20 (→ 5칸 바) |
| 기본 방어력 | `player.defenseBase` | 0 |
| 방어력 상한 | (파생: 상점 `defense` 2 × 4스택) | **8** |
| 데미지 하한 비율 | `player.damageFloorRatio` | **0.25** |
| 저체력 임계 | `player.lowHpThreshold` | 0.30 |
| 위독 임계 | `player.lowHpCriticalThreshold` | 0.15 |
| 잔기 | — | **없음 (확정)** |

- HP는 **연속 정수**. 하트/라이프제 아님. 적 탄은 탄마다 데미지가 다르다(`bullets[].dmg`).
- **방어력은 상점에서만 획득한다.** 패시브 스탯 어휘에 `defenseAdd`가 **없다**(§9.6) → 방어력이 8을 넘는 경로가 구조적으로 존재하지 않고, 칩딜 완전 무효화가 불가능하다.
- 관대함의 산술적 보증: `hpMax 100`, 최대 단발 데미지 22(`bullets` 저작 상한), i-frame 1.0 게임초 → **죽으려면 최소 5초 동안 매초 최대 피해를 연속으로 맞아야 한다.** 즉사·순삭이 구조적으로 불가능.

### 2.2 이동 모델 — **관성 없음 (확정)**

| 항목 | 키 | 값 |
|---|---|---|
| 기본 속도 | `player.moveSpeed` | **280** (PxSec) |
| 반응 시상수 | `player.moveResponseTau` | **0.0** (Sec) |
| 대각선 정규화 | `player.diagonalNormalize` | **true** (×0.70710678) |
| SOCD (반대키 동시) | `input.socd` | **`"lastInput"`** — 마지막에 눌린 키 우선 |
| 이동 속도 상한 | (파생: 상점 3스택 ×6% + 패시브 `moveSpeedMul` 최대 20%) | 280 × 1.18 × 1.20 = **396.5** |

- `velocity = dir × moveSpeed`. **가속/감속 없음, 즉시 정지.** 관성 = 미세 조작 실패 = 트위치 = 기둥 위반.
- 코드에 지수 스무딩 항이 **존재하고** 기본값이 0이다 → "살짝 미끄럽게"가 필요해도 **숫자만** 바뀐다.
- SOCD를 **상쇄(0)로 하지 않는 이유**: 양쪽을 눌렀을 때 기체가 멎는 것은 정확히 "미세 조작 실패"다. 마지막 입력 우선이 곧 의도대로 움직인다.
- 적/보스와 겹쳐도 밀리지 않는다 (`enemy.contactPush = false`). 관통 통과.

### 2.3 히트박스

| 대상 | 값 |
|---|---|
| 플레이어 스프라이트 | 원 r = `player.spriteRadius` = **14** |
| **플레이어 히트박스** | 원 r = `player.hitboxRadius` = **4** (스프라이트의 29%) |
| 적 탄 히트박스 | 원 r = `bullets[].radius × bullet.hitboxScale(0.8)` |
| 적 기체 히트박스 | 원 r = `enemies[].radius` |
| 보스 코어·부위 | 원 r = `core.radius` / `parts[].radius` |

- **히트박스는 상시 렌더** (`hud.hitboxAlwaysVisible = true`). **그려진 흰 코어 도트의 반지름 = 히트박스 반지름 = 4.** 크기도 거짓말하지 않는다.
- 그 둘레에 **현재 스탠스 색 링**(stroke 2px, r 4→6). 눈이 이미 가 있는 곳에서 스탠스를 읽는다.
- **그레이즈 없음 (확정).** 관련 키·UI·점수 항목 전부 없음.

### 2.4 무적시간 (i-frame)

| 항목 | 키 | 값 |
|---|---|---|
| 길이 | `player.iframeSec` | **1.0** (게임초) |
| 깜빡임 | `visual.iframeBlinkHz` | **8.0** (alpha 0.35 ↔ 1.0) |
| 넉백 | — | **없음** |
| 이동 경직 | — | **없음** |
| 피격 시 탄 소거 | — | **없음** (소거는 폭탄·`hitBulletClearRadius`의 영역) |
| 폭탄 발동 무적 | `bomb.iframeSec` | **1.5** (게임초) |
| 컨티뉴 부활 무적 | `flow.continueIframeSec` | **3.0** (게임초) |

- ★ **모든 피해원이 i-frame을 공유한다.** 적 탄 / 몸통 충돌 / 설치 장판 / 레이저 / 새떼 전부. **그러므로 게임초당 최대 1회만 피격된다.**
  - → 오라/필드·오빗/퍼리미터의 "파고들어 몸빵"이 다중 히트 즉사로 붕괴하는 것이 구조적으로 불가능. 12패밀리 중 2개의 설계 의도가 이 한 줄로 성립한다.
  - → **장판(`zone`)에 `dps` 개념은 존재하지 않는다.** 장판도 "적용 1회 = `dmg`"이며 i-frame이 게이트한다. **모든 피해는 "적용 1회"로 통일된다.**
- **실드 흡수 시에도 i-frame이 동일하게 발동**하고 데미지는 0이며 실드 1개가 소모된다. 실드 2장 = "2초 연속 무적"이 아니라 **최대 2회의 피격이 각각 없던 일이 되는 것**이며, 게이트는 정상 피격과 완전히 동일하다. (틱당 실드 1개.)
- 히트박스 링은 깜빡임 중에도 **계속 선명하게 유지**(위치는 항상 읽힘).
- i-frame은 **게임초 기준** — 배속이 곧 실제 반응 여유를 줄이는 것이 난이도의 정의.

### 2.5 몸통 충돌 — 데미지 있음, i-frame 공유

- 데미지 = **`enemies[].contactDmg` / `bosses[].core.contactDmg` / `bosses[].parts[].contactDmg`** (개체별 필수 필드, 기본값 폴백 없음).
- 저작 가이드(코드 키 아님): 잡몹 6~8 / 엘리트는 `elite.contactDmgMul` 1.5배 / 중간보스·보스·부위 14~16.
- **왜 0이 아닌가**: 접촉이 무해하면 "적 위에 얹혀서 오라로 갈기기"가 무위험 최적해 → *의미 있는 이동* 기둥 소멸.
- **왜 즉사가 아닌가**: 오빗/오라의 "파고들기" 자세가 성립 불가해진다.
- **결론**: 접촉 = 게임초당 최대 1회 + 탄보다 낮은 데미지 → **파고들기는 감수할 만한 비용**.

### 2.6 런 시작 상태 (확정)

| 항목 | 값 |

### 2.7 상태이상 모델 (확정 — 초안에 없던 blocker)

| 항목 | 키 | 값 |
|---|---|---|
| 둔화 이동 배율 | `status.slowMoveSpeedMul` | **0.55** |
| 둔화 기본 지속 | `bullets[].statusDurationSec` | 탄별 (저작 상한 4.0) |
| 스턴 지속 | `bullets[].statusDurationSec` | 탄별, **`fairness.maxStunSec = 1.0` 강제** |
| 스택 규칙 | `status.stackMode` | **`"refresh"`** — 중첩 금지. 재적용 시 `잔여 = max(잔여, 신규 × (1 − statusResist))`. **배율은 절대 중첩하지 않는다.** |
| 저항 의미 | `status.resistAffects` | **`"duration"`** — 지속시간만 감소. 강도(0.55)는 불변 |
| 저항 상한 | (상점 `resist` 0.20 × 3스택) | **0.60** (지속시간 −60%) |
| 둔화 + 스턴 동시 | — | 스턴이 우선(이동 0). 둔화 타이머는 계속 흐른다 |

**★ 스턴 중에 무슨 일이 일어나는가 (확정)**

| 대상 | 스턴 중 |
|---|---|
| 이동 입력 | **무시** |
| **자동발사** | **계속된다** |
| **스탠스 전환 (QWER)** | **유효하다** |
| 폭탄 (Space) | **유효하다** |
| 시간 토큰 (Shift) | **유효하다** |

> 근거: 스턴이 스탠스를 잠그면 그건 **게임의 핵심 시스템을 뺏는 것**이다 = feel-bad + 정보 기반 선택 배신. 그리고 폭탄이 "살아남는 카드"인데 스턴 중에 못 쓰면 폭탄의 정의가 무너진다. **스턴은 위치를 잠그는 것이지 판단을 잠그는 것이 아니다.**

## 3. 데미지 공식 (구조 동결 — 항 추가/제거 금지)

### 3.1 플레이어 → 적

```
1. base   = w.dmg                                          // 무기 레벨 행에서 읽음
2. dmgMul = 1 + Σ(패시브 dmgMul)                            // 가산 풀 → 1회 적용 (곱연산 폭주 방지)
3. elem   = elements.matrix[stampElement][target.element]   // ∈ {0.5, 1.0, 2.0}
   if (elem > 1) elem = 1 + (elem - 1) × elementBonusMul    // 패시브 resonance. 기본 1.0
4. gate   = target.isCore ? (boss.coreGateMul ^ aliveArmorPartCount) : 1
5. final  = base × dmgMul × elem × gate
6. 적용은 float 누산. 표시만 Math.round.
```

| 항 | 확정 사항 |
|---|---|
| **크리티컬** | **없음 (확정).** 식에 crit 항이 **존재하지 않는다.** `critChance`/`critMul`은 패시브 스탯 어휘에도 없다. |
| **데미지 난수** | **없음 (확정).** 데미지 경로에서 RNG를 호출하지 않는다 → 결정성 + 가독성 + 시뮬 분산 축소. |
| **적의 방어력** | **존재하지 않는다 (확정).** `enemies.json`·`bosses.json` 어느 스키마에도 defense 필드가 없다. 초안 F 식의 7번 항(`1 − def/(def+defenseK)`)은 **플레이어→적 경로에서 삭제**된다 — 그것은 적→플레이어 전용이며 §3.2로 이동했다. (이것이 blocker "def가 정의 안 됨"의 답이다: **평가할 항 자체가 없다.**) |
| `elementBonusMul` | ★ **의미 고정**: `elem = 1 + (elem − 1) × k` 의 **k**다. `resonance` 패시브 values = `[1.10, 1.20, 1.30, 1.40, 1.50]` → **유효 상성 배율 ×2.1 ~ ×2.5**. (초안 C의 `[2.1…2.5]`를 그대로 넣으면 ×3.1~3.5가 되는 함정 — 값은 k로 재작성되었다.) `elem ≤ 1`에는 절대 적용되지 않는다(×0.5·×1 불변). |
| `gate` | 코어 전용. 살아있는 **`armor` 타입 부위 수**만 지수에 들어간다. `mobility`·`armament`는 게이트에 **영향 없음** → §8.13의 트레이드오프가 성립한다. |
| 지속형 | 오라·장판·오빗의 `dmg`는 **1회 적용당** 값이며 `tickIntervalSec`가 주기다. DPS 정규화 없음. |

### 3.2 적 → 플레이어

```
1. raw   = source.dmg                                       // bullets[].dmg | contactDmg | zone dmg
2. taken = ceil( max( raw − player.defense , raw × player.damageFloorRatio ) )
3. hp   -= taken                                            // i-frame 게이트 통과 시에만
```

| 항 | 확정 사항 |
|---|---|
| **방어력 모델** | **정액 감산 (확정).** 정률(`def/(def+K)`)·곱연산(`×0.92`) 모델은 폐기. 정액이라 잡탄 칩딜은 거의 무해해지고(관대 기둥), 큰 탄·레이저는 여전히 아프다(위치잡기 기둥). |
| 하한 | 원본의 **25%** (`damageFloorRatio`). 하한이 있어 무적화 불가. |
| 실측 | 방어력 8(최대) 기준: 소형탄 8 → `max(0, 2) = 2` / 레이저 22 → `max(14, 5.5) = 14`. **칩딜 4배 감소, 큰 탄은 36% 감소** = 정확히 의도한 비대칭. |
| i-frame | **모든 피해원 공유, 1.0 게임초** (§2.4). 데미지 계산 전에 게이트한다. |
| 실드 | 게이트 통과 시 실드 스택 > 0이면 `taken = 0` + 실드 −1 + i-frame 발동. 무피격 판정 유지(§11.3). |

## 4. 속성 · 스탠스

### 4.1 상성 (LOCKED)

| 공격 ↓ \ 방어 → | 노말 | 불 | 물 | 풀 |
|---|---|---|---|---|
| **노말** | ×1 | ×1 | ×1 | ×1 |
| **불** | ×1 | ×1 | ×0.5 | **×2** |
| **물** | ×1 | **×2** | ×1 | ×0.5 |
| **풀** | ×1 | ×0.5 | **×2** | ×1 |

- 물 > 불 > 풀 > 물. **노말은 항상 ×1**(안정적 축).
- **적의 공격에는 속성이 없다 (확정).** `bullets.json`에 `element` 키가 **존재하지 않는다** — 스키마가 규칙을 강제한다.
- **표 순서 = 키 순서 = `Q(노말) W(불) E(물) R(풀)`.** 전 문서·전 UI·전 카드·전 패널이 이 순서다. 초안들의 "물·불·풀" 나열은 전부 이 순서로 정정한다.

### 4.2 투자

| 항목 | 키 | 값 |
|---|---|---|
| 투자 가능 속성 | `elements.investable` | **`["fire", "water", "grass"]`** — **노말은 투자축이 아니다.** 드래프트에 `노말 +1` 카드는 존재하지 않는다 |
| 개별 상한 | `player.elementCapPerElement` | **4** (= 무기 슬롯 수, LOCKED) |
| **합계 상한** | `player.elementCapTotal` | **6** (튜너블) |
| 합계 상한 제약 | check.mjs | `4 ≤ elementCapTotal < 3 × elementCapPerElement` |

- **합계 6의 근거**: 개별 4만 있으면 `불4·물4·풀4` = 항상 최적 매칭 = 문서가 막으려던 붕괴. 합계 6이면 `4/2/0`·`3/3/0`·`2/2/2` 같은 **진짜 다이얼**이 생기고, `4/2/0` 빌드는 남은 한 속성의 테마 스테이지를 전부 무속성(×1)으로 싸운다 = 설계된 대가.
- 초안 C의 8은 **`4/4/0`을 허용**해 3속성 중 2개를 만렙으로 만든다 → 최종 보스(3속성 armor 관)의 커버리지 시험이 무뎌진다. **6 채택.**

### 4.3 부여 (스탠스)

| 규칙 | 값 |
|---|---|
| 부여 순서 | 현재 스탠스 속성의 투자 레벨을 `N`이라 할 때 **슬롯 1..N의 무기가 그 속성으로 발사**, 나머지는 무속성(×1) |
| 노말 스탠스(Q) | **N = 0 강제.** 전 슬롯 무속성 = 전부 ×1을 **명시적으로 선언**하는 키 |
| 전환 쿨다운 | `stance.switchCooldown` = **0** |
| 전환 딜레이 | 없음. **다음 틱부터 즉시 적용** |
| 런 전체 유지 | `stance.persistAcrossStages` = **true** — 스테이지 전환·보스 진입·드래프트·상점·컨티뉴·사망 후에도 유지 |
| 재계산 시점 | 속성 레벨 상승 / 새 무기 획득 / 슬롯 재정렬 시 **즉시(다음 틱)**. 드래프트 화면에서 `불 +1`을 고르면 그 화면에서 바로 부여 프리뷰가 갱신 |
| 슬롯 재정렬 | 가능. **키보드 전용**(§5), **드래프트 화면에서만** |

**★ 노말 스탠스(Q)가 하는 일 (확정)**: 기계적으로는 "투자 0인 속성 스탠스"와 동일하지만 **의도의 표명**이라는 UX 역할이 있다 — ① ×0.5 사고 방지(잘못된 스탠스보다 Q가 항상 안전) ② 노말 부위/노말 적 앞에서 "여긴 상성이 없다"를 확인하는 리셋 버튼 ③ 런 시작 기본값. **4키 유지.** 시각적으로도 구분된다: Q = **은색 링 정상 확장**, 투자 0인 W/E/R = **탈색 링**(§7.5).

### 4.4 ★ 속성 각인 규칙 (stamp mode) — 하나

**정의**: 각인은 **"피해를 주는 개체"** 단위로 결정된다. 무기 엔티티가 아니라 데미지 소스가 기준이다.

| `elementStampMode` | 의미 | 해당 패밀리 |
|---|---|---|
| **`spawn`** | **피해 개체가 생성되는 순간의 스탠스로 각인, 이후 불변** | `forward` `fan` `seeker` `lance` `mine` `boomerang` `barrage` `omni` `drone` `nova` |
| **`live`** | **플레이어에 부착되어 지속하는 피해 개체. 매 적용마다 현재 스탠스 재평가** | `orbit` `aura` |

- **`spawn`이 막는 것**: "불 기뢰 대량 설치 → 물 전환 → 전부 물" 같은 각인 세탁. 기뢰·부메랑·포격 착탄은 전부 설치/발사 시점 각인.
- **`drone`이 `spawn`인 이유 (초안 C·F 충돌의 해소)**: 드론 **본체는 피해를 주지 않는다.** 피해를 주는 것은 드론이 쏜 **탄**이고, 그 탄은 발사 순간 각인된다. 즉 "드론은 매 틱 현재 스탠스를 반영한다"(F)와 "드론 탄은 spawn 각인"(C)은 **같은 동작의 두 가지 서술**이었다. 정본은 데미지 소스 기준이므로 `spawn`이다.
- **`nova`**: 폭발 시점 각인 (폭발이 곧 피해 개체).
- `elementStampMode`는 **패밀리별 구조 결정 = 잠금 키**다. 밸런싱 대상이 아니다.
- **I-2 (No Lying Color) 직접 귀결**: 화면에 그려진 색 = 그 개체가 지금 적용할 배율. `spawn` 각인 기뢰는 스탠스를 바꿔도 **색이 그대로다**(배율도 그대로이므로). 색만 바뀌거나 배율만 바뀌면 **구현 버그**다.

### 5.6 QWER 할당의 근거 (LOCKED 확인)

1. **Q = 노말 = 홈/기본.** 런 시작 스탠스이자 "모르겠으면 Q" 안전 바닥.
2. ★ **상성 니모닉: "적 속성 키의 *오른쪽 이웃*이 정답 키. R 다음은 W로 순환."**

   | 적 속성 | 그 키 | 오른쪽 이웃 | = 눌러야 할 키 | 검증 |
   |---|---|---|---|---|
   | 불 | W | E | **E (물)** | 물 → 불 ×2 ✔ |
   | 물 | E | R | **R (풀)** | 풀 → 물 ×2 ✔ |
   | 풀 | R | (순환) W | **W (불)** | 불 → 풀 ×2 ✔ |

   상성 사이클이 **키 배열의 물리적 순환에 그대로 사상된다** → 상성표를 외울 필요 없이 손가락으로 외운다.
3. `1234` 폐기 — 홈 포지션에서 손을 위로 뻗어야 하고, 드래프트 3택 선택(`Digit1~3`)과 정면 충돌.
4. `WASD` 폐기 — `W`가 스탠스 키와 물리 충돌. 이동은 방향키 확정.
5. `Space` = 폭탄(엄지, **손이 전혀 안 움직임**) / `Shift` = 토큰(새끼, **약간 움직임**) — 패닉 vs 신중의 **의도된 비대칭**.
6. **모드별 배타**로 오조작 원천 차단: 인게임에서 `1/2/3`·`F`·`Tab`은 죽은 키, 드래프트에서 `QWER`·`Space`·`Shift`는 죽은 키.

### 5.7 입력 처리 규칙

- **매 고정 틱마다 키 상태를 폴링**한다 (keydown 이벤트 큐 아님). 고정 타임스텝·헤드리스 시뮬 재현성의 전제.
- 스탠스 키 동시 입력 → **마지막에 눌린 키** 채택.
- 게임이 사용하는 모든 키에 `preventDefault` (특히 `Space`, `Tab`, 방향키).
- **브라우저 포커스 상실(`blur`) → 자동 일시정지 + `acc = 0`** (`input.pauseOnBlur = true`). 세이브 없는 30분 런이 알트탭으로 죽는 것은 회복 불가능한 손실.

## 6. 난이도 · 시간

### 6.1 배속의 구현 — **틱 소비율 (dt 스케일 아님)**

```
TICK_HZ = 60          // 잠금. 영원히 변하지 않음
TICK_DT = 1/60 게임초  // 잠금. 배속과 무관한 상수
```

| 난이도 | `difficulty[].speed` | 실제 1초당 소비 틱 | dt | 보스 타이머(실제) | 런 길이(실제) | `difficultyMul` |
|---|---|---|---|---|---|---|
| **Normal** | 1.0 | 60 | 1/60 | 3:00 | ~30분 | **1.0** |
| **Hard** | 1.5 | 90 | 1/60 | 2:00 | ~20분 | **1.6** |
| **Hell** | 2.0 | 120 | 1/60 | 1:30 | ~15분 | **2.5** |
| **Disaster** | 3.0 | 180 | 1/60 | 1:00 | ~10분 | **4.0** |

**★ dt 스케일은 존재하지 않는다.** 왜:
- dt를 ×3 하면 탄이 1틱에 13px 이동 → 반경 4px 플레이어를 **관통(터널링)**한다.
- 같은 시드가 난이도마다 다른 결과를 내 **시뮬 인증이 무효**가 된다.
- 틱 수 스케일은 dt가 상수라 **판정이 난이도 불변**이고, **게임시간 시뮬 1회가 4난이도 전부의 물리를 동시에 인증**한다.
- → 난이도 차이가 발생하는 유일한 지점 = **§10.4 봇의 반응지연(실제 ms → 틱 환산)**. 이것이 "난이도 = 인간 반응시간뿐"의 정확한 기계적 표현이다.

**배속이 코드에 존재하는 곳 = 정확히 1곳** (`main.js`의 `tickDur`):
```js
const tickDur = 1000 / (TICK_HZ * speed);            // 실제 ms
acc += Math.min(now - last, rules.loop.maxFrameGapMs);
let steps = 0;
while (acc >= tickDur && steps < rules.loop.maxStepsPerFrame) { step(world, pollInput()); acc -= tickDur; steps++; }
if (acc >= tickDur) acc = 0;                          // 나선형 죽음 방지: 남은 시간 폐기(빨리감기 금지)
render(world, acc / tickDur);                         // alpha 보간은 위치만. 로직 금지
```
`loop.maxStepsPerFrame = 6`, `loop.maxFrameGapMs = 250`.

- `src/core/**`는 `speed`를 **모른다.** 배속은 core 바깥의 루프 속도일 뿐이다.
- 렌더 보간은 위치 lerp만. 파티클·데미지 숫자는 **core 바깥**(결정론 무관).
- **난이도는 런 시작 후 변경 불가.** 4종 전부 처음부터 해금(해금 그라인드 = 기둥 위반).

### 6.2 게임시간 vs 실제시간 경계 (완전 목록)

| 사용 | 위치 | 이유 |
|---|---|---|
| **게임초** | 모든 JSON `*Sec` 키, 모든 판정·회피 예산·타이머·i-frame·상태이상·점수의 시간 항 | 배속 = 게임 클럭 하나 원칙 |
| **실제 ms** | ① `loop.maxFrameGapMs` ② `render.fxMinRealMs` ③ `bot.reactionMs` `bot.reactionJitterMs` `bot.stanceSwitchMs` | ① 루프 안전장치 ② 순수 장식이 고배속에서 소실되지 않게(판정 무관) ③ 시뮬 봇의 인간 모델 |

**회피 예산(리드타임·탄속·스폰 거리)은 전부 게임시간으로 정의되고 배속을 그대로 탄다. 실시간 하한 없음. 예외 없음.**
무-트위치 기둥은 **Normal에서 보장된다** — Hell/Disaster에서 어려워지는 것은 *읽고 반응할 시간*이지 *무엇인지 알아보는 것*이 아니다. 가독성 채널(색·모양·레이어·알파·오디오)은 배속과 무관하게 항상 동일하다.

- 타이머 정지 조건 = **PAUSE + 보스 페이즈 전환** 정확히 둘. (드래프트는 보스전에 발생하지 않는다 → §6.4.)
- 타이머 만료 = **즉사** (`boss.timerExpire = "kill"`). 1.5초 처형 연출 후 DEATH. 반격 불가 — 60초/30초 2단 경고가 그 반격 창이었다.

### 6.4 ★ 드래프트 발생 규칙 (초안 4종이 3개 답을 냈던 항목)

| 구간 | 드래프트 |
|---|---|
| **웨이브 구간** (0 ~ `crisisStartSec`) | **즉시 발생. 게임 클럭 완전 정지**(적·탄·스폰·페이즈 타이머 전부). 무제한 사고 시간. |
| **위기 세션** (95 ~ 120) | **큐에 적립.** HUD에 `LV UP ×N` 배지 |
| **보스 페이즈** | **구조적으로 발생 불가** (아래) + 만일을 위해 **큐에 적립** |
| **잡몹 페이즈 종료 직후** (보스 인트로 **전**) | **큐를 전부 순차 소화** |
| **STAGE_CLEAR 직후** | 남은 큐 순차 소화 |

**★ 보스전 중 레벨업이 구조적으로 불가능한 이유**: `boss.xp = 0` + `parts[].xp = 0` + **스테이지 보스는 잡몹을 소환하지 않는다**(`partType` 어휘에 소환 효과가 없음, §8.12) → **보스 페이즈에 XP 획득원이 존재하지 않는다.** 그리고 잡몹 페이즈 XP는 종료 시 전액 자동수집·정산되므로 보스 진입 시점에 이미 소진 상태다.

그러므로:
- "드래프트가 보스 타이머를 멈추는가"라는 구조적 충돌이 **소거**된다. 초안 F의 `pauseBossTimer` 논쟁은 **적용 대상이 없다.**
- 보스전 중 슬롯 재정렬도 **자동으로** 불가능해진다(재정렬 = 드래프트 화면 전용).
- 위기 세션의 큐 적립 근거: 밀도 최고 구간에서 3초마다 정지 = 리듬 파괴.
- 동시 다중 레벨업 = **병합 없이 순차 드래프트** (`xp.levelUpQueueMode = "serial"`).

### 6.5 전역 상태 기계 (확정)

```
TITLE ──Space──► DIFFICULTY ──Enter──► RUN_START ──► [스테이지 루프 ×6] ──► RESULTS
  │  └──O──► OPTIONS ──Esc──┘                                                  │
  └◄──Esc─────────────────────────────────────────────────────────────────────┘

스테이지 루프 (i = 1..6):
  THEME_BANNER ──► MOB_PHASE ──► DRAFT_QUEUE ──► BOSS_INTRO ──► BOSS ──► STAGE_CLEAR
                    │  └ 0~95s: DRAFT 즉시                        (격파)      │
                    │  └ 95~120s: CRISIS (내부 서브구간, DRAFT 큐)            │
                    └────────────────────────────────────────────────────────┘
  STAGE_CLEAR ──► DRAFT_QUEUE ──► HEAL ──► THEME_BANNER(i+1) ──► SHOP ──► MOB_PHASE(i+1)
  (i = 6이면 STAGE_CLEAR ──► RESULTS)

어디서든: Esc ──► PAUSE (게임 클럭·보스 타이머 정지) ──O──► OPTIONS(렌더/오디오만)
HP 0 또는 타이머 만료 ──► DEATH ──컨티뉴──► 직전 상태 재개 / ──포기──► RESULTS
RESULTS ──Space(같은 난이도 즉시 재시작)──► RUN_START │ ──Esc──► TITLE
```

- ★ **`CRISIS`는 독립 상태가 아니다.** `MOB_PHASE` 내부의 마지막 25초 서브구간이다. (초안 E의 다이어그램은 CRISIS를 별개 상태로 그렸으나 본문·D·타이밍 값 전부가 내부를 전제 — 내부로 확정.)
- 진입 상태 = `TITLE`. **스테이지 6 보스 격파 = 클리어(승리). 엔드리스 없음.** 새로고침/이탈 = 런 소멸(세이브 없음).
- 상점 방문 = **스테이지 1~5 클리어 후 총 5회.** 스테이지 1 시작 전 없음(코인 0), 스테이지 6 클리어 후 없음(런 종료).
- **전이 순서의 근거**: 빌드(DRAFT_QUEUE) → 실제 HP(HEAL) → 다음 위협(THEME_BANNER)을 **전부 알고 나서** 지갑을 연다(SHOP). 회복이 상점 뒤면 물약 구매가 도박, 테마가 상점 뒤면 유틸 구매가 장님.

### 6.6 일시정지

| 규칙 | 값 |
|---|---|
| 키 | `Escape` — 언제든(보스전 포함) |
| 게임 클럭 | **정지** (보스 타이머 포함) |
| 화면 내용 | **조회 전용** — 현재 빌드(무기 4슬롯 + 부여 상태), 속성 투자 + 합계 예산, **상성표 4×4**, 키캡 그림 |
| 금지 | 재정렬·구매·아이템 사용·드래프트 **전부 불가** |
| 허용 | `O` → OPTIONS (**렌더·오디오 옵션만**: mute / volume / cbMode / reduceFlash / screenShake). 게임 상태가 아니므로 안전 |
| 재개 | `Escape` → 1.5초 카운트다운 |
| 자동 | `blur` 시 자동 일시정지 |

> 게임 상태 변경이 불가하므로 일시정지는 이득을 만들지 못한다 → 무제한 허용해도 안전하다. 상성표 상시 비치 = 온보딩 비용 0.

## 7. 팔레트 · 시각 원칙 (단일 진실)

### 7.1 불변식 4개 (코드 리뷰 체크리스트)

| # | 불변식 | 의미 |
|---|---|---|
| **I-1** | **테두리가 있으면 나를 아프게 한다** | 검은 하드 외곽선(`palette.outline`)은 **적 탄·텔레그래프 전용.** 플레이어 탄·FX·픽업·오라는 외곽선 **금지**. |
| **I-2** | **색은 거짓말하지 않는다** | 화면에 그려진 속성색 = 그 개체가 **지금 적에게 적용할 배율.** 색이 바뀌면 배율도 바뀐 것이고 그 반대도 참. **크기도 포함**(히트박스 도트 r = 히트박스 r). 예외 없음. |
| **I-3** | **텔레그래프 없이는 발사 없다** | 텔레그래프 예산이 곧 공격 예산. 예산 초과 시 **공격을 연기**한다. "텔레그래프 생략하고 발사"는 어떤 상황에서도 금지. |
| **I-4** | **적 탄은 절대 가려지지 않는다** | 적 탄은 항상 최상단 게임플레이 레이어, 불투명, additive 금지. 플레이어의 어떤 이펙트도 적 탄을 덮을 수 없다(레이어 구조가 물리적으로 보장). |

**★ 텔레그래프는 노랑이 아니라 자홍이다 (확정).** 근거:
- **점선 = 미래, 실선 채움 = 현재.** 같은 색의 2상태 규칙 **하나**로 8종 거동 전부를 커버 → 학습 1회로 끝난다. 색을 나누면 "노란 게 뭐였더라"가 추가된다.
- 노랑(`#FFE24A`)은 호박(`#FFB000`) 상태이상 채널과 **명도·색상이 사실상 같다** → "내 상태가 나쁘다"와 "곧 아플 곳"이 섞인다. 자홍은 두 채널을 동시에 살린다.

**예약 규칙**
- 적 탄·텔레그래프에 **4속성 색 사용 절대 금지**.
- 하드 외곽선은 **적 탄·텔레그래프 전용**. 플레이어 요소 금지.
- 호박은 **플레이필드 밖(화면 가장자리 밴드) + 온-플레이어 아이콘에만**. 탄·적·배경 금지. → 코인 금색과 **영역이 분리**되어 충돌하지 않는다.
- 자홍 hue는 **배경 사용 절대 금지**.

### 7.3 색맹 안전 (`options.cbMode`)

**목표 선언**: 1차 = 적록색약(deuteranopia/protanopia, 남성 ~8%). 2차 = 청황 및 완전 색각이상 — **글리프 채널이 100% 커버**한다.

> **철학**: 4개 카테고리를 어두운 배경 위에서 **색만으로** 이색각자에게 분리시키는 것은 이론적으로 불가능하다. 목표는 **"색이 완전히 실패해도 게임이 100% 성립하는 것"**이고, 그 보증은 **글리프**다. 색맹 모드는 그 보증을 강제로 강화하는 스위치일 뿐이다.

| | `off` | `cvd` | `mono` |
|---|---|---|---|
| 팔레트 | `palette.element` | **`palette.elementCvd`** | `elementCvd` + 채도 0 |
| 글리프 크기 | ×1.0 | **×1.5** | ×1.5 |
| 글리프 렌더 | LOD 적용 | **항상 렌더** | 항상 |
| 적 외곽선 | 2px | **3px** | 3px |
| 플레이어 탄 | 속성색 + 글리프 | **글리프가 주(主)** | 글리프만 |
| 배경 명도 상한 | 0.28 | **0.22** | 0.22 |
| HUD/카드/고지 | 색 + 글리프 | + **텍스트 라벨 강제** | 글리프 + 라벨 |

**명도 사다리 팔레트** (`palette.elementCvd`) — 완전 흑백에서도 4단 분리:
```
normal #F2F6FA (L*96) │ fire #FFA83A (L*76) │ water #2E86E0 (L*55) │ grass #1F6B3A (L*40)
```
- 인접 간격 ≥ 15 L\*. 각 속성 탄은 **같은 hue의 밝은 코어(L\*+25)**를 가지므로 어두운 풀(L\*40)도 배경(≤22) 위에서 확실히 보인다.
- **6px에서도 100% 구분되는 4실루엣**(원/삼각/마름모/십자)만 채택. 육각형·별 등 "원과 헷갈리는 형태" 금지.

### 7.4 적 탄 · 텔레그래프 언어

**적 탄에는 속성이 없다(확정) → 적 탄은 단 하나의 정체성을 독점한다.**

| | 플레이어 탄 | **적 탄** |
|---|---|---|
| 색 | 4속성 hue | **자홍 고정** |
| 코어 | 같은 hue의 밝은 코어(L\*+25) | **흰 스페큘러 점** |
| 엣지 | 소프트(글로우) | **하드 + 검은 외곽선 2px** |
| 합성 | additive | **불투명(source-over)** |
| 알파 | ≤ 0.80 | **1.0 고정** |
| 레이어 | 4 | **9 (최상단 게임플레이)** |
| 모양 | 속성 글리프 (●▲◆✚) | **원 / 육각(상태이상)** |

> **3초 안에 배우는 한 문장: "부드럽게 빛나면 내 것, 테두리 있고 딱딱하면 적 것."**

**텔레그래프 2상태 (전 거동 공통)**

| 상태 | 표현 | 의미 |
|---|---|---|
| **곧 아플 곳** | **자홍 점선 외곽선 + 내부 알파 0.5, 채움(fill) 금지** | 텔레그래프 |
| **지금 아픈 것** | **자홍 채움 + 불투명 + 검은 외곽선** | 실탄 |
| 전환 순간 | **흰색 1프레임 플래시** | 발사 |

**거동별 텔레그래프 모양 (어휘 확정 — 애드혹 금지)**

| 이미터 / 상황 | 텔레그래프 모양 | `telegraphSec` 하한 |
|---|---|---|
| `straight` | 발사구 자홍 원 확대 | **0.55** |
| `fan` | **점선 부채꼴**(실제 각도 그대로) | 0.60 |
| `aimed` | **플레이어 위 조준 브래킷 4각 수축** | 0.60 |
| `ring` | 발사구 점선 원 | 0.60 |
| `spiral` | 발사구 점선 원 + 회전 화살 | 0.60 |
| `wall` | **점선 수평선 + 틈이 미리 비어 있음** | 0.80 |
| `zone` | **지면 점선 원 수축**(레이어 1) | 0.90 |
| `laser` | **점선 라인 → 폭 0→최종폭 보간** | **1.20** |
| 상태이상(slow) 탄 | 발사구 원 + **호박 테두리 링** | 0.80 |
| **상태이상(stun) 탄** | 발사구 + **화면 가장자리 호박 1회 사전 펄스** | **1.50** |
| 중간보스 패턴 | 전신 자홍 림 라이트 | 1.20 |
| **보스 대형 패턴** | **전신 광휘 + 패턴 형상 점선 프리뷰** | **1.50** |

- 각 거동은 **정확히 하나의 텔레그래프 모양**만 갖는다 → 스테이지·보스가 늘어도 학습이 이월된다.
- 배속 보정은 **렌더 전용**(게임플레이 무영향): `telegraph.emphasisBySpeed = {1.0: 1.0, 1.5: 1.15, 2.0: 1.30, 3.0: 1.50}` — 배속이 높을수록 **선 굵기·알파만** 증가.

### 7.5 스탠스 · 부여 판독 (기체 부착 3중)

"투자 N만큼 슬롯 순서대로 앞의 N개"는 인지적으로 추상적이다 → **눈이 있는 곳(기체)에 답이 있어야 한다.**

| 요소 | 규격 | 레이어 |
|---|---|---|
| **① 히트박스 코어 링** ★ | **흰 코어 r=4 (= 히트박스) + 현재 스탠스 색 링 stroke 2px (r 4→6)**. 투자 0인 스탠스면 **은색**(= 실제로 아무것도 부여 안 됨 → I-2 준수) | **10 (적 탄보다 위)** |
| **② 슬롯 스트립(임뷰 칩)** ★ | 기체 하단 **+14px**, pip **7px × 4칸**, 간격 2px (총 34px). **좌→우 = 슬롯 1~4 = 부여 우선순위.** 부여된 앞 N개 = 속성색 채움 + 글리프 / 나머지 = 은색 ● | **6 (적 탄보다 아래)** |
| **③ 상태이상 배지** | 기체 위 **−16px**. 아이콘 12px(∿ 둔화 / ✳ 스턴) + 잔여 감소 바 12×2px | **6 (적 탄보다 아래)** |
| ④ 기체 림 오라 | 현재 스탠스 색 소프트 글로우, 알파 0.40 | 6 |

**★ z-order 확정 (초안 A vs B의 정면 충돌)**
- **히트박스 도트 = 적 탄보다 위 (레이어 10).** 근거: 도트는 **r=4짜리 점 하나**라 은폐 면적이 물리적으로 무의미하고, 동시에 **회피의 전제**다 — 추적 대상이 안 보이면 관대함이 성립하지 않는다.
- **슬롯 스트립·상태 배지 = 적 탄보다 아래 (레이어 6).** 근거: 이들은 **면적이 있으므로** 탄을 가리면 I-4 위반이다.
- → 두 초안의 근거가 각각 옳았고, 대상이 다른 것이었다.

**전환 순간 피드백** (쿨다운 0 = 신뢰가 즉시 필요)

| 케이스 | 연출 |
|---|---|
| **투자 있는 속성으로 전환** | 기체 중심 **속성색 링 r 0→28px, 0.18초 확장 후 소멸** + 도트/오라/스트립 **즉시 갱신** + SFX(속성별 음정) |
| **투자 0인 속성으로 전환** ★ | 링이 속성색으로 시작 → **확장 중 회색으로 탈색**되며 소멸 + 스트립 전부 은색 유지 + **"빈" 저음** |
| **노말(Q) 스탠스** | **은색 링 정상 확장**(탈색 없음) + 스트립 전부 은색 |

> 투자 0 전환의 **탈색 링**이 "이 속성엔 투자가 없다"를 첫 판에 말없이 가르친다. 동시에 `Q`(정상 은색 링)와 `투자 0인 W/E/R`(탈색 링)이 시각적으로 구분되어 **Q키가 시각적 no-op가 아니게 된다.**

```
visual.stance = { ringExpandSec: 0.18, ringMaxRadiusPx: 28, ringStrokePx: 2,
                  emptyDesatSec: 0.18, dotRadiusPx: 4, dotRingPx: 2,
                  auraAlpha: 0.40, pipPx: 7, pipGapPx: 2, pipOffsetYPx: 14 }
```

### 7.6 적 속성 판독 (군중 속 <200ms)

| 채널 | 규칙 |
|---|---|
| **외곽선** | 속성색 2px (cvd 3px) — 본체 크기 무관 항상 |
| **코어 글리프** | 본체 중앙 속성 글리프. 크기 = `max(6, bodyPx × glyph.bodyRatio(0.42))`, 상한 22 |
| **림 라이트** | 진행 방향 반대쪽 속성색 소프트 림(알파 0.35) |
| **본체** | **중립 차콜 `#20242C` — 속성별로 본체를 칠하지 않는다** |

> **결정적 규칙**: 배경(저채도·저명도) ↔ 적 실루엣(명도 분리) ↔ 속성(고채도 라인 분리)의 **3층 분리** → **"붉은 배경 위 붉은 불 적"이 구조적으로 불가능**해진다.

- **겹침(새떼 70기)**: 외곽선이 각 개체를 분리, 글리프는 최상위 개체만 렌더(`glyph.occludedSkip = true`).
- **혼합 30% 적**: 테마와 다른 속성 = 다른 hue 외곽선 + 다른 실루엣 글리프 → 같은 색 군중 속에서 "다른 모양 테두리"로 튄다. 추가 표식 불필요.
- **엘리트 표식 (필수)**: **회전하는 이중 외곽선(속성색)** + 개체 위 **속성색 HP 바** + 상시 글리프. 중간보스와의 구분 = 엘리트는 HP 바가 개체에 붙고, 중간보스는 화면 상단 바.
- **보스 부위**: 속성색 외곽선 3px + 부위 중심 글리프 + 부위 위 소형 HP 바. **피격 귀속** = 맞은 부위 외곽선이 0.1초 밝게 점멸. **코어(항상 노말)** = 은색 외곽선 + ● 글리프. **파괴된 부위** = 외곽선 소멸 + 본체 `#3A3A3A` + 연기 + 글리프 제거.

### 7.7 히트 피드백 — ×2 / ×1 / ×0.5 는 **다른 감각**

| | **×2 (상성)** | **×1 (중립)** | **×0.5 (저항)** |
|---|---|---|---|
| 적 반응 | 속성색 화이트-핫 1프레임 + **0.04초 임팩트 프리즈(스케일 1.12)** | 짧은 백색 플래시 | **회색 방패 호(arc) 1회** |
| 파티클 | 속성색 3개, 큰 것 | 1개 | **0개** |
| 처치 FX | **속성색 대형 파열 + 흰 링 확장** | 소형 파열 | — |
| **SFX** | 저역 **"크런치"**(짧고 두꺼움) | 얇은 "틱" | **"틴"**(금속 튕김) |
| 숫자 | 속성색 ×1.5 크기, `×2` 접미 | 은색 ×1.0 | 회색 ×0.7, `×½` 접미 |

> **핵심**: ×0.5는 "약한 히트"가 아니라 **"튕겼다"**로 표현한다. 파티클 0 + 방패 호 + 금속음 → 플레이어가 **무기를 탓하지 않고 스탠스를 탓한다.** 정보 루프가 여기서 닫힌다.
> **혼재 히트**(1번 무기 ×2, 3번 무기 ×1이 같은 프레임): 배율 마커는 **그 프레임의 최고 배율 1개만**(`markerPolicy = "maxOnly"`).

**★ 데미지 숫자 — 잡몹에는 없다 (확정).** 밀도 붕괴 1순위 원인(4무기 × 30적 = 초당 수백 개 = 적 탄 은폐 = I-4 위반).
```
visual.hitFx = { numberTargets: ["boss","midboss","elite"], numberAggregateSec: 0.15,
                 numberMinPx: 14, numberOutlinePx: 2, numberLifeSec: 0.7, numberDriftPx: 18,
                 markerPolicy: "maxOnly", markerCooldownSecPerEntity: 0.25,
                 superFreezeSec: 0.04, superFreezeScale: 1.12,
                 particles: { super: 3, neutral: 1, resist: 0 } }
```
- 숫자는 0.15초 윈도우로 **개체별 합산 후 1개만 팝** → 보스전에서도 숫자가 4개를 넘지 않는다.
- 임팩트 프리즈는 **개체 단위 렌더 연출**이며 게임 클럭을 멈추지 않는다(보스 타이머·결정성 무영향).
- `damageNumbers` 옵션 토글은 **없다** — 잡몹 숫자가 애초에 없으므로 밀도 문제가 존재하지 않는다.

### 7.8 픽업 판독 (유도탄과 절대 안 헷갈리게)

| | 픽업 | 유도탄 |
|---|---|---|
| 색 | 금 / 백 | 자홍 |
| 외곽선 | **없음** | 검은 2px |
| 글로우 | **없음(무광·납작)** | 있음 + 잔상 꼬리 |
| 레이어 | 2 (바닥 쪽) | 9 (최상단) |
| 크기 | ≤ 6px | ≥ 8px |
| **운동** ★ | **자석 반경 밖: 플레이어를 절대 안 쫓음**(스크롤 따라 하강만, 아레나 하단 경계에서 **정지·누적**). **반경 안: 직선 스트릭으로 0.2초 내 도달** | **느리게 곡선으로 추적** |

> **한 문장: "천천히 곡선으로 쫓아오면 100% 탄. 순식간에 직선으로 빨려오면 100% 보상."** 운동 특성이 겹치는 구간이 **존재하지 않는다.**

- ★ **초안 F의 `orbAutoHomeDelaySec: 3.0` / `orbAutoHomeSpeed: 55`는 폐기한다.** "3초 후 반경 무관 느린 추적"은 위 표에서 **유도탄의 정의 그 자체**이며, 픽업/유도탄 분리 장치를 붕괴시킨다.
- 픽업은 **아레나 밖으로 나가지 않는다** — 하단 경계에서 정지·누적. 만료 없음(`pickup.lifetimeSec = null`).
- **페이즈 종료 시 반경 무시하고 전량 자동수집** (`stage.phaseEndAutocollect = true`) → 화면 밖 개념 자체가 없고, 무-노가다 기둥과 "전환 손실 0"이 동시에 성립.
- **자석 반경 가시화**: 기체 주위 은색 점선 원, 알파 0.12 상시 + 반경 증가 구매 시 0.5초 확장 연출. (상점에서 사는 것이 보여야 한다.)
- `player.magnetRadius = 90` (상점 `magnet` +30% × 3스택 → 최대 171).

### 7.9 배경 · 광과민 · 텍스트

| 규칙 | 값 |
|---|---|
| 배경 채도/명도 상한 | 0.25 / 0.28 (cvd·mono에서 0.22) |
| 배경 hue | 4속성 hue 사용 가능하나 **위 상한 내에서만** (화산 = 붉은 차콜, 늪·숲 = 녹회색 차콜). **자홍 금지** |
| 패럴랙스 | `bg.parallaxLayers = 2`, `bg.maxScrollSpeed = 90` (PxSec) |
| 배경 대비 | **엣지 대비 금지(소프트만)** → 하드 엣지 = 게임플레이 오브젝트 전용 |

> ★ 초안 B의 "배경 스크롤 90 ≤ 적 탄 최저속(≥140)의 절반" 규칙은 **폐기**한다. B가 인용한 `140`은 B 자신의 `enemyBulletMinSpawnRadiusPx`(**반지름**)이며 B는 최저 탄속 키를 정의한 적이 없다(B 내부 오류). **배경과 탄의 분리는 속도가 아니라 hue(자홍 금지) + 엣지(소프트만) + 레이어(0)가 보증한다** — B §4 자신이 이미 그렇게 썼다. 속도 기반 규칙은 불필요하고 존재하지 않는다.

```
                fullscreenFlashMaxPerSec: 3, fullscreenFlashMaxAlpha: 0.35 }
```

- 풀스크린 플래시는 **초당 3회 · 알파 0.35 하드캡**(폭탄·노바·피격 전부 이 예산 공유).

### 8.2 속성 혼합 — 70/10/10/10 (확정)

테마 속성을 `T`, 상성상 `T`를 이기는 속성을 `counter`(= 플레이어의 정답 스탠스), `T`가 이기는 속성을 `prey`라 할 때:

| 구간 | 값 | 정답 스탠스(`counter`)로 때릴 때 |
|---|---|---|
| 테마 `T` | **0.70** | **×2** |
| `counter` | 0.10 | ×1 |
| `prey` | 0.10 | **×0.5** ← 특화의 세금 |
| 노말 | 0.10 | ×1 |

- 예) 화산(불): 불 70 / 물 10 / 풀 10 / 노말 10. 정답 = **물 스탠스**.
- **데이터 표현**: `stages[].mix`는 **4속성 실제 키로 전개된 가중치 맵 하나**다. 화산이면 `{fire: 0.70, water: 0.10, grass: 0.10, normal: 0.10}`. counter/prey 규칙은 **빌드타임 생성 규칙**이며 `check.mjs`가 준수를 검증한다. → **스키마 하나, 예외 0.**
- **비율의 기준 = 스테이지 잡몹 총 개체 수** (HP 총량 아님, 스폰 확률 아님).
- **롤 granularity = 웨이브 단위** (`stages[].mixGranularity = "perWave"`, 개체 단위 롤 금지). 웨이브 리스트는 **빌드타임에 결정론적으로 편성**되어 스테이지 총합이 오차 **±3%p** 안에 들어간다 → **테마 고지가 준 정보가 실제로 참**이 된다.
- **혼합 비율 제외 대상**: 엘리트 · 중간보스 · 새떼 · 보스 (각자 별도 속성 규칙).
- **테마 고지에는 이 4개 수치가 그대로 표시된다** (대략이 아님). 초안 E의 `70/12/12/6`, F의 임의 데이터는 폐기.

**★ 온보딩이 여기서 공짜로 해결된다**: 스테이지 1은 `introOk` 테마 + 70/10/10/10이므로, **한 스탠스를 켜고 있으면 ×2(70%)와 ×0.5(10%)가 자연히 같은 화면에 나온다.** 초안 E의 "좌 레인 = 테마 / 우 레인 = counter" **대비 웨이브는 폐기**한다 — 그것은 ±3%p 게이트를 위반해 스테이지 1을 로드 실패시키고, **혼합 규칙 자체가 이미 대비를 만들기 때문에 불필요하다.** 새 규칙 0, 게이트 위반 0.

### 8.3 스테이지 진행 곡선 — 난이도(배속)와 완전히 별개

> 난이도 = 배속 하나뿐이라는 원칙은 **런 간(across runs)** 규칙이다. 아래는 **런 내(within a run)** 진행 곡선이며 모든 난이도에서 동일하다. 두 축은 곱해질 뿐 서로를 침범하지 않는다.

| 스테이지 | `enemyHpScale` | `xpScale` | `spawnDensityScale` | `midBossCount` | `elitePerWaveChance` | `swarmTotalScale` | `rearSpawnAllowed` |
|---|---|---|---|---|---|---|---|
| 1 | 1.0 | 1.0 | 0.70 | 1 | 0.10 | **0.5** | false |
| 2 | 1.5 | 1.6 | 0.85 | 1 | 0.15 | **0.7** | false |
| 3 | 2.2 | 2.4 | 1.00 | 2 | 0.20 | **0.85** | true |
| 4 | 3.2 | 3.4 | 1.15 | 2 | 0.25 | **1.0** | true |
| 5 | 4.5 | 4.6 | 1.30 | 2 | 0.30 | **1.0** | true |
| 6 (최종) | 6.0 | 6.0 | 1.50 | 2 | 0.35 | **1.0** | true |

- **적 HP 스테이지 스케일은 이 테이블이 유일한 소유자다.** 초안 F의 개체별 `hpScalePerStage: 0.18`(선형, 스테이지 6 = 1.9배)은 **폐기** — 15 아키타입 × 개별 계수는 곡선을 튜닝 불가능하게 만든다.
- `xpScale`은 **초안 D의 미결(XP 곡선이 후반에 마르는 문제)을 닫는다.** `enemyHpScale`과 짝지어 레벨업 횟수를 50~60으로 유지하는 축.
- ★ **`swarmTotalScale`은 여기서 정의된다.** 초안 D §5.2가 참조만 하고 값이 없던 열 + 초안 E의 `stage1CrisisScale: 0.5`는 **같은 개념의 두 이름**이었다 → **하나로 통합.**
- **후반 = 스펀지 방지**: HP만 올리면 후반이 무르다 → **아키타입 해금**(`unlockStageMin`)이 함께 올라 *다른 적*이 나온다. 스테이지 로스터 교체는 **없음**(콘텐츠 물량 5배 방지).

### 8.4 적 이동 어휘 `moveId` (8종, 동결)

- `charge.windUpSec ≥ fairness.minTelegraphSec`.

### 8.5 ★ 이미터 어휘 `emitterType` (8종, 동결) — 유도·상태이상은 **탄 속성**이다

| `type` | 파라미터 | 텔레그래프 하한 |
|---|---|---|
| `straight` | `count, spreadDeg, speed` | 0.55 |
| `fan` | `count, arcDeg, speed` | 0.60 |
| `aimed` | `count, spreadDeg, speed, leadSec` | 0.60 |
| `ring` | `count, speed, rotOffsetDeg` | 0.60 |
| `spiral` | `count, speed, rotStepDeg, durationSec, rateSec` | 0.60 |
| `laser` | `chargeSec, widthPx, activeSec, angleDeg, trackDuringCharge` | 1.20 |
| `zone` | `radius, activeSec, dmg` | 0.90 |
| `wall` | `count, gapCount, gapWidthPx, speed` | 0.80 |

공통 파라미터: `type, bulletId, from, telegraphSec, everySec, offsetSec, repeat, restSec`

**★ 초안 D의 6종 어휘(`straight fan homing laser zone status`)는 폐기하고 F의 분해를 채택한다.** 근거:
- **`homing`은 이미터가 아니라 탄의 속성**이다 → `bullets[].turnRateDegSec > 0`.
- **`status`도 이미터가 아니라 탄의 속성**이다 → `bullets[].status ∈ {null, "slow", "stun"}`.
- 이 분해로 D의 모든 콘텐츠가 표현되면서(`straight` 이미터 + `homingBullet`) **`aimed`/`ring`/`spiral`/`wall` 4종이 공짜로 추가**되어 보스 탄막 표현력이 늘어난다. 어휘가 늘지만 **코드 함수 8개는 D의 6개 + 2개**이며 전부 데이터 구동.
- **사격 안 함** = `enemies[].attack: null` (`none` 타입은 존재하지 않는다).
- ★ **`zone`의 `dps`는 존재하지 않는다.** 장판도 "적용 1회 = `dmg`"이며 i-frame이 게이트한다(§2.4). **모든 피해는 적용 1회로 통일.**
- 이 이미터 엔진을 **플레이어 무기와 적이 공유**한다 (문서의 "탄 엔진 재사용" = 이 표).

### 8.6 밴드 · 아키타입 · 엘리트

**HP 밴드 (4종)** — `hp = archetype.hp × band.hpMult × enemyHpScale[stage]`

| `band` | `hpMult` | 크기 | `coinDropChance` | 역할 |
|---|---|---|---|---|
| `chaff` | 1.0 | 소 | **0.0** | 1~2히트, XP 사료 |
| `line` | 2.5 | 소중 | **0.0** | 편대 구성원 |
| `turret` | 6.0 | 중 | **0.15** | 화면 체류, 압박 유지 |
| `bruiser` | 12.0 | 대 | **0.15** | DPS 체크 |

> **코인 드랍 주체 확정**: `turret`·`bruiser`(확률) + 엘리트(확정) + 중간보스(확정) + 보스(확정). `chaff`·`line`은 **코인 0.** 이것이 LOCKED의 "일부 잡몹만"의 정의다.

**공용 아키타입 9종 + 테마 시그니처 6종 = 15종.** 전부 `(band × moveId × emitter)` 조합이므로 **엔진 코드 추가 0.**
- 공용: `drifter` `spitter` `columnAnt` `flanker` `hexer` `turretPod` `stalker` `mortarHulk` `rearDart`
- 시그니처: `sirenRay`(바다) `frostLance`(빙원) `magmaBomb`(화산) `dustRunner`(사막) `thornWeaver`(숲) `bogHexer`(늪)
- 새떼 전용 2종: `swarmChaff`(90%, 사격 안 함) `swarmLancer`(10%)
- **테마당 정확히 4종** (공용 3 + 시그니처 1) + 각 엔트리의 `unlockStageMin` → 같은 테마라도 스테이지 2에서 2종, 스테이지 5에서 4종.
- **`element`는 아키타입 필드가 아니다** — 웨이브 편성이 주입한다(같은 아키타입이 테마마다 다른 속성으로 등장).
- **로스터·시그니처의 구체 편성은 「적·스테이지」 섹션 소관**이며, 정본은 **어휘·규칙·개수만** 확정한다.

**★ 엘리트 = 접두(prefix) 플래그다 (확정)** — 별도 개체가 **아니다.**

| 키 | 값 |
|---|---|
| 지정 방식 | 웨이브 레코드의 `eliteIndex: int \| null` — 그 웨이브의 n번째 개체에 플래그 |
| `elite.perWaveMax` | 1 |
| `elite.perWaveChance` | 스테이지별 (§8.3), `rng.elite` |
| `elite.hpMult` | 4.0 |
| `elite.sizeMult` | 1.4 |
| `elite.contactDmgMul` | 1.5 |
| `elite.xpMult` | 6.0 |
| `elite.coin` | **3 (확정 드랍)** |
| `elite.healDropChance` | **0.12** ← LOCKED "드물게 엘리트 드랍"의 정의 |
| `elite.bandAllowed` | `["line", "turret", "bruiser"]` — `chaff` 엘리트 금지(허수아비) |
| `elite.elementAllowed` | `["fire", "water", "grass"]` — **노말 엘리트 금지**(상성 재미 0) |

- ★ 초안 F의 "`tier: "elite"` = 별도 로스터 엔트리"는 **폐기**한다. 근거: **콘텐츠 물량.** 접두 모델은 15 아키타입 그대로 엘리트를 전부 커버하고, 엔트리 모델은 테마당 엘리트 개체를 추가 저작해야 한다(4주 예산). 그리고 `enemies[].tier` 필드는 **삭제**된다 — `enemies.json`은 잡몹 아키타입 15 + 새떼 2 = 17종만 담는다.
- **엘리트 ≠ 중간보스**: 엘리트는 웨이브의 일부(이탈 타이머 없음, 화면을 지나감). 중간보스는 독립 이벤트 개체.

### 8.7 편대 · 스폰 · 웨이브 스케줄

**편대 어휘 `formationId` (6종, 동결)**: `lineH` `columnV`(column 전용) `vWedge` `arc` `pincer`(strafe 전용) `scatter`

**★ 웨이브 스케줄 = 절대 타임라인이 아니라 순서 리스트다 (확정 — blocker `waveClearAdvance` 시맨틱)**

```
waves: [ { formationId, archetypeId, count, element, spawnEdge, eliteIndex } ]   // 시각 없음, 순서만
```

| 규칙 | 값 |
|---|---|
| 스포너 | 다음 웨이브 소환 시각 = **`max(직전 스폰 시각 + waveIntervalSec, 직전 웨이브 전멸 시각)`** |
| **`waveClearAdvance` 의미** | ★ **남은 스케줄 전체를 압축한다.** "다음 1개만 당기고 나머지는 절대시각 유지"가 **아니다.** |
| `waveIntervalSec` | 9.0 |
| `mobPhaseMaxWaves` | 14 (도달 시 추가 스폰 정지 = 무한 파밍 방지) |
| `crisisSuspendsWaves` | **true** (95초에 웨이브 스포너 정지, 밀도 폭발 방지 = 관대함) |
| `spawnEdge` | `top` \| `left` \| `right` \| `bottom`. **`bottom`은 `rearSpawnAllowed[stage]`일 때만** |
| 동시 개체 오써링 상한 | `enemyConcurrentMax = 40`, 새떼 `swarmConcurrentMax = 70` |
| 초과 시 | **`defer`** — 스포너가 다음 틱에 재시도. **스킵 금지**(스킵하면 밀어붙이는 플레이가 웨이브를 공짜로 지우고 시뮬 재현성도 깨진다) |

> ★ **초안 D의 `at_gs` 절대시각 레코드와 F의 `waves[].atSec` 절대 타임라인은 둘 다 폐기한다.** 둘 다 `waveClearAdvance` 규칙을 **자료구조로 표현할 수 없었다**(비평가 지적). 순서 리스트 + 간격 모델이 규칙과 자료구조를 일치시킨다.
> 예외: `elites` / `midBoss` / `crisis`는 **시각 기반**(`atSec`)으로 남는다 — 이들은 웨이브 스포너와 독립된 이벤트이며 전멸 압축의 대상이 아니다.

### 8.8 잡몹 페이즈 — 파밍 = 진짜 선택

| 키 | 값 |
|---|---|
| `mobPhaseSec` | **120** (고정 게임시간 하드 타이머) |
| **`mobPhaseSkippable`** | **false** — 보스로 조기 진입 불가 |
| `mobPhaseExitFadeSec` | 3.0 (잔존 잡몹 페이드아웃 소멸) |
| `mobPhaseExitClearBullets` | true (보스 시작을 공정하게) |
| **`phaseEndAutocollect`** | **true** — 미수집 XP·코인은 **반경 무시하고 전부 자동 수집** |
| `enemyExitForfeitsReward` | **true** — 화면 밖으로 나간 적의 XP·코인은 **영구 소멸** |

> ★ 초안 F의 `skippable: true` + "남은 적은 소멸하고 XP는 자동 정산 없이 소멸"은 **폐기**한다. 후자는 **무-노가다 기둥을 정면으로 위반**한다(비평가 지적).

**리스크가 무엇인지 (메커니즘, 선언 아님)**

| 장치 | 규칙 | 만드는 선택 |
|---|---|---|
| **① 이탈 = XP 영구 소멸** | `enemyExitForfeitsReward` | 잡으려면 **위로 올라가 요격**해야 함 → 상단 = 반응 여유 최소 = **위치 리스크.** 트위치가 아니라 "얼마나 앞에 설 것인가"의 각오 |
| **② 전멸 = 즉시 다음 웨이브** | `waveClearAdvance` (§8.7) | **플레이어가 자기 화면 밀도를 스스로 올린다** |
| **③ 고XP 도망자** | `dustRunner` (XP ×5, 초고속 `dive`) | 등장 즉시 위로 뛰어야 잡힘 |
| **④ 중간보스 병행 압박** | 교전 중에도 웨이브는 계속 온다 | 큰 XP vs 이중 압박 |

> **이 구조가 "잘 골랐나"를 3분 뒤 보스 타이머로 채점한다.** 파밍 부족 = 언더레벨 = 타이머 실패. 파밍 과다 = 잡몹 페이즈에서 죽음. **둘 다 선택의 결과이지 운이 아니다.**

### 8.9 중간보스

- **단일 몸체. 부위 없음.** 3종 **전 테마 공용** (`bosses.json`, `tier: "mid"`, `parts: []`, `phases` 1개).
- ★ **소속 파일 확정 (blocker)**: **`bosses.json`.** 근거: 중간보스는 **인트로·이탈 타이머·페이즈**를 가지므로 적 스키마로 표현 불가. `enemies[].tier`는 삭제되었으므로 `enemies.json`에 중간보스는 존재하지 않는다.
- 초안 F의 테마 전용 `kraken_mini`는 폐기 (콘텐츠 물량 6배).

| `midBossId` | 이름 | `moveId` | 이미터 | 시험하는 것 |
|---|---|---|---|---|
| `mbHammer` | 파쇄추 | `anchor`(좌우 큰 왕복) | `fan` + `zone` 교대 | 공간 압박 — 위치 판단 |
| `mbLancer` | 창병 | `charge`(주기적 돌진) | `laser` | 라인 회피 — 텔레그래프 읽기 |
| `mbNest` | 산란모함 | `anchor`(느린 하강) | `aimed`(유도탄) + **잡몹 소환** | **DPS 체크 — 보스전 예고편** |

| 키 | 값 |
|---|---|
| `midBossCount` | 스테이지별 (1~2, §8.3) |
| `midBossAtSec` | `[35]` / `[30, 70]` |
| hp | `midBoss.hp × enemyHpScale[stage]` |
| **`midBossLeaveAfterSec`** | **30** ← **"선택적"의 정의** |
| `midBossElementRule` | **`"notThemeAndNotNormal"`** (= `counter` 또는 `prey`) |
| `midBossForcedLeaveOnCrisis` | true |
| `midBossSummonsAllowed` | **`mbNest`만** (중간보스는 잡몹 페이즈에 있으므로 XP 획득이 정상) |
| 보상 | `xp` = chaff XP × 25 / `coin` = **5**(확정) / `healDropChance` = **0.35** ← 회복 3채널 중 "드랍" 채널의 주 수도꼭지 / `score` |

**"선택적"의 정확한 의미**: 등장 후 30 게임초가 지나면 **화면 위로 이탈해 사라진다.** 이탈 = 보상 0. **무시해도 쫓아오지 않는다**(추격 금지 — 그러면 "선택"이 거짓이 됨). 다만 **이탈 전까지는 계속 사격한다** → 무시 = "30초간 추가 탄막을 회피하며 웨이브를 파밍한다"는 선택. **공짜 회피가 아니다.**

**속성 규칙 = 보스전의 예고편**: 테마 정답 스탠스(예: 화산에서 물)를 켜둔 채로는 중간보스가 잘 안 죽는다 → **스탠스를 바꾸게 만든다.** 스테이지 보스가 요구할 전환을 잡몹 페이즈에서 미리 연습시키는 장치.

### 8.10 위기 세션 (새떼) — 화력 페이오프

| 키 | 값 |
|---|---|
| `crisisPerStage` | **1** (매 스테이지 필수) |
| `crisisStartSec` / `crisisDurationSec` / `crisisWarnSec` | 95 / **25** / **3.0** |
| `crisisSuspendsWaves` | true |
| `crisisTotal` | **60 × `swarmTotalScale[stage]`** |
| `crisisSubWaves` | 6 (10기 × 6파, `arc`/`vWedge`) |
| `crisisFailCondition` | **없음** |
| `crisisElementRule` | **`"themePure"`** — 테마 속성 **100%** (혼합 규칙의 명시적 예외) |
| 보상 | `swarmXp` = chaff XP × 0.5 → 60기 = **스테이지 총 XP의 ~25%** |

- ★ **이것이 새떼 설계의 핵심**: 60기 전부가 테마 속성 → **정답 스탠스에 투자한 플레이어는 25초 동안 ×2로 전부 갈아버린다.** 투자가 화면 가득한 폭발로 되돌아오는 순간 = "선택이 만든 양학"이 가장 크게 보이는 장면 = **제출 영상의 클라이맥스.** 투자를 안 한 플레이어는 ×1로 밀려난다 — **처벌이 아니라 정보**다.
- **실패해도 죽지 않는다**: 종료 시 잔존 새떼 전원 소멸. 하단으로 물러나 지나가길 기다릴 수 있다 → XP 포기가 유일한 대가.
- **막다른 운 없음 (확정)**: 새떼의 정답은 3개 — ① `nova`(캡스톤 정답) ② `aura`(밀집 학살) ③ **폭탄**(잡몹 광역 즉사급 + 탄 소거, 상점에서 항상 구매 가능). → **무기 드래프트 운과 무관하게 보편 해답이 항상 존재한다.** 새떼는 "특정 무기를 못 뽑으면 죽는 시험지"가 아니라 **"화력을 준비했는가"**의 시험지다. (인증: `crisisClearWithoutCapstone ≥ 0.80`)
- 초안 F의 `durationSec: 18, warnSec: 2.5`는 폐기 (D·E 일치).

### 8.11 복합 보스 — 구조

> **모든 스테이지 보스는 복합. 중간보스는 단일. 예외 없음.**

| 키 | 값 |
|---|---|
| `boss.partCount` | **4** (최종만 5) |
| `boss.partRegen` | **false** — 부위 파괴는 영구. `parts[].regenSec`는 **존재하지 않는다** |
| `boss.summonsAllowed` | **false** — 스테이지 보스는 잡몹을 소환하지 않는다 |
| `boss.partHitPriority` | `"outermostFirst"` — 히트박스가 겹치면 바깥 부위 우선. 부위 파괴 시 히트박스가 제거되어 **자동으로 코어가 노출**(별도 규칙 불필요) |
| **HP 구조** | ★ **코어 HP = 보스 HP.** `core.hp` + `parts[].hp` **절대값**(합산 아님, `hpShare` 없음) → **3분 타이머가 재는 것 = 코어 DPS**가 명확해진다 |
| `boss.phaseThresholds` | **`[0.6, 0.3]`** — **코어 HP** 기준 3페이즈 |
| `boss.introSec` | 3.0 (등장 연출 중 보스 무적) |
| `boss.timerStartsAfterIntro` | true (연출은 3분에서 차감 안 됨) |
| 부위 XP / 보스 XP | **0 / 0** (§6.4의 전제) |
| 부위 score | 부위는 **개체**이며 자체 `score`를 갖고 **상성 처치 보너스 대상**이다 |
| 드랍 | `boss.coin` = **12**(확정) / `boss.partCoin` = **2**/파괴한 주변부 1개당 / `boss.healDrop` = **없음** |
| 폭탄 | `bomb.bossDmgRatio`는 **코어 최대 HP 기준**, **부위에는 적용 안 함**. 상한 `bomb.bossDmgCap` |

- ★ 초안 D의 `hp_total` + `hp_share` 합계 1.0 모델은 폐기 — 절대값이 시뮬의 부위별 독립 튜닝을 가능하게 하고, 검증 항목이 하나 줄어든다.
- 페이즈 = **코어 HP 임계만** 사용(부위 파괴와 무관) → 규칙 하나, 조합 폭발 없음. 페이즈마다 각 부위의 `patternSet` 인덱스가 교체된다(`everySec` 단축, `count`/`arcDeg` 증가). **새 이미터 타입이 생기지는 않는다.**

### 8.12 ★ 부위 타입 어휘 `partType` (4종, 동결) — 애드혹 보스 금지

| `partType` | 파괴 시 효과 (**규칙이며 보스별 특수 코드 아님**) | 개수 |
|---|---|---|
| `mobility` | 보스 이동 속도 × `boss.mobilityPenalty`(**0.5**), 회피 기동 패턴 중단 | 선택 |
| `armament` | **그 부위가 담당하던 이미터가 영구 삭제** | 선택 |
| `armor` | **코어 피해 게이트 1단계 해제** (§8.13) | **1~2개** |
| `core` | **보스 사망** | **정확히 1개** |

**이 4개가 전부다.** 6개 보스가 전부 이 조합이므로 **보스별 특수 코드 = 0.**

> ★ 초안 F의 `onDestroy: [{op, v}]` 6종 어휘(`moveSpeedMul dmgMul emitterOff spawnWave phaseAdvance openCore`)는 **폐기**한다. 근거: ① 타입별 고정 효과가 규칙 하나로 끝나고 조합 폭발이 없다 ② F의 `spawnWave`는 `boss.summonsAllowed: false`와 **직접 충돌**하며, 그 충돌이 §6.4(보스전 중 레벨업 구조적 불가)의 전제를 깬다 ③ `op` 목록은 AI가 임의 조합해 보스마다 다른 규칙을 만들 수 있다.

### 8.13 ★ 코어 소프트 게이트 (설계의 급소)

> **질문**: 부위를 무시하고 코어만 때려 죽일 수 있는가?
> **답**: 때릴 수는 있다. 하지만 산술적으로 3분 안에 못 죽인다.

```
코어_피해배율 = boss.coreGateMul ^ (살아있는 armor 부위 수)
boss.coreGateMul = 0.4
```

| 살아있는 `armor` | 코어 피해 배율 |
|---|---|
| 2 | **×0.16** |
| 1 | ×0.40 |
| 0 | **×1.00** |

- **강제 게이팅이 아니라 배율** → "코어 직행" 자유는 존재하되 명백히 손해. **소프트 게이트**이므로 플레이어가 규칙을 몰라도 벽에 부딪히지 않고 *느려짐으로* 배운다(정보 전달이 곧 피드백).
- ★ **`mobility`·`armament` 파괴는 코어 배율에 영향이 없다.** → **3분 타이머 하의 진짜 트레이드오프**: *"시간을 써서 무장을 깨고 안전하게 갈 것인가, `armor`만 깨고 코어로 직행할 것인가."* 이것이 보스전의 결정이다.
- ★ 초안 F의 `damageTakenMulWhilePartsAlive: 0.25`(**부위 종류 무관, 하나라도 살아있으면**)는 **폐기**한다. 그 규칙은 mobility 부위를 안 깨면 영원히 ×0.25이므로 **모든 부위 파괴를 강제**하고, 위 트레이드오프를 통째로 소멸시킨다 — D가 명시적으로 배제한 것을 게이트에 넣은 것이다.

### 8.14 부위 속성 배정 규칙 R1~R6 (AI 하드 제약 — **전부 `check.mjs`가 강제**)

| # | 규칙 | 키 | 이유 |
|---|---|---|---|
| **R1** | `core` 속성 = **항상 노말** | `boss.coreElement = "normal"` | **브릭 방지** — 어떤 스탠스로도 ×1 마무리 가능 |
| **R2** | 주변부 속성에 **노말 금지** | `boss.partNormalForbidden = true` | 모든 주변부는 상성 대상 |
| **R3** | 주변부는 **서로 다른 속성 ≥ 2종** | `boss.partElementDistinctMin = 2` | **스탠스 1개 샌드백 금지** |
| **R4** | **테마 속성은 최대 1개 부위** | `boss.partThemeElementMax = 1` | 보스 = 테마의 반복이 아니라 **커버리지 시험** |
| **R5** | **`armor` 부위 속성 ≠ 테마 속성** | `boss.armorElementNotTheme = true` | 가장 오래 때리는 부위가 **다른 스탠스를 강제** |
| **R6** | `armor` 부위 수 ∈ {1, 2} | `boss.armorPartCountRange = [1, 2]` | 게이트 강도 |

> **R4 + R5의 결합이 이 게임의 셀링 포인트를 규칙으로 보장한다**: 테마 정답 스탠스 하나만 갈고 온 특화 빌드는 **보스에서 반드시 스탠스를 바꿔야 한다.** 6개 보스를 AI가 생성해도 전부 스탠스 퍼즐이 됨이 보장된다.
> ★ 초안 F의 검증 게이트는 R3만 강제했다 → **R1·R2·R4·R5·R6이 검증기에 없어 AI 생성물이 D의 의도를 합법적으로 우회**할 수 있었다. 정본은 **R1~R6 전부를 `check.mjs`에 넣는다.**

### 8.15 「강철 가오리 (Manta)」 — 정본 판본 (바다/물 테마)

| 부위 | `partType` | 속성 | **정답 스탠스** | 파괴 효과 |
|---|---|---|---|---|
| 스러스터 | `mobility` | **물** | 풀 | 이동 ×0.5, 좌우 회피 기동 중단 |
| 좌 지느러미 | `armor` | 불 | **물** | 코어 게이트 1/2 해제 |
| 우 지느러미 | `armor` | 풀 | **불** | 코어 게이트 2/2 해제 |

| **로스터** | 공용 9 + 시그니처 6 = **15종 총출동** (지나온 5개 테마의 회고) |

| **클리어** | 테트라크 격파 = **런 클리어**(종료 조건) |

> ★ 초안 F의 `stages[].element ∈ {water, fire, grass}` **스키마 강제**는 최종을 **표현조차 못 했다**(비평가 지적). 정본은 `element: string | null` + `null`은 `finale`만 + `mix`를 4속성 가중치 맵으로 통일해 **예외 없이** 표현한다.

**「테트라크 (TETRARCH)」 5부위**

| 부위 | `partType` | 속성 | **정답 스탠스** | 파괴 효과 |
|---|---|---|---|---|
| 물의 관 | `armor` | 물 | **풀** | 게이트 1/3 |
| 불의 관 | `armor` | 불 | **물** | 게이트 2/3 |
| 풀의 관 | `armor` | 풀 | **불** | 게이트 3/3 |
| 왕좌(포대) | `armament` | **노말** | 없음(항상 ×1) | 유도탄 이미터 영구 삭제 |
| 핵 | `core` | 노말 | 아무거나 | **사망** |

- **armor 3개 → 코어 ×0.4³ = ×0.064.** 코어 직행 완전 불가 → **3속성 전부를 반드시 깨야 한다 = 3속성 투자의 졸업 시험.**
- **왕좌(노말)** = 상성이 통하지 않는 유일한 주변부 → **"항상 ×1인 안전 바닥"의 존재 증명 부위.**
- **고유 연출 (규칙)**: 관이 하나 파괴될 때마다 아레나 배경이 그 속성의 색을 잃는다 → 3개 다 깨지면 **무채색 아레나에서 핵과 단둘이** = 시각적 클라이맥스.
- **특화 빌드도 클리어 가능 (막다른 운 없음의 수치 정의)**: 불+4 단일 특화는 풀의 관에 ×2, 나머지 두 관에는 노말 스탠스로 ×1(불 스탠스로 ×0.5를 맞느니 Q가 낫다 — **Q의 존재 이유가 여기서 증명된다**). `finale` 코어·부위 HP는 **"단일 속성 +4 특화 빌드가 두 관을 ×1로 밀어도 3분 안에 격파 가능"**한 값으로 시뮬 튜닝한다 → **특화 = 아슬아슬하게 성공, 분산 = 여유롭게 성공.**

### 9.5 `weapons.json` — 코드/데이터 힌지

**구조**: 12 패밀리 = **코드의 update 함수** (`src/core/weapons/<family>.js`, 각 30~60줄). **DSL 없음**(DSL은 4주 예산에서 몇 주짜리 항목). 각 패밀리는 **폐쇄된 파라미터 계약**을 선언하고 모든 수치를 JSON에서 읽는다.

```json

  "weapons": [{
    "id": "seeker", "family": "seeker", "name": "시커", "desc": "최근접 적을 자동 추적",
    "elementStampMode": "spawn",
    "base": { "dmg":6, "cooldownSec":0.9, "count":1, "projSpeed":190, "projRadius":5,
              "lifetimeSec":2.4, "pierce":0, "hitCooldownSec":0.2,
              "turnRateDegSec":220, "acquireRadius":320, "retargetSec":0.3,
              "targetMode":"nearest" },
    "levels": [ {}, {"dmg":8}, {"count":2,"dmg":9}, {"dmg":12,"turnRateDegSec":260},
                {"count":3}, {"dmg":16,"cooldownSec":0.75}, {"pierce":1},
                {"count":4,"dmg":22} ],

```

| 규칙 | 내용 |
|---|---|
| **id 네임스페이스** | ★ **`id == family`** (12종 1:1). `forward fan seeker lance orbit aura mine boomerang barrage omni drone nova`. 초안 C의 `blitz/fanout/pulse/return/rearguard/option` 네임스페이스는 **폐기.** `startWeaponId = "forward"` |
| `levels` | **정확히 8행**, `base`에 대한 부분 오버라이드 (§9.3의 유일한 예외) |
| 레벨업 = 파라미터 델타 | 거동 변화도 파라미터로만 (`pierce: 0→1`, `count: 1→3`). **Lv1~7의 레벨업은 계약 안의 숫자만 바꾼다** |
| **★ 진화 (확정)** | **Lv7 → Lv8 레벨업 카드 그 자체가 진화 카드다.** 자동도, 별도 카테고리도, 패시브 페어링도 아니다 |
| **★ 진화의 코드 표현 (blocker)** | 각 패밀리의 update 함수는 **`w.evolved` 불리언 하나를 읽는다.** 진화 = 같은 함수 안의 `if (w.evolved)` 분기 **정확히 1개** + `evolution.params`의 **`evo*` 접두 파라미터**(계약에 선언됨). **패밀리당 ~15줄 × 12 = 약 180줄이 4주 예산의 확정 입력값이다.** |
| `family` 변경 | **금지.** 진화해도 슬롯의 패밀리 정체성은 유지된다. 초안 F의 `evolution.family` 변경(seeker→drone)은 폐기 — 계약이 두 개 겹치고 슬롯 정체성이 몰래 바뀐다 |
| 진화 `flags` | **폐기.** 초안 C의 `flags: ["distinct_targets", ...]` 임의 문자열은 AI가 발명할 수 있다 → `evolution.params`의 선언된 `evo*` 키로만 |
| 불가역 / 슬롯 | 진화체는 **같은 슬롯 인덱스 유지** → 슬롯 순서·임뷰 계산에 특별 취급 없음. 스탠스 임뷰도 동일하게 받는다 |
| Lv 종료 | Lv8에서 종료. Lv9 없음 → `weaponLevel` 풀에서 제외 |

**패밀리 파라미터 계약 (폐쇄 목록)** — 공통: `dmg cooldownSec count projSpeed projRadius lifetimeSec pierce hitCooldownSec targetMode`

|---|---|---|---|---|
| `forward` | 벌컨 | `spreadDeg jitterDeg burstCount burstIntervalSec` `evoRampSec evoRampFireRateMul` | `forward` | spawn |
| `fan` | 팬아웃 | `arcDeg` `evoBlastRadius` | `forward` | spawn |
| `seeker` | 시커 | `turnRateDegSec acquireRadius retargetSec` `evoDistinctTargets evoRetargetOnKill` | `nearest lowestHp randomInArena` | spawn |
| `lance` | 랜스 | `beamWidthPx chargeSec rangePx` `evoFullHeight` | `forward nearest` | spawn |
| `orbit` | 오빗 | `orbitRadius angularSpeedDegSec bodyCount` `evoBulletClearCooldownSec` | — | **live** |
| `aura` | 펄스필드 | `radius tickIntervalSec falloff` `evoPullForce` | — | **live** |
| `mine` | 마인필드 | `placeIntervalSec armSec triggerRadius blastRadius maxAlive` `evoClusterCount evoClusterRadius` | — | spawn |
| `boomerang` | 리턴 | `outRangePx returnSpeed canRehit` `evoChainCount` | `forward nearest` | spawn |
| `barrage` | 바라지 | `strikeIntervalSec strikesPerVolley blastRadius telegraphSec` `evoRadiusMul` | `randomInArena densest` | spawn |

**★ `targetMode` 어휘 (폐쇄)**: `forward` `nearest` `lowestHp` `densest` `randomInArena`

- ★ **`bossPartPriority`는 존재하지 않는다 (확정).** 초안 F의 "복합 보스에서 부위를 지정할 수단을 데이터로 제공"은 **"의미 있는 이동" 기둥의 정면 위반**이다 — 자동 타겟이 스탠스 퍼즐을 대신 풀어버린다. **보스 부위는 각각 독립 타겟 엔티티이고 `nearest` 계열은 가장 가까운 부위를 노린다 → 플레이어가 위치로 부위를 고른다.**
- ★ **`knockback`은 존재하지 않는다 (확정, 미결 해소).** 초안 C·F 양쪽 계약에 있었으나 **모델이 없었다**(단위·스크립트 이동에의 적용·편대 붕괴·보스 적용 전부 미정). 스크립트 경로를 도는 적(`moveId`)에 넉백을 적용하면 `column`·`anchor`·`pincer` 편대가 깨져 **§8.4의 이동 어휘 전체와 충돌**한다. 어휘에 남겨두면 AI가 의미 없는 값을 생성한다. **삭제가 가장 싸고 안전하다.**

| 삭제 | 이유 |
|---|---|
| `critChance` `critMul` | **크리티컬 없음(확정, §3.1).** 식에 항이 없으므로 스탯도 없다 |
| `defenseAdd` | 방어력은 **상점 전용, 상한 8**(§2.1). 패시브로 열면 칩딜 완전 무효화 경로가 열린다 |
| `statusResistMul` | 상태이상 저항은 **상점 전용 비-빌드 유틸**(LOCKED). 패시브면 빌드 축이 된다 |
| `iframeSecAdd` | 초안 A가 "체감 약해 폐기"한 것 — 관대함의 산술 보증(§2.1)이 `iframeSec = 1.0` 고정에 의존 |
| `durationMul` `bombDmgMul` `projSpeedMul` `magnetRadiusMul` | 훅 12개 = 패시브 12개의 1:1을 유지. 자석은 상점, 폭탄 데미지는 봉인 대상 |

**★ `resonance`의 함정 (비평가가 찾은 것) — 해소 완료**: `elementBonusMul`은 `elem = 1 + (elem−1) × k`의 **k**다. 초안 C의 `values [2.1 … 2.5]`(= 결과 배율)를 이 슬롯에 그대로 넣으면 **×3.1~3.5**가 된다. 정본의 `values`는 **k로 재작성**되어 `[1.10 … 1.50]` → 유효 상성 배율 **×2.1 ~ ×2.5** = C의 원래 의도 그대로.
**★ `bulkhead`의 스케일 붕괴 (비평가가 찾은 것) — 해소 완료**: 초안 C의 `+1/+1/+2/+2/+3`은 HP ~10 규모를 전제했으나 정본 `hpMax = 100`이다. `[6,12,18,24,30]`으로 재작성(Lv5 = +30%).

**★ `afterimage`(고스트)와 이미 발사된 유도탄 (미결 해소)**: **적용된다.** 고스트 중에는 **유도탄이 재조준을 멈추고 마지막 방향으로 직진한다.** 근거: "적의 조준·유도 타겟에서 제외"라는 문구와 동작이 일치해야 한다(I-2의 정신). 표현 수단: `bullets[].retargetSec` 키가 존재하므로 "재조준 대상" 개념이 스키마에 실재한다 — 고스트는 **재조준 시점에 타겟 후보에서 플레이어를 뺀다.** (밸런서가 규칙이 아니라 값을 만지게 된다.)

### 9.7 `bullets.json` · `enemies.json`

```json
// bullets.json
{ "schemaVersion": 1,
  "bullets": [ { "id":"roundM", "radius":7, "hitboxScale":0.8, "speed":140, "dmg":8,
                 "shape":"circle", "status":null, "statusDurationSec":0,
                 "accel":0, "turnRateDegSec":0, "retargetSec":0 } ] }
```

- ★ **`element` 키가 존재하지 않는다** — **스키마가 "적 공격에는 속성이 없다"를 강제한다.** 색은 `palette.threat.enemyBullet` 단일.
- `status ∈ {null, "slow", "stun"}`. `stun`은 `fairness.minStunTelegraphSec`(1.5) + `maxStunSec`(1.0) 강제 + `difficulty.stunMinDifficulty = "hard"`.
- `turnRateDegSec > 0` = **유도탄** (이미터 타입이 아니라 탄 속성, §8.5).
- ★ **적 탄 데미지의 소유자 = `bullets[].dmg` (확정).** 초안 A의 전역 티어 6종(`enemyBullet.damageSmall: 8` 등)은 **저작 가이드**로만 남고 **코드 키가 아니다**(소형 8 / 중형 14 / 대형·레이저 22 / 장판 10 / 상태이상 6). 초안 D의 `attack_params`에는 damage 필드가 아예 없었다 → 이미터 모델로 대체되며 소멸.

### 9.10 ★ 아트 방침 — 절차적 도형 (확정, 미결 해소)

**스프라이트 에셋은 존재하지 않는다.** 초안 F의 `spriteId: "drifter"`는 **스프라이트 레지스트리·에셋 파일·아틀라스를 정의한 섹션이 없는 죽은 참조**였다. 정본은 초안 B의 절차적 모델을 채택한다.

| 항목 | 확정 |
|---|---|
| 적·보스 본체 | **코드가 그리는 도형**: 중립 차콜 `#20242C` 본체 + 속성색 외곽선 2px + 코어 글리프 + 림 라이트 (§7.6) |
| **도형 어휘 `shapeId` (12종, 동결)** | `wedge` `delta` `hexPod` `orb` `cross` `spike` `ring` `slab` `fin` `claw` `dart` `bulb` |
| 적 데이터 | `{ shapeId, radius }` 두 필드. `sizePx`는 `band`가, 색은 `element`가 결정 |
| 플레이어 탄 12+12 | **속성 글리프(●▲◆✚) × `projRadius` × 궤적.** 무기별 별도 아트 **0** — §7.2의 글리프 규칙이 이미 정체성을 보증한다 |
| 진화체 | 같은 글리프, `projRadius`·`count`가 커지고 속성색이 같음 → **별도 아트 0** |
| 보스 6종 | **부위 도형 조합** (`shapeId` + `anchor` + `radius`) → **별도 아트 0** |
| 배경 | 절차적 (저채도·저명도 차콜 + 소프트 엣지 2레이어 패럴랙스) |
| **총 아트 에셋** | **0 바이트** — 400KB 예산의 대부분이 JSON과 코드로 간다 |

> 근거: ① 400KB 예산 ② 4주 솔로에 스프라이트 15+12+12+6종 저작 시간이 없다 ③ **AI 콘텐츠 파이프라인이 도형 어휘를 데이터로 생성할 수 있다**(픽셀 아트는 못 한다) ④ B의 3층 분리(배경/실루엣/속성 라인)가 도형을 전제로 설계되었다.

## 10. 결정성

### 10.1 고정 타임스텝

```
TICK_HZ = 60          // 잠금
TICK_DT = 1/60 게임초  // 잠금. 배속과 무관한 상수 (§6.1)
```
- 배속은 **초당 소비 틱 수**이며 `main.js`의 `tickDur` **1곳**에만 존재한다. `src/core/**`는 `speed`를 모른다.
- 입력은 **틱마다 키상태 폴링**(이벤트 큐 아님).
- 렌더 보간은 **위치 lerp만**. 파티클·데미지 숫자·임팩트 프리즈는 **core 바깥**(결정론 무관).

### 10.2 시드 RNG

```js
// src/core/rng.js — sfc32. 정수 연산만 → 모든 엔진에서 비트 동일
export function makeRng(seed)              // → { u32(), f(), range(a,b), int(a,b), pick(arr), weighted(map), shuffle(arr) }
export function stream(masterSeed, name)   // → makeRng(hash32(masterSeed, name))
```

**독립 스트림 (7종, 동결)** — 한 스트림에 draw를 추가해도 다른 스트림이 밀리지 않는다. 이것이 없으면 **콘텐츠를 하나 고칠 때마다 이전 시뮬 인증이 전부 무효**가 된다.

| 스트림 | 용도 |
|---|---|
| `theme` | 5개 테마 순서. **런 시작 시 한 번에 전부 추첨** |
| `draft` | 드래프트 후보 추첨 · 리롤 |
| `spawn` | 웨이브 스폰 · 편대 배치 |
| `elite` | 엘리트 접두 부여 |
| `drop` | 코인 · 회복 드랍 |
| `pattern` | 탄막 jitter |
| `boss` | 보스 페이즈 내 선택 |

- ★ **`shop` 스트림은 존재하지 않는다** — 상점은 **전체 목록 상시**이며 랜덤 진열이 없다(§11.2).
- 마스터 시드 = uint32. 브라우저: `(Date.now() ^ (performance.now() × 1000)) >>> 0` — **비결정성이 들어오는 유일한 지점이며 core 바깥(`main.js`)에서 생성해 core로 주입**한다. 시뮬: 명시 시드.
- 시드는 결과 화면에 **8자리 hex**로 표시(오락실 감성 + 버그 리포트 재현, 비용 0).
- `Math.random`은 core에서 **금지**(정적 검사).
- **데미지 경로에서 RNG를 호출하지 않는다** (크리티컬 없음, 데미지 난수 없음 → §3.1).

### 10.3 결정성 범위 (명시적 선언)

| 수준 | 보장 | 근거 |
|---|---|---|
| **L1 콘텐츠** | 같은 시드 → 같은 테마 순서·웨이브·드래프트 후보. **엔진 무관 보장** | RNG가 정수 연산뿐 |
| **L2 런** | 같은 시드 + 같은 입력열 → 같은 결과. **동일 엔진 내 보장** | 아래 규칙 |
| **L3 리플레이** | 엔진 간 비트 동일 — **비보장·비목표** | `Math.sin/cos` ULP 차이. 리플레이 기능 없음, 시뮬 인증은 통계적이라 무관 |

**L2 유지 규칙 (전부 `check.mjs` 또는 리뷰가 강제)**
- 사전할당 풀 + `alive` 플래그, **인덱스 오름차순 순회만**
- 게임플레이 로직에서 `Map`/`Set` 순회 금지, `for...in` 금지
- `sort` 사용 시 반드시 `id` **총순서 타이브레이크**
- core 내 객체 생성 금지 (**핫패스 0 bytes/tick**)
- **오버플로 정책이 결정적**이어야 한다(§12.1) — `evictOldest`는 순수 FX에만 허용

### 10.4 헤드리스 시뮬이 동일 로직을 재사용하는 방법

```bash
node tools/check.mjs                                   # 순수성·스키마·공정성·구조 게이트
node tools/sim.mjs --runs 8000 --seed 0 --out tools/report/
node tools/sim.mjs --certify                           # 임계값 판정, 실패 시 exit 1
```

**★ 재사용 방식**: `sim.mjs`가 `../src/core/step.js`를 **그대로 import** 한다. 렌더·오디오·DOM은 **애초에 core에 없으므로 스텁조차 필요 없다.** 루프는 `while (!world.over) step(world, bot(world))` — rAF도 시계도 없이 게임시간만 흐른다.
> **시뮬이 인증하는 코드 = 배포되는 코드 (문자 그대로 동일 파일).** 이것이 §9.1 모듈 경계 규칙의 존재 이유다.

**봇 모델 (`meta.json > bot`)**
```json
"bot": {
  "reactionMs": 250, "reactionJitterMs": 80, "stanceSwitchMs": 180,
  "dodgeLookaheadSec": 0.8, "aimErrorPx": 12, "grazeTolerancePx": 4,
  "policies": {
    "draft":  ["generalist","specialist","weaponRush","elementRush","greedyDps","random"],
    "farm":   ["maxFarm","balanced","passive"],
    "stance": ["greedyNearest","majorityOnScreen","static"]
  },
  "baseline": { "draft":"generalist", "farm":"balanced", "stance":"greedyNearest" },
  "probes":   { "dpsProbe": true, "forceNoElement": true }
}
```

★ **핵심**: 반응 지연은 **실제 ms로 선언하고 난이도 배속으로 게임틱에 환산**한다.
```
latencyTicks = round(reactionMs / 1000 × TICK_HZ × speed)
// Normal ×1 → 15틱 / Disaster ×3 → 45틱 동안 눈이 감긴 채 진행
```
이것이 없으면 게임시간 시뮬은 4난이도에서 **비트 동일**해 난이도를 아무것도 증명하지 못한다. **이 한 줄이 "난이도 = 인간 반응시간뿐"을 검증 가능한 모델로 만든다.**

**★ 정책은 3개의 직교 축이다 (게이트 설계의 전제)**: `draft` × `farm` × `stance`. `noDeadLuck.minRunClearWorstDraftPolicy`는 **draft 축만** 순회하며 farm·stance는 baseline으로 고정한다. (이것이 없으면 `stance: "static"` 봇이 draft 게이트를 실패시킨다 — 축이 다른데 같이 세면 게이트가 자기모순이 된다.)

**★ 두 개의 시뮬 모드 (게이트 상호 만족의 열쇠, §13)**

| 모드 | 무엇을 재는가 | 봇 |
|---|---|---|
| **`run`** (기본) | **런 클리어율** — 6스테이지 생존 + 격파 전부 | 정상 (사망함) |
| **`dpsProbe`** | ★ **보스 DPS 시험 통과율** — 보스 i를 3분 안에 코어 격파할 **화력이 있는가.** 사망은 판정 대상이 아니다 | **무적**, 회피 로직은 그대로 |

> 이 분리가 없으면 "보스당 격파율"과 "런 클리어율"이 같은 지표로 오인되어 `p⁶` 합성 모순이 발생한다(비평가 지적). **두 지표는 다른 것을 재며 서로 합성되지 않는다.** §13에서 동시 만족 가능성을 계산으로 보인다.

`summary.json`(인증 지표 전부) · `weapons.csv`(픽률·승률·DPS 기여) · `elements.csv` · `bosses.csv`(격파 시간 분포·타임아웃률) · **`stages.csv`(★ `stagePar[i]` = 스테이지별 표준 보스 잔여시간)** · `economy.csv`(스테이지별 코인 수급/지출) · `deaths.csv`(사인 분류: HP / 타이머)

> ★ **`stagePar[i]` 미결 해소**: 초안 E가 `bossParGhost`(라이브 페이스 고스트)와 결과 화면 파 막대의 **입력으로 계약 요구**했으나 초안 F의 산출물 목록에 없었다. 정본은 **`report/stages.csv`를 산출물에 추가**하고, `--certify` 통과 시 그 중앙값이 **`meta.json > flow.stagePar[6]`에 기입**된다. E의 "이유 모를 죽음" 제거 장치 2개가 데이터 소스를 갖는다.

**개발 흐름**: AI가 JSON 생성 → `node tools/check.mjs` → `node tools/sim.mjs --certify` → **통과 시에만 커밋.** 어느 하나라도 실패하면 커밋하지 않는다. **이 파이프라인 자체가 "AI가 설계·밸런싱·공정성 인증"이라는 제출 스토리의 실물이다.**

## 11. 드래프트 · 상점 · 점수 — 코어 규칙

> **정본은 모델과 키를 확정한다. 세부 로스터·문구는 섹션 소관이다.**

### 11.1 드래프트 (`meta.json > draft`)

**뽑는 방식**: 카테고리를 먼저 뽑지 않는다. **유효 후보 아이템 전체를 가중치 목록으로 만들고 비복원 3장 추첨.**
```
weight(item) = categoryWeights[item.category] × modifier(item)
```

```json
"draft": {
  "optionCount": 3,
  "slotAssign": "append",
  "categoryWeights": { "newWeapon":20, "weaponLevel":40, "elementLevel":20, "passive":20 },
  "newWeaponSlotScale": [3.0, 1.5, 1.0],
  "weaponLevelEvolutionBonus": 2.0,
  "elementFirstLevelBonus": 1.5,
  "passiveNewBonus": 1.3,
  "distinctItemsPerDraft": true,
  "filterInvalid": true,
  "newWeaponWhenSlotsFull": "excluded",
  "elementLevelOfferRequiresWeaponCount": true,
  "guaranteeElementCardOnFirstDraft": true,
  "guaranteeNewWeaponUntilSlots": 2,
  "elementCardPity": 6,
  "reroll": { "granularity":"all3", "canRepeatPrevious":false, "maxPerDraft":2 },
  "fallback": { "id":"resupply", "name":"보급", "coins":40 },
  "pauseGame": true
}
```

| 결정 | 값 | 근거 |
|---|---|---|
| 가중치 모델 | **아이템 단위 + 동적 스케일** (초안 C) | `weaponLevel: 40`이 가장 큼 = 28픽 싱크가 가장 크므로 정합. 슬롯 상황에 따라 동적 → 초반엔 무기가 잘 뜨고 만석 후 자동으로 다른 축으로 흐름. 초안 F의 고정 비율 + `renormalizeOnEmptyCategory`는 폐기 |
| 피티 | **6** (C) — F의 8은 폐기 | 속성축이 운으로 통째 봉인되는 것을 더 빨리 차단 |
| 폴백 코인 | **40** (C·E 일치) — F의 25는 폐기 | |
| 드래프트당 리롤 | **`maxPerDraft: 2`** (C) — E·F의 무제한은 폐기 | 무제한이면 "풀 전체 검색" = 운 완화가 아니라 완전 제거 |
| **드래프트 중 클럭** | **완전 정지** (적·탄·스폰·페이즈 타이머 전부) | 무제한 사고 시간 = 무-트위치. `pauseBossTimer` 논쟁은 §6.4에 의해 **적용 대상이 없다** |

**유효성 필터 (무효 카드는 애초에 풀에 없음)**

| 상황 | 처리 |
|---|---|
| **무기 4칸 만석** | `newWeapon` 카테고리 **전체를 풀에서 제외. 교체 제안 없음** |
| 무기 Lv8 | 그 무기의 `weaponLevel` 카드 제외 |
| 패시브 6칸 만석 | 미보유 패시브 카드 제외 |
| 패시브 Lv5 | 그 패시브 카드 제외 |
| 속성이 `elementCapPerElement`(4) 도달 | 그 속성 카드 제외 |
| 속성 합계가 `elementCapTotal`(6) 도달 | **3속성 카드 전부 제외** |
| **속성 레벨 ≥ 현재 무기 수** | 그 속성 카드 제외 — **죽은 투자 차단** (무기 2개인데 `불+3`이 뜨면 정보 기반 선택의 신뢰가 깨진다) |
| 중복 | 한 드래프트의 3장은 **서로 다른 아이템**. 카테고리 중복은 허용 |

> **무기 교체를 거부하는 이유 (확정)**: ① 투자한 레벨·진화가 증발 = 게임 최악의 feel-bad ② 교체가 **슬롯 순서를 조용히 바꿔 임뷰가 바뀜** — 무관한 선택에서 핵심 시스템이 몰래 변함 = 정보 기반 선택 배신 ③ "버릴 것 고르기" UI + 확인 플로우 = 스코프. 4칸이 차기 전에는 `newWeaponSlotScale`이 획득을 강하게 밀어주므로 **만석 도달은 정상 진행이지 손해가 아님.**

**폴백 (`resupply`)**: 유효 후보가 3장 미만이면 부족분을 코인 +40 카드로 채운다. **절대 무효가 아니다**(남은 코인 = 점수). 빈 드래프트 화면이 **물리적으로 불가능**해진다. 성장 예산상 정상 런에서는 거의 안 뜨고, 뜬다면 "다 찍었다"는 뜻이라 페이오프 구간의 보상으로도 읽힌다.

**리롤 (밴시 폐기 후 유일한 운 완화 밸브)**

| 규칙 | 값 |
|---|---|
| 구매 / 사용 | **상점에서 스톡 구매 → 드래프트에서 `F`로 소비** |
| 대상 | **3장 전체 재추첨** |
| 이전 카드 재등장 | **금지** — 방금 본 3장은 그 드래프트 동안 풀에서 제외 |
| 한 드래프트당 | `maxPerDraft: 2` |
| 가격 상승 리셋 | 없음 (런 전체 누적) |

> 리롤이 **재도박**이면 운 완화가 아니다. 이전 3장을 제외해야 리롤이 **확정적인 개선**이 되어 "정보 기반 선택 > 운빨" 기둥을 실제로 지탱한다. 대신 2회/드래프트 상한으로 "풀 전체 검색"은 차단.

**카드가 반드시 표시하는 것 (정보 기둥의 집행 지점)**

| 카테고리 | 필수 표기 |
|---|---|
| `newWeapon` | 이름 · 거동 1줄 · **거동 미니 프리뷰(루프)** · **`→ 슬롯 N에 배치`** · 현재 스탠스에서 **임뷰되는지 여부** |
| `weaponLevel` | 대상 슬롯 · `Lv3 → Lv4` · **변하는 파라미터의 before → after** · Lv7 카드는 **`다음 = 진화` 마크** · Lv8 카드는 **진화 폼 프리뷰** |
| `elementLevel` | 속성 글리프 · `불 +2 → +3` · ★ **`전환 시 부여 무기: 2개 → 3개` + before/after 슬롯 그림** ← **이 한 줄이 슬롯 순서 규칙을 가르치는 유일한 지점** · 합계 예산 잔여 |
| `passive` | 이름 · 훅 1줄 · `Lv2 → Lv3` 수치 before → after |
| `resupply` | `+ 코인 40` |

**성장 예산 (레벨업 50~60회와의 대차대조표)**

| 투자처 | 픽 수 | 계산 |
|---|---|---|
| 새 무기 | **3** | 시작 무기 1 지급 → 4칸 중 3칸 |
| 무기 레벨 | **28** | 4무기 × Lv1→8 = 7픽 × 4 (**Lv8픽 = 진화**) |
| 속성 레벨 | **6** | `elementCapTotal` |
| 패시브 | **30** | 6칸 × Lv5 (획득 픽 포함: 6 + 24) |
| **총 싱크** | **67** | |
| **런 레벨업** | **~54** (`levelUpsPerRunTarget: 54`, 범위 50~60) | **충족률 ≈ 81%** |

→ **전부 못 찍는다**가 산술적으로 성립(54 < 67). 동시에 후반 풀이 마르지 않는다. 정적 검사: **`certify.growthBudget.maxLevelUps(60) < totalSink(67)`** ✔

### 11.2 상점 (`meta.json > shop`)

| 규칙 | 확정 | 근거 |
|---|---|---|
| **진열** | ★ **전체 목록 상시. 랜덤 진열 없음. 재고 제한 없음(`maxPurchases`만).** | **상점 항목 간 시너지가 0**이라 조합 최적화가 **수학적으로 발생할 수 없다** — 독립 가격 곡선 위의 항목별 그리디 결정뿐이며, 풀 최적해를 "푸는" 대상이 아니다. 유일한 결정은 *"코인 = 점수를 포기하고 살까"*. 그리고 **운 완화 밸브(리롤)가 랜덤 진열 뒤에 있으면 자기모순**이다. → 초안 E의 "고정 4 + 로테이션 2"는 존재하지 않는 위험(바둑/체스)에 대한 방어이므로 폐기 |
| 시간 제한 | **없음** | 무-트위치 |
| 열리는 시점 | **스테이지 1~5 클리어 후 총 5회** (§6.5의 전이 순서) | |
| 표시 필수 | 아이콘 · 이름 · 효과 1줄 · **보유/상한**(`2/3`) · **현재가** · **다음가** · 보유 코인 · ★ **`= −N 점` 환산 실시간 표시** | 환산값이 안 보이면 "아껴서 점수 vs 써서 생존" 긴장이 **플레이어에게 존재하지 않는다** |
| 가격 누적 | **항목별 독립 / 런 전체 누적** (스테이지 리셋 없음) | |

**가격 공식 (전 항목 공통, 이것만 코드에 있음)**
```
price(item, n) = ceil(item.basePrice × item.growth ^ n)     // n = 그 항목의 런 내 구매 횟수
```
> 초안 F의 `priceCurve: {type: "mul"|"add"}` 2종은 폐기 — 하나의 공식 + 항목별 `growth`로 동일한 튜닝 자유도를 얻는다(rule: 더 단순한 쪽).

**항목 (10종 확정)**

| 분류 | `id` | 효과 | `basePrice` | `growth` | `maxPurchases` | 스톡/스택 상한 |
|---|---|---|---|---|---|---|
| **선택권** | `reroll` | 드래프트 1회 재추첨 스톡 | 30 | 1.60 | 6 | `rerollStockMax: 3` |
| **생존/보험** | `potion` | HP `healPct` = **0.35** 즉시 회복 | 40 | 1.50 | 8 | — |
| | `bomb` | 폭탄 스톡 +1 | 50 | 1.35 | 5 | `bombStockMax: 3` |
| | `shield` | 피격 1회 무효 | 60 | 1.50 | 4 | **`shieldStockMax: 2`** |
| | `timeToken` | 보스 타이머 **+30 게임초** | 80 | 1.80 | 2 | `timeTokenStockMax: 2` |
| **비-빌드 유틸** | `defense` | 방어력 **+2** (정액 감산) | 45 | 1.45 | **4** → 상한 **8** | — |
| | `maxhp` | 최대 HP **+10** (구매 즉시 같은 양 회복) | 55 | 1.50 | **4** → 140 | — |
| | `movespeed` | 이동 속도 **+6%** | 55 | 1.50 | **3** → +18% | — |
| | `magnet` | 수집 반경 **+30%** | 40 | 1.40 | **3** → 171px | — |
| | `resist` | 상태이상 **지속시간 −20%** | 40 | 1.40 | **3** → −60% | — |
| **폐기** | ~~밴시~~ / ~~무적시간↑~~ / ~~무기·속성 레벨~~ / ~~상태이상 *해제*~~ | | | | | |

- ★ **컨티뉴는 상점에 없다 (확정).** **사망 시점에 제시**한다(오락실식, §11.4). 초안 C·F의 상점 판매 모델은 폐기.
- **`resist`는 저항만.** '해제'는 발동 키가 필요한데 양손(방향키 + QWER + Space + Shift)이 이미 만석 → **조작 설계 변경을 유발**한다. 저항은 영구 정률이라 키가 필요 없다.
- **모든 스탯 항목에 `maxPurchases`** → 무한 스택으로 **코인이 사실상 빌드를 사는** 대원칙 우회가 구조적으로 차단된다.
- **실드는 순차 소모** (1피격 = 1개, 틱당 1개). i-frame은 정상 발동(§2.4).

**희소성 설계 (항상 살짝 부족하게)**: 코인은 `turret`/`bruiser`(15%) + 엘리트(3) + 중간보스(5) + 보스(12 + 부위당 2)만 드랍. 가격 상승. ★ **남은 코인 = 런 종료 시 점수 환산** → 쓰면 점수를 잃는다. 모든 구매가 진짜 기회비용. (인증: `coinScarcity`)

### 11.3 점수 (`meta.json > score`)

```json
"score": {
  "superEffectiveDamageShare": 0.5,
  "superEffectiveKillBonusRatio": 0.5,
  "attribution": "damageShare",
  "timeBonusPerGameSec": 50,
  "bossClearBonus": 2000,
  "midBossClearBonus": 500,
  "runClearBonus": 10000,
  "noHitScope": "perStage", "stageNoHitBonus": 3000,
  "perfectScope": "perRun",  "perfectBonus": 30000,
  "shieldPreservesNoHit": true,
  "timeTokenForfeitsTimeBonus": true,
  "coinToScore": 20
}
```

| 항목 | **판정 규칙 (숫자가 아니라 규칙)** |
|---|---|
| 처치 점수 | 처치 즉시. **보스 부위도 개체** → 자체 `score` |
| ★ **상성 처치 보너스** | **막타 기준이 아니다.** 개체마다 `총 피해`·`×2 피해` **2개 누산기**. 처치 시 `×2 피해 / 총 피해 ≥ 0.5`면 성립 → 처치 점수 **× (1 + 0.5)** |
| **남은 시간 보너스** | **게임시간 잔여 초 × 50.** 게임시간 기준이므로 배속에서 이중 보상 없음. ★ **시간 토큰 사용 시 그 보스는 0** |
| 보스 격파 | 스테이지 보스 2000 / 중간보스 500 |
| ★ **무피격 보너스** | **스테이지 단위** — 그 스테이지 전체(잡몹 + 위기 + 보스)를 피격 0으로 클리어 시 3000 |
| ★ **퍼펙트 보너스** | **런 단위 = 6/6 스테이지 무피격 + 컨티뉴 미사용** → 30000 |
| 클리어 보너스 | 스테이지 6 보스 격파 → 10000 |
| 남은 코인 환산 | 런 종료 시 잔액 × **20** |
| **× 난이도 배율** | 위 전부의 **합계에 최종 곱** (1 / 1.6 / 2.5 / 4) |

**부속 판정 (전부 확정)**
- **실드로 막은 피격 = 피격 아님** (`shieldPreservesNoHit: true`). 실드의 문구가 "1회 피격 무효"이므로 무피격도 유지되어야 문구와 동작이 일치한다. 코인으로 무피격을 살 수 있지만 **코인 = 점수라서 자기상쇄**된다(실드 2개 ≈ 156코인 = 3120점 포기 vs 무피격 3000점).
- **사망해도 점수는 집계된다** — 그때까지의 획득 + 남은 코인 환산 + 난이도 배율 (오락실 규칙).
- ★ **왜 막타가 아니라 지분인가**: 막타 기준이면 최적 플레이가 *"막타만 ×2로 맞추기"*가 되어 사용자가 명시적으로 싫어하는 **체스식 최적화 + 트위치 정밀도**를 동시에 소환한다. 지분 기준이면 최적 플레이는 그냥 **"제대로 된 스탠스로 싸우기"** = 설계 의도 그 자체. 누산기 2개 = 구현 비용 0.
- ★ **무피격 스코프가 스테이지인 이유**: 런 전체 스코프는 "2스테이지에서 깨지면 나머지 28분이 무의미"라는 **feel-bad**를 만든다. 스테이지 단위면 매 스테이지가 살아있는 목표이고, 런 전체 목표는 `perfectBonus`가 **별도로** 담당한다. (초안 F의 `noHitScope: "perBoss"`는 잡몹 페이즈를 무피격 판정에서 빼버려 "스테이지를 완주했다"는 서사가 사라진다.)
- ★ **`timeTokenForfeitsTimeBonus`는 LOCKED다**: 대가가 없으면 **코인으로 3분 허들을 삭제**할 수 있다. 대가가 점수라서 토큰은 **"이 판을 포기할래 점수를 포기할래"**라는 진짜 결정이 된다. (초안 F의 score에 이 키가 없어 토큰이 공짜가 되는 구멍이 있었다 — 정본이 닫는다.)
- 초안 F의 `superEffectiveKillMul: 1.5`(배율)와 E의 `+0.5`(가산)는 **같은 결과**다. 정본은 `superEffectiveKillBonusRatio: 0.5` = 처치 점수의 0.5배를 **보너스로 가산**한다(총 ×1.5).

|---|---|
| 횟수 | **런당 1회 하드캡** |
| 비용 | **150 코인, 고정** (1회 캡이라 인상 불필요) |
| 부족 시 | 항목 회색 + 가격 표시 → 다음 런의 학습("아껴야 하는구나") |
| **카운트다운** | ★ **없음. 무기한 대기.** 오락실 감성은 카운트다운이 아니라 *동전을 낼 것인가*의 결정에서 온다. **카운트다운 = 트위치 = 기둥 위반** |
| 대상 | **두 사인 모두** (체력 소진 / 시간 초과) |

**부활 상태 (전부 확정)**

| 항목 | 값 |
|---|---|
| HP | 최대치 |
| 화면 | **적 탄 전량 소거**, 잡몹은 유지 |
| 위치 | 아레나 하단 중앙, **3.0 게임초 무적**(점멸) |
| 빌드 | 레벨·무기·속성·패시브·슬롯 순서·스탠스 **전부 유지** |
| 코인 / 스톡 | 150 차감, 나머지 유지 / 폭탄·실드·리롤·토큰 유지 |
| 잡몹 페이즈 사망 | 그 자리에서 재개. 페이즈 타이머 유지 |
| **보스 페이즈 사망** | **보스전 재개. 보스 HP는 깎은 그대로 유지.** 타이머 = **`max(현재 잔여, 60)`** |
| 타이머 만료 사망 | 동일 — 보스 HP 유지 + 타이머 60초 |
| **상실** | `perfectBonus` + **모든 스테이지의 `stageNoHitBonus`(런 전체 소급 무효)** + 코인 150 = 점수 −3000 자동 → **이중 비용** |

> ★ 타이머를 3:00으로 리셋하지 않는 것이 핵심. 리셋하면 컨티뉴는 "6분 보스전"이 되어 **게임의 유일한 허들이 코인으로 삭제**된다. **60초 = 마무리할 수 있는 시간이지, 다시 싸울 수 있는 시간이 아니다.**

**DEATH 화면**: ① **사인 명시** — `체력 소진` / `시간 초과` (아이콘 + 1단어). *왜 죽었는지 모르는 죽음을 만들지 않는다.* ② 도달 스테이지 / 경과 시간 / 점수 소계 ③ **스테이지별 보스 잔여시간 vs 파(par) 막대** — 어디서 뒤처지기 시작했는지가 그림으로 보임 ④ 컨티뉴 제시 / `포기 → RESULTS`

**RESULTS 화면**: 헤더 `CLEAR` / `GAME OVER — 체력 소진` / `GAME OVER — 시간 초과` · 항목별 집계 + 오락실식 순차 타이핑(Space 즉시 완료) · **놓친 보너스를 회색으로 함께 표시**(예: `무피격 3/6 · 퍼펙트 ✕ · 시간토큰 1회 → 시간 보너스 −1`) — **이 게임엔 언락이 없다. 결과 화면이 유일한 교사다.** · **파 막대 6개**(`flow.stagePar[i]` 고스트 라인) · 난이도별 최고점수 비교 + 신기록 연출 · 런 시드(8자리 hex) · `Space = 같은 난이도 즉시 재시작` / `Esc = 타이틀`

**보스 페이스 고스트**: 보스 HP 바에 **파 고스트 마커** — "지금쯤 여기까지 깎았어야 함"(`flow.stagePar[i]` 기반). 텍스트 0, 구현 비용 ~0, **20분 뒤에야 알게 되는 언더파를 지금 알려준다.**

**폭탄 발동 규칙 (미결 해소)**

| 항목 | 확정 |
|---|---|
| **캐스트** | ★ **없다.** 발동 즉시(다음 틱) 효과. 초안 F의 `castSec: 0.4`는 **폐기** — 폭탄은 패닉 버튼(LOCKED "살아남는 카드")이고, 0.4초 캐스트는 **패닉 순간에 타이밍을 요구**한다 = 트위치 = 기둥 위반 |
| 무적 | `bomb.iframeSec = 1.5` 게임초 |
| 이동 / 스탠스 전환 | **둘 다 자유** (잠기지 않음) |
| **스턴 중 발동** | **가능** (§2.7) |
| 발동 차단 | 드래프트 / 상점 / 일시정지 / 보스 인트로 / DEATH 중 **입력 무시**. 스톡 0이면 무시 |
| 효과 | 잡몹 `mobDmg: 9999`(광역 즉사급) + **적 탄 전량 소거**(보스전 포함) / 보스는 **코어 최대 HP의 4%**, 상한 220, **부위엔 미적용** |
| 시각 | 흰 플래시가 아니라 **자홍→흰 링 1회 확장** (소거되는 탄을 플래시가 가리는 자기모순 회피) |

## 12. 한계 · 캡 · 공정성 하한

### 12.1 ★ 2층 구조 — 오써링 예산 ≠ 안전망 캡

> **캡은 안전망이지 밸런스 노브가 아니다.** 오써링된 밀도 예산은 캡보다 낮아야 하며, 시뮬 인증은 **`capHits == 0`**을 요구한다. **캡에 닿는 콘텐츠는 콘텐츠 버그다.**
> 초안 B(320/220/90)·F(640/512/160)·D(적 40)의 3중 충돌은 **서로 다른 두 층을 같은 이름으로 부른 것**이었다. 정본은 두 층을 분리한다.

**A층 — 오써링 예산 (`rules.fairness`, `check.mjs`가 콘텐츠에 강제)**

| 키 | 값 | 강제 |
|---|---|---|
| `enemyConcurrentMax` | **40** | 웨이브 편성 정적 검사 |
| `swarmConcurrentMax` | **70** | 새떼 전용 예외 |
| `maxSimultaneousEnemyBullets` | **320** | 보스 `patternSet` 3페이즈 전개 시뮬 |
| `telegraphConcurrentMaxPerEntity` | **2** | 보스·중간보스·잡몹 **1개체** 기준 정적 검사 |

**B층 — 안전망 캡 (`rules.caps`, 엔진 풀 크기)**

| 키 | 값 | 초과 시 정책 |
|---|---|---|
| `enemies` | **96** | **`defer`** — 스포너가 다음 틱 재시도 (웨이브가 공짜로 사라지지 않음) |
| `enemyBullets` | **384** | **`rejectSpawn`** |
| `playerBullets` | **256** | **`rejectSpawn`** |
| `pickups` | **256** | **`merge`** — 신규 픽업 값을 최근접 기존 픽업에 합산 (**손실 0 = 무-노가다 기둥 보존**) |
| `zones` | 64 | `rejectSpawn` |
| `drones` | 8 | `rejectSpawn` |
| `particles` | 400 | `evictOldest` |
| **`telegraphs`** | **8** | ★ **`deferAttack` — 공격 자체를 연기** (I-3) |
| `damageNumbers` | 10 | `evictOldest` |
| `effectMarkers` | 12 | `evictOldest` |

**초과 정책의 규칙 (결정성 필수)**
- **게임플레이 개체(적 탄·플레이어 탄) 초과 = `rejectSpawn`. 오래된 것 재활용 절대 금지** — 날아가던 탄이 사라지면 화면이 거짓말한다(I-2 위반), 플레이어 발밑에 탄이 순간이동한다(불공정).
- **텔레그래프 초과 = `deferAttack`** — 발사가 아니라 **공격을 연기**한다. 위기 세션(새떼)에서 화면이 텔레그래프로 뒤덮이는 것도 이 예산이 막는다.
- **순수 FX(파티클·숫자·마커) 초과 = `evictOldest`.** core 바깥이므로 시뮬 무관.
- ★ **모든 캡에 정책이 있다.** 초안 B는 `caps.enemies`·`caps.pickups`를 선언하면서 **그 둘의 초과 정책을 `overflow` 맵에 안 썼다**(제목이 "초과 시 동작(시뮬 결정성 필수)"였음에도). 정본은 **10개 캡 전부에 정책을 명시**한다.

**2층 검증 (A < B, 여유 확인)**
```
enemies:      A 40 (+ 새떼 70)  <  B 96   ✔  (crisisSuspendsWaves → 새떼와 웨이브 동시 최대 = 70 + 잔존)
enemyBullets: A 320             <  B 384  ✔  (여유 20%)
telegraphs:   A 2/개체          <  B 8    ✔  (잡몹 다수가 각자 1~2개 → 8이 안전망)
```
> ★ 초안 D의 "적 40"과 §5.1 서술의 "60기가 화면을 뒤덮고"는 **충돌이 아니었다** — 40은 웨이브 예산이고 60은 **새떼**(별도 예외 `swarmConcurrentMax: 70`)였다. 정본은 두 값을 분리해 표기한다.
> ★ 초안 F의 `caps.enemyBullets: 640` vs 자신의 `fairness.maxSimultaneousEnemyBullets: 320` **자기 충돌**은 정확히 이 2층 구조의 무의식적 발현이었다. 정본은 384/320으로 명시적으로 분리한다.

### 12.2 ★ 플레이어 FX 하드캡 (4칸 상한의 실제 집행 장치)

**플레이어의 모든 발광 FX(레이어 3~4)는 별도 오프스크린 캔버스에 그린 뒤 `globalAlpha = render.playerFxCompositeAlpha`로 화면에 1회 합성한다.**

→ 무기가 몇 개든, 진화체가 얼마나 화려하든, **플레이어 쪽 밝기 총량이 수학적으로 상한**을 넘지 못한다. **스테이지 5~6 맥스 빌드에서 화면이 하얗게 날아가는 실패 모드가 구조적으로 발생 불가.** 전체 화면 `drawImage` 1회 = 바닐라 60fps에서 무시 가능한 비용.

```json
"render": { "playerFxCompositeAlpha": 0.45, "playerBulletMaxAlpha": 0.80,
            "playerBulletMaxRadiusPx": 10, "particleMaxAlpha": 0.60,
            "particleMaxLifeSec": 0.35, "fxMinRealMs": 50, "targetFps": 60 }
```

## 13. 검증 게이트 — 상호 동시 만족 가능한 **하나의 세트**

> ★ 이 절이 비평가가 지목한 **"어떤 숫자로도 동시 통과 불가능한 두 개의 `exit 1` 게이트"** 문제를 닫는다.
> **원인 진단**: 초안 D는 **보스당 격파율**을, F는 **런 클리어율**을 요구했고, 둘 다 "clear rate"라 불렸다. `p⁶` 합성이 성립한다고 가정하는 순간 D의 0.95 → 런 0.735 > F의 0.65 상한, D의 0.55 → 런 0.028 << F의 0.20이 되어 **베이스라인이 자기 게이트를 실패**했다.
> **해결**: **두 지표는 다른 것을 재며 서로 합성되지 않는다.** §10.4의 두 시뮬 모드(`run` / `dpsProbe`)로 분리하고, 아래에서 **동시 만족 창을 산술로 보인다.**

### 13.1 게이트 세트 (`meta.json > certify`)

```json
"certify": {
  "runs": 8000,
  "runMode": {
    "runClearRate":     { "policy":"baseline", "difficulty":"normal", "min":0.35, "max":0.65 },
    "bossTimeoutRate":  { "max":0.25 },
    "noDeadLuck":       { "minRunClearWorstThemeOrder":0.15, "minRunClearWorstDraftPolicy":0.20 },
    "stanceValue":      { "minRunClearDeltaVsStaticStance":0.20 },
    "difficultySpread": { "disasterRunClearRate":{"min":0.02,"max":0.12} },
    "dominance":        { "maxWeaponPickShare":0.16, "maxWeaponWinShare":0.20,
                          "maxElementWinShare":0.35, "maxArchetypeLethalityShare":0.25,
                          "maxThemeClearStddev":0.06 },
    "coinScarcity":     { "medianEndCoins":{"min":0,"max":80},
                          "medianAffordablePerVisit":{"min":1,"max":3} },
    "farmXpRatio":      { "min":2.0 },
    "crisisClearWithoutCapstone": { "min":0.80 }
  },
  "dpsProbe": {
    "runsPerBoss": 1000, "difficulty":"normal", "farm":"maxFarm",
    "balancedPass":   { "min":0.95 },
    "specialistPass": { "min":0.85 },
    "noElementPass":  { "min":0.55 }
  },
  "static": {
    "growthBudget": { "maxLevelUps":60, "minTotalSink":67 },
    "capHits":      { "max":0 },
    "fairnessViolations": { "max":0 }
  }
}
```

### 13.2 ★ 동시 만족 가능성 — 계산

**① `dpsProbe.balancedPass ≥ 0.95` vs `runClearRate ∈ [0.35, 0.65]`**

`dpsProbe`는 **무적 봇**이므로 사망이 판정에 없다. `run`은 `클리어 = Π(d_i × s_i)`, `d_i` = 보스 i의 DPS 통과, `s_i` = 스테이지 i 생존.

```
d_balanced = 0.95 (게이트 하한) → Π d = 0.95⁶ = 0.735
필요:  0.35 ≤ 0.735 × Π s ≤ 0.65
    →  0.476 ≤ Π s ≤ 0.884
    →  스테이지당 생존율 s ∈ [0.885, 0.980]
```
**결론: 동시 만족 가능** ✔ — 스테이지당 생존율 88.5~98%는 완전히 자연스러운 창이다. `d`를 0.99까지 올려도 `Πd = 0.941` → `Πs ≤ 0.69` → `s ≤ 0.94`로 여전히 창이 열려 있다.
(초안 D+F 조합이 불가능했던 이유는 `d`와 `run clear`를 **같은 숫자로 본 것** 하나뿐이었다.)

**② `dpsProbe.noElementPass ≥ 0.55` vs `noDeadLuck.minRunClearWorstDraftPolicy ≥ 0.20`**

`noElementPass`는 **프로브 지표**다 — "속성 무투자여도 3분 안에 죽일 화력이 있는가" = **브릭 없음의 증명**. **런 클리어 지표가 아니다.**
그리고 **속성 0인 런은 구조적으로 발생하지 않는다**: `guaranteeElementCardOnFirstDraft` + `elementCardPity: 6`이 속성 카드를 **반드시 제시**한다. 항상 거절하는 것은 정책이 아니라 거부다 → `forceNoElement` 프로브로만 측정한다.

```
worst draft policy = "random"
  random은 속성 카드가 뜨면 1/3 확률로 집음 → 속성 총합 기대 ≈ 2~3
  d_random ≈ 0.88  →  Π d = 0.88⁶ = 0.464
  Π s (baseline farm/stance) ≈ 0.88
  runClear(random) ≈ 0.464 × 0.88 = 0.41  ≥ 0.20  ✔  (여유 2배)
```
**결론: 동시 만족 가능** ✔ (그리고 `noDeadLuck`은 **draft 축만** 순회한다 — §10.4의 3축 직교 규칙.)

**③ `stanceValue.minRunClearDeltaVsStaticStance ≥ 0.20`**

```
static 봇 = 전환 안 함 = 시작 스탠스(normal) 고정 = 항상 ×1 = noElement와 동등한 화력
  d_static ≈ 0.55  →  Π d = 0.55⁶ = 0.028  →  runClear(static) ≈ 0.025
  runClear(baseline) ∈ [0.35, 0.65]
  Δ = 0.325 ~ 0.625  ≥ 0.20  ✔  (여유롭게 통과)
```
**결론: 동시 만족 가능** ✔ — 그리고 이 게이트가 `noElementPass ≥ 0.55`의 **존재 이유를 확정**한다: 0.55는 "무투자 봇이 런을 깬다"가 아니라 **"무투자여도 벽에 안 부딪힌다(브릭 없음)"**이고, 동시에 `static`과 `baseline`의 격차가 **속성 스탠스가 장식이 아니라 실제 축임을 수치로 증명**한다. 두 게이트가 서로를 방해하지 않고 **각자 다른 것을 증명한다.**

**④ `runClearRate ≤ 0.65` vs `bossTimeoutRate ≤ 0.25`**

```
runClear = 0.35 (하한) → runFail = 0.65
  bossTimeoutRate ≤ 0.25 (전체 런 대비 절대 비율)
  → HP 사망 ≥ 0.40
runClear = 0.65 (상한) → runFail = 0.35
  bossTimeoutRate ≤ 0.25 → HP 사망 ≥ 0.10
```
**결론: 동시 만족 가능** ✔ — 두 끝점 모두에서 양수 창.

**⑤ `farmXpRatio ≥ 2.0` (비평가의 A4 검산 — 게이트를 유지할 수 있는가)**

비평가의 지적: 웨이브 간격 9.0 + 스폰 창 95초 + 상한 14웨이브 → **웨이브 수 비율의 천장은 ~1.3배**.
```
웨이브 수:  passive = floor(95/9) + 1 = 11
            maxFarm = 14 (상한)
            비율 = 14/11 = 1.27   ← 비평가 계산 정확
```
**그러나 `farmXpRatio`는 웨이브 수 비율이 아니라 XP 비율이다.** 비율의 대부분은 **처치율**에서 나온다:
```
웨이브 XP:      passive 11 × 처치율 ~0.45 = 4.95   (하단 고정 → 사거리 밖 이탈이 대부분, enemyExitForfeitsReward)
                maxFarm 14 × 처치율 ~0.98 = 13.7
                → 2.77배
중간보스:       passive 0 (30초 후 이탈)  /  maxFarm 1~2기 × chaffXP×25   → ∞배
새떼 60기:      passive ~0.20  /  maxFarm ~0.90  → 4.5배  (스테이지 총 XP의 ~25%)
dustRunner:     passive 0 (초고속 이탈)  /  maxFarm 대부분 × XP 5배      → ∞배
```
**결론: `farmXpRatio ≥ 2.0`은 달성 가능하며 게이트를 유지한다** ✔
**단, 메커니즘의 정정이 정본에 명시된다**: 비율을 만드는 것은 `waveClearAdvance`(1.27배)가 **아니라** `enemyExitForfeitsReward`(이탈 = XP 영구 소멸, §8.8)다. `waveClearAdvance`는 **밀도 리스크**를 만드는 장치이지 XP 배율 장치가 아니다. **이 정정이 A4 blocker의 답이다** — 게이트를 낮출 필요도, 메커니즘을 바꿀 필요도 없다. **잘못된 것은 서술이었다.**

**⑥ `growthBudget` (정적)**
```
totalSink = 3(새 무기) + 28(무기 레벨) + 6(속성) + 30(패시브) = 67
maxLevelUps = 60  <  67   ✔   (충족률 상한 90%, 목표 54 → 81%)
```
**결론: 동시 만족 가능** ✔ — 그리고 이 부등식은 **"전부 못 찍는다 = 선택이 의미를 가진다"**의 산술적 성립이다. XP 곡선을 바꿔 레벨업이 60을 넘기면 **인증 실패.**

**⑦ `capHits == 0` + `fairnessViolations == 0` (정적 + 런타임)**
```
enemies:      A층 40 (+새떼 70)  <  B층 96   ✔
enemyBullets: A층 320            <  B층 384  ✔
telegraphs:   A층 2/개체          <  B층 8   ✔
텔레그래프:   거동별 표 최소값 0.55  ≥  minTelegraphSec 0.55  ✔ (동일, 등호 허용)
```
**결론: 동시 만족 가능** ✔ — 2층 구조(§12.1)가 이것을 **구조적으로** 보장한다.

**⑧ `difficultySpread.disasterRunClearRate ∈ [0.02, 0.12]` vs `runClearRate(normal) ∈ [0.35, 0.65]`**
난이도 차이의 유일한 원천 = 봇 반응지연 `15틱(×1) → 45틱(×3)`(§10.4). 회피 창이 1/3로 줄어들면 스테이지당 생존율 `s`가 `0.94 → 0.62` 수준으로 떨어진다:
```
Π s(disaster) ≈ 0.62⁶ = 0.056  → runClear(disaster) ≈ 0.735 × 0.056 = 0.041 ∈ [0.02, 0.12]  ✔
```
**결론: 동시 만족 가능** ✔ (그리고 이 게이트가 **봇 지연 모델이 살아있는지**를 검증한다 — 모델이 죽으면 disaster가 normal과 같아져 게이트 실패.)

## 14. localStorage 스키마

> ⚠️ 이 게임은 블로그 전체(`jasonleex1995.github.io`)와 **단일 origin의 localStorage를 공유**한다. **접두사 없는 키는 블로그 상태와 충돌한다.**

```json
{ "v": 1,
  "best": { "normal":   { "score":0, "seed":null, "at":0, "stage":0 },
            "hard":     { "score":0, "seed":null, "at":0, "stage":0 },
            "hell":     { "score":0, "seed":null, "at":0, "stage":0 },
```

|---|---|
