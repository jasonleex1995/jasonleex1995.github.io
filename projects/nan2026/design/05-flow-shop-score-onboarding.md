# 섹션 — 런 플로우 · 상점 · 점수 상세 · 온보딩

> **지위**: 이 문서는 정본(`design/CANON.md`)의 **하위 상세 계층**이다. 정본이 확정한 상태 기계(§6.5)·페이즈 길이(§6.3)·드래프트 발생(§6.4)·키맵(§5)·상점 모델(§11.2)·점수 키(§11.3)·컨티뉴(§11.4)·온보딩 3키(§9.9)를 **참조만** 하고 **재정의하지 않는다.**
> 정본이 닫은 것 = **모델·키·값.** 이 문서가 닫는 것 = **화면의 내용, 전이의 트리거, 입력의 경계, 정산의 순서, 온보딩의 인과 사슬.**
> 정본과 충돌하는 서술이 이 문서에 있다면 **이 문서가 틀린 것이다** (C-1).
> 값은 전부 `data/meta.json > flow / shop / score / onboarding` 에 산다 (C-2). 새로 정한 값은 §10에 전수 표기했다.

## 0. 이 문서가 지키는 3개의 파생 규칙

정본에서 직접 파생되며, 아래 모든 화면 명세가 이 셋을 전제한다.

| # | 규칙 | 파생원 |
|---|---|---|
| **F-1** | **클럭이 멈춘 화면에서는 I-4(적 탄은 절대 가려지지 않는다)가 적용되지 않는다.** DRAFT·SHOP·PAUSE·DEATH·STAGE_CLEAR·HEAL·THEME_BANNER는 게임 클럭이 정지하므로 회피가 존재하지 않고, 따라서 플레이필드를 덮어도 된다. **I-4는 "회피의 전제"를 지키는 규칙이지 렌더 순서의 미신이 아니다.** | §7.1 I-4 · §6.4 · §6.6 |
| **F-2** | **새 색을 만들지 않는다.** 이 섹션의 모든 화면은 §7.2 팔레트의 기존 키만 쓴다. 위협은 자홍(`palette.threat.enemyBullet`), 중립·정상은 은색(`palette.element.normal`), 속성은 4속성 키, 호박(`palette.status.band`)은 **플레이필드 밖 + 온-플레이어 아이콘에만**(§7.2 예약 규칙). → 타이머 경고·위기 예고·파 고스트는 전부 **기존 채널의 재사용**이다(§4.1, §2.6, §6.5). | §7.2 |
| **F-3** | **화면이 바뀌는 순간 눌려 있던 키는 먹지 않는다.** `flow.edgeTriggerOnStateEnter = true` — 상태 전이 시점에 이미 눌려 있던 `Space`/`Enter`/`Escape`는 **뗄 때까지 무시**한다(상승 엣지 전용). 정본 §5.2가 "인게임 Space = 폭탄 → 드래프트가 뜨는 순간 연타 = 카드 오선택"을 **`Enter` 확정**으로 막았다. 그러나 같은 사고가 **DEATH**(Space = 즉시 재시작 → 컨티뉴 기회 상실)와 **RESULTS**에도 그대로 존재한다. F-3은 그 계열 사고 **전부를 규칙 하나로** 닫는다. | §5.2 · §5.5 · §5.7 |

## 1. 상태 기계 — 화면 단위 전개

정본 §6.5의 다이어그램을 **화면·트리거·입력·클럭** 4열로 전개한 것이다. 상태·전이는 정본과 1:1이며 추가·삭제가 없다.

| 상태 | 클럭 | 나가는 트리거 | 다음 |
|---|---|---|---|
| `TITLE` | 메뉴(§4.4) | `Space` / `O` / 무입력 `flow.attractIdleSec`(20) | DIFFICULTY / OPTIONS / ATTRACT |
| `ATTRACT` | 게임(§2.2) | 아무 키 / 잡몹 페이즈 종료 | TITLE |
| `DIFFICULTY` | 메뉴 | `Enter` / `Escape` | RUN_START / TITLE |
| `RUN_START` | — | **0프레임 논리 상태** (시드 발급 → 테마 5종 일괄 추첨 → 로드아웃 초기화) | THEME_BANNER(1) |
| `THEME_BANNER` | **정지** | `flow.themeBannerSec`(2.5) 경과 또는 `Space` | MOB_PHASE(i) |
| `MOB_PHASE` | **진행** | `stage.mobPhaseSec`(120) 경과 | DRAFT_QUEUE |
| ┗ `DRAFT` | **완전 정지** | 카드 확정 | MOB_PHASE 복귀 |
| ┗ `CRISIS` | **진행** | — (내부 서브구간 95~120, 독립 상태 아님) | — |
| `DRAFT_QUEUE` | **정지** | 큐 소진 (0장이면 0프레임 통과) | BOSS_INTRO / HEAL |
| `BOSS_INTRO` | **진행**(보스 무적·발사 없음·**타이머 정지**) | `stage.bossEntrySec`(3.0) 경과 | BOSS |
| `BOSS` | **진행** | 코어 격파 / HP 0 / 타이머 만료 | STAGE_CLEAR / DEATH |
| `STAGE_CLEAR` | 정지 | `flow.stageClearSec`(2.0) 또는 `Space` | DRAFT_QUEUE |
| `HEAL` | 정지 | `flow.healSec`(1.5) 또는 `Space` | THEME_BANNER(i+1) |
| `SHOP` | 정지 | `Escape` + 확인 1회 | MOB_PHASE(i+1) |
| `PAUSE` | **전 클럭 정지** | `Escape` → `flow.pauseResumeCountdownSec`(1.5) | 직전 상태 |
| `OPTIONS` | 정지 | `Escape` | 직전 상태(TITLE 또는 PAUSE) |
| `DEATH` | 정지 | 컨티뉴 / 포기 / `Space` | 직전 상태 재개 / RESULTS / RUN_START |
| `RESULTS` | 메뉴 | `Space` / `Escape` | RUN_START / TITLE |

**★ 스테이지 루프에 `DRAFT_QUEUE`가 두 번 있다 (정본 §6.4·§6.5의 정확한 독해)**
`MOB_PHASE → DRAFT_QUEUE → BOSS_INTRO` (위기 세션 25초 동안 쌓인 큐를 **보스 인트로 전에** 소화) 와
`STAGE_CLEAR → DRAFT_QUEUE → HEAL` (보스 페이즈 큐 — **구조적으로 항상 0장**, §6.4).
전자가 실질이고 후자는 **안전망**이다. 후자가 0장이므로 §6.5의 `STAGE_CLEAR → DRAFT_QUEUE`는 정상 런에서 0프레임을 통과한다. → **"보스 깨고 나서 카드를 고른다"는 체감은 존재하지 않고, 카드는 항상 보스 *전에* 소화된다** = 보스 = 가져온 빌드의 시험지(§8.8).

**포기(abandon)는 PAUSE에 없다.** 정본 §5.5의 PAUSE 열은 `Escape`(재개) / `O`(OPTIONS) **둘뿐**이며 배타적이다. 런 이탈 = 새로고침 = 런 소멸(§6.5). 이는 누락이 아니라 오락실 규칙이다 — **런을 끝내는 방법은 클리어 아니면 죽음뿐**이고, 그래서 `bossTimerSec`가 진짜 허들이 된다. (명시 키 요청: §11-④)

## 2. 화면 명세 — 런 진입까지

### 2.1 TITLE

| 요소 | 내용 |
|---|---|
| 헤더 | **`TETRA`** (§m5) |
| 기록 | 난이도별 최고점수 4줄 — `Q W E R` 아닌 **난이도 순서 Normal → Hard → Hell → Disaster** 고정. 각 줄에 `점수 · 도달 스테이지 · 시드(8자리 hex)` (localStorage `best.<diff>`, §14). 미기록은 `— — —` |
| 조작 | **키캡 그림**(텍스트 아님): 방향키 / `Q W E R` + 4속성 글리프(●▲◆✚) / `Space` 폭탄 / `Shift` 토큰 |
| 상성 | **4×4 상성표**(§6.6이 PAUSE에 비치한 것과 **같은 컴포넌트 재사용**, 순서 `Q W E R`) |
| 키 | `Space` = DIFFICULTY · `O` = OPTIONS · 무입력 20초 = ATTRACT |
| 배경 | 어트랙트 대기 중에도 배경은 절차적 스크롤(§9.10) — 정지 화면 아님 |

> 상성표를 TITLE·PAUSE 두 곳에 **같은 컴포넌트로** 비치하는 것이 "튜토리얼 텍스트 0"의 인프라다. 온보딩이 못 가르치면 플레이어는 언제든 `Esc`로 표를 본다. 비용 = 컴포넌트 1개.

### 2.2 ATTRACT — 심사자가 보는 첫 3초

**정의: 인증 봇이 실제로 스테이지 1을 플레이하는 것.** 리플레이도, 스크립트도, 영상도 아니다.

| 항목 | 확정 |
|---|---|
| 구현 | `flow.attract.difficulty = "normal"`, 스테이지 1, 봇 정책 = `bot.baseline`(generalist / balanced / greedyNearest). 시드는 매 회 새로 발급 |
| 루프 | 게임과 **문자 그대로 동일한 `step()`** 에 `bot(world)`의 입력을 주입 (§10.4의 시뮬 루프와 동일) |
| 드래프트 | 카드 3장을 **표시하고** `flow.attract.draftDwellSec`(0.8) 후 봇이 선택 → 카드가 뽑히는 것이 보인다 |
| 종료 | **잡몹 페이즈 종료 시점**(`flow.attract.endAfterMobPhase = true`) → TITLE. 보스전은 보여주지 않는다 |
| 중단 | **아무 키** → 즉시 TITLE (F-3에 의해 그 키는 TITLE에서 다시 먹지 않는다) |
| 기록 | 어트랙트 런은 **점수를 집계하지 않고 localStorage를 건드리지 않는다** |
| HUD | **전체 HUD 그대로** — 봇의 스탠스 전환이 좌 패널에서 보여야 한다 |

**★ 왜 잡몹 페이즈에서 끊는가**: 잡몹 페이즈 120초의 마지막 25초 = **위기 세션**이고, 그 60기(스테이지 1은 `swarmTotalScale[0]=0.5` → 30기)는 **전원 테마 속성**(`crisisElementRule: "themePure"`, §8.10)이다. 봇이 정답 스탠스를 켜두면 화면이 ×2 화이트-핫 + 저역 크런치로 뒤덮인다. **어트랙트가 게임의 셀링 포인트에서 정확히 끝난다.** 보스전(최대 3분)을 붙이면 어트랙트 1회가 5분이 되고, 클라이맥스가 중간에 파묻힌다.

**★ 왜 봇인가 (비용 논증)**: 봇은 인증 파이프라인(§10.4)에 **이미 존재하는 코드**다. 어트랙트를 위해 만드는 것이 아니라, 만들어진 것을 화면에 띄우는 것이다. 그리고 이것은 공모전 스토리의 실물이다 — **"타이틀 화면에서 지금 플레이하고 있는 그 봇이 이 게임의 밸런스를 인증했습니다."** (구조 요청: §11-⑤)

### 2.3 DIFFICULTY

4종 **전부 처음부터 선택 가능**(§6.1, 해금 그라인드 = 기둥 위반). 커서 기본 = **Normal 고정**(마지막 선택을 기억하지 않는다 — `opts`에 그 키가 없고, 만들면 §14 스키마 변경이다).

각 줄에 §6.1 표의 값을 **그대로** 표시한다:

```
NORMAL    ×1.0   보스 타이머 3:00   런 ~30분   점수 ×1.0    최고 ——
HARD      ×1.5   보스 타이머 2:00   런 ~20분   점수 ×1.6    최고 124,300
HELL      ×2.0   보스 타이머 1:30   런 ~15분   점수 ×2.5    최고 ——
DISASTER  ×3.0   보스 타이머 1:00   런 ~10분   점수 ×4.0    최고 ——
```

> **"Hell = 적이 세다"가 아니라 "Hell = 15분 스피드런"임이 선택 시점에 읽혀야 한다.** 난이도의 유일한 축이 배속이므로(§6.1), 화면이 보여줄 것도 **시간**이다. 적 스탯을 언급하는 문구는 거짓말이 된다(적 스탯은 난이도로 변하지 않는다).
> **난이도는 런 시작 후 변경 불가**(§6.1). 이 화면이 그 결정의 유일한 지점이다.

### 2.4 RUN_START — 0프레임 논리 상태

| 순서 | 동작 |
|---|---|
| 1 | 마스터 시드 발급 — `main.js`가 생성해 core에 주입 (§10.2). **비결정성이 들어오는 유일한 지점** |
| 2 | `rng.theme`로 **5개 테마를 한 번에 전부 추첨**(§8.1). 스테이지 6 = `finale` 고정 |
| 3 | 로드아웃 초기화 = **정본 §2.6 표 그대로** (HP 100 / 스탠스 노말 / `forward` Lv1 슬롯 1 / 투자 0-0-0 / 폭탄 1 / 코인 0 / Lv1) |
| 4 | → THEME_BANNER(1) |

**시작 무기 선택 화면 없음**(§2.6). t=0의 선택은 정보가 없어 가짜다.

## 3. 화면 명세 — 스테이지 루프

### 3.1 THEME_BANNER — 정보 기둥의 유일한 사전 전달 지점

전체화면, `flow.themeBannerSec`(2.5), `Space` 스킵.

| 요소 | 내용 |
|---|---|
| 테마 | 이름(`themes[].name`) + 스테이지 번호 `2 / 6` |
| **구성비** | ★ **`stages[].mix`의 4개 수치를 그대로** — `물 70% · 불 10% · 풀 10% · 노말 10%`. **순서는 `Q W E R` = 노말 · 불 · 물 · 풀**(§4.1). 각 항목에 속성색 + 글리프 + (cvd/mono에서) 텍스트 라벨 |
| **상성 미니휠** | 4속성 글리프의 순환 다이어그램. **이 테마의 70%에 ×2인 글리프가 밝게** — 나머지는 알파 0.35. 텍스트 0 |
| 보유 | 현재 속성 투자 3종 + 합계 예산 `투자 4 / 6` (§m1) — **다음 상점에서 무엇을 살지, 다음 드래프트에서 무엇을 고를지를 지금 결정하게 한다** |
| finale | `element = null` → 이름 `최종`, 구성비 `물 30 · 불 30 · 풀 30 · 노말 10`, 배너 문구 **`테마 없음 — 4속성 혼재`**(§8.16). 미니휠은 **4글리프 전부 밝게** |

> **수치를 숨기지 않는다.** "대략 물이 많습니다"는 정보 기둥에 대한 배신이고, `mixGranularity: "perWave"` + `check.mjs` S8의 **±3%p 게이트**가 이 수치를 **참으로 만든다**(§8.2). 게이트가 없으면 배너는 광고이고, 게이트가 있으면 배너는 계약이다.

### 3.2 MOB_PHASE — HUD 계약

이 상태에는 전용 화면이 없다. 플로우가 성립하기 위해 **상시 표시되어야 하는 것**의 목록은 §9의 계약표에 있다. 이 상태에서 이 섹션이 소유하는 것은 **위기 세션 예고**와 **드래프트 큐 배지** 둘뿐이다.

**위기 세션 예고 (`stage.crisisWarnSec` = 3.0)**

| 채널 | 표현 |
|---|---|
| ① 상단 띠 | 페이즈 잔여 표시가 **`새떼 3 → 0` 카운트**로 전환 |
| ② 아레나 | ★ **스폰 라인(`view.spawnLineY` = −40) 바로 아래에 자홍 점선 수평선 1개**, 3.0초 지속 |
| ③ 오디오 | 상승 스윕 + `bgm ×0.5` 덕킹(§7.10) |

> ★ **②가 이 설계의 핵심**: 자홍 점선 수평선은 §7.4의 **`wall` 이미터 텔레그래프와 문자 그대로 같은 도형**이다. 새 어휘 0, 새 색 0. 플레이어는 이미 "자홍 점선 = 곧 저기가 아프다"를 배웠으므로 **위기 예고를 배울 필요가 없다.** 그리고 3.0 ≥ 모든 텔레그래프 하한(§12.4)이므로 예고 없는 스파이크가 구조적으로 불가능하다.
> 화면 가장자리 호박 펄스는 **쓰지 않는다** — 호박은 §7.2에 의해 **내 상태이상·저체력 전용 채널**이고, 새떼 진입은 내 상태가 아니라 외부 위협이다(F-2).

**드래프트 큐 배지**: 위기 세션 중 레벨업 = 큐 적립(§6.4) → HUD에 **`LV UP ×N`**. 이 배지는 위기 세션 25초 동안만 존재하고, 페이즈 종료 시 DRAFT_QUEUE가 소진한다.

### 3.3 DRAFT — 화면

**전체화면 상태.** 정지된 플레이필드가 알파 0.25로 비친다(정지되어 있으므로 **맥락**이지 정보가 아니다 — F-1).

```
┌──────────────────────────────────────────────────────────────┐
│  Lv.23 → 24     테마 뱃지 [화산 불70/물10/풀10/노말10]         │
│  투자  불 +2   물 +1   풀 0        합계 3 / 6      리롤 ×2    │
├────────────────┬────────────────┬────────────────────────────┤
│    [ 1 ]       │    [ 2 ]       │    [ 3 ]                   │
│    카드        │    카드        │    카드                    │
│                │                │                            │
├────────────────┴────────────────┴────────────────────────────┤
│  슬롯   ① forward Lv5 [불]   ② lance Lv3 [불]   ③ aura Lv1 [—]   ④ —   │
│  [1][2][3] 즉시 선택   [←→] 커서   [Enter] 확정   [F] 리롤   [Tab] 재정렬 │
└──────────────────────────────────────────────────────────────┘
```

| 영역 | 내용 |
|---|---|
| 상단 | 레벨 전이 · **테마 뱃지**(구성비 축약) · **속성 투자 3종 + 합계 예산**(§m1) · 리롤 스톡 |
| 카드 3장 | §11.1의 **필수 표기 표를 그대로** 렌더 (카테고리별) |
| **하단 슬롯 스트립** | ★ **무기 4칸 상시 표시 + 현재 스탠스 기준 부여 상태.** 기체의 임뷰 칩(§7.5 ②)과 **같은 표기 언어**(속성색 채움 = 부여 / 은색 = 무속성) |
| 하단 키 힌트 | 키캡 그림 |

**★ 하단 슬롯 스트립이 없으면 `elementLevel` 카드는 무의미한 숫자다.** "불 +2 → +3"이 무엇을 바꾸는지는 **슬롯 순서를 볼 때만** 읽힌다. 그래서 이 스트립은 장식이 아니라 **§11.1의 `elementLevel` 필수 표기(`전환 시 부여 무기: 2개 → 3개` + before/after 슬롯 그림)의 그릇**이다.

**입력** (§5.2 그대로, 재정의 없음): `←→` 커서 / `Digit1~3` 즉시 선택 / `Enter` 확정 / `F` 리롤(최대 2회/드래프트) / `Tab` 재정렬 / `QWER`·`Space`·`Shift` **죽은 키**.

**클럭**: 완전 정지(§6.4). 적·탄·스폰·페이즈 타이머 전부. **무제한 사고 시간.**

**카드 선택 직후 (같은 화면에서 일어나는 일)**
- `elementLevel` 확정 → **하단 스트립이 그 자리에서 갱신**(§4.3 "재계산 시점 = 즉시"). 부여가 1칸 늘어나는 것을 **결정한 화면에서 본다.**
- `newWeapon` 확정 → `draft.slotAssign = "append"` = 가장 앞의 빈 슬롯에 자동 배치 → **그 화면에서 바로 `Tab` 재정렬 가능**(§5.3). 획득 순서가 몰래 부여 우선순위가 되는 사고를 여기서 막는다.
- `resupply` 확정 → 코인 +40, 코인 표시와 **환산 점수 표시가 동시에** 오른다.

### 3.4 슬롯 재정렬 모드

| 항목 | 확정 |
|---|---|
| 진입 | 드래프트 화면에서 `Tab` (§5.3) |
| 조작 | `←→` 슬롯 커서 / `Space` 집기·놓기(스왑) / `Tab`·`Escape` 종료 |
| 표시 | 하단 스트립이 확대되어 화면 중앙으로 승격. 집은 슬롯은 들린 상태로 표시. **집는 순간 부여 프리뷰가 실시간 갱신** — 놓기 전에 결과가 보인다 |
| 비용 | 없음. 횟수 제한 없음 |
| **가능 시점** | ★ **드래프트 화면에서만.** 상점·일시정지·인게임 전부 불가(§5.3) |

**정규 창구 = `MOB_PHASE → DRAFT_QUEUE`** (위기 세션 큐 소화 = 보스 직전). 보스가 요구할 스탠스 퍼즐을 **알기 전**이 아니라 **테마를 알고 중간보스를 겪은 뒤**에 순서를 정한다.

**보스전 재정렬은 코드 한 줄 없이 봉쇄된다**: 재정렬 = 드래프트 전용 + 보스전에 드래프트가 구조적으로 발생 불가(§6.4) → **곱하면 0.** "부위마다 정지 → 재정렬"이라는 체스식 최적화가 **금지 규칙 없이** 사라진다.

### 3.5 BOSS_INTRO — 3초는 연출이 아니라 정보 창이다

`stage.bossEntrySec`(3.0). 보스 무적 + 발사 없음 + **타이머 정지**(§6.3, `timerStartsAfterIntro`).

| 초 | 무슨 일이 |
|---|---|
| 0.0 | 보스가 스폰 라인에서 하강. 상단 띠 타이머는 **`3:00`에 정지한 채 표시** |
| 0.5~2.5 | ★ **부위가 하나씩 점등** — `parts[]` 순서대로 속성색 외곽선 3px + 부위 글리프 + 소형 HP 바가 켜진다. 코어는 마지막, 은색 + ● |
| 2.5~3.0 | 전신 자홍 림 라이트 1회(§7.4 보스 대형 패턴 언어의 예고편) |
| 3.0 | **타이머 시작 + 보스 피격 가능** (같은 프레임) |

> **★ 부위 순차 점등이 이 3초의 정체다.** 강철 가오리라면 `물 → 불 → 풀 → 노말(코어)` 이 순서대로 켜지고, 플레이어는 **3초 동안 정답 스탠스 3개를 읽는다**(풀 → 물 → 불). 이것이 "정보 기반 선택 > 운빨"의 보스전 판이며, 정본이 이미 가진 채널(§7.6 부위 판독)의 **재생 순서만 정한 것**이다. 새 어휘 0.
> 이 3초는 **타이머 180에서 차감되지 않는다**(§6.3). 정보는 공짜로 준다.

### 3.6 STAGE_CLEAR → HEAL

| # | 단계 | 내용 |
|---|---|---|
| 1 | **STAGE_CLEAR** (`flow.stageClearSec` 2.0, `Space` 스킵) | 스테이지 소계 4줄: `처치 · 상성 보너스 · 보스 격파 · 남은 시간 (잔여 M:SS × 50)` + **파 막대 1개**(§6.5) + **`무피격 ✔ / ✕`** |
| 2 | **DRAFT_QUEUE** | 큐 소화 (§1 — 정상 런에서 0장, 0프레임 통과) |
| 3 | **HEAL** (`flow.healSec` 1.5, 스킵 가능) | **회복량 = `ceil(현재 최대HP × flow.stageClearHealPct(0.25))`.** "현재 최대HP" = `player.hpMax` + 상점 `maxhp` 스택 + 패시브 `bulkhead` → 100~170. **비율이므로 최대HP 투자가 회복량에도 되돌아온다** |
| 4 | THEME_BANNER(i+1) → 5 | SHOP → 6 | MOB_PHASE(i+1) |

**`무피격 ✔`는 약속이지 지급이 아니다.** STAGE_CLEAR는 무피격 성립을 **표시**하고 HUD 무피격 pip(6칸) 중 한 칸을 채우지만, **점수는 붙지 않는다.** 근거는 §6.2 — 컨티뉴가 무피격을 **런 전체 소급 무효**(§11.4)로 만들 수 있으므로, 이미 지급한 점수를 회수하는 UI가 존재해서는 안 된다.

**스테이지 6은 STAGE_CLEAR → RESULTS**(§6.5). HEAL·THEME_BANNER·SHOP 없음.

## 4. 보스 타이머 UX

### 4.1 3단 상태 — 새 색 0개 (F-2)

타이머는 상단 띠 중앙(§1.2), 게임시간 `3:00 → 0:00` 표시. 옆에 배속 배지 `×3.0`.

| 상태 | 조건 | 표현 | 오디오 |
|---|---|---|---|
| **정상** | 잔여 > `stage.timerWarnSec`(60) | **은색**(`palette.element.normal`) 숫자 | — |
| **경고** | 잔여 ≤ 60 | ★ **색 불변.** `visual.timer.warnScale`(1.15) 확대 + `warnPulseHz`(0.5) 맥동 | — |
| **빨간불** | 잔여 ≤ `stage.timerRedAlertSec`(30) | ★ **자홍**(`palette.threat.enemyBullet`) + `alertScale`(1.30) + `alertPulseHz`(1.0) 펄스 | **1Hz 틱**(§7.10) |

> **★ 왜 경고(60초)에 색을 안 쓰는가**: 정본 §6.3의 라벨 `(노랑)`은 팔레트 키가 아니다 — §7.2의 `palette`에 노랑은 **존재하지 않고**, 노랑 `#FFE24A`는 §7.2가 **호박 상태이상 채널과 사실상 같다는 이유로 명시적으로 배제**한 색이다. 여기서 노랑을 되살리면 "내 상태가 나쁘다"(호박)와 "시간이 없다"가 섞인다. **60초 경고는 명도·크기·주기 채널로 충분하다**(§7.2 대원칙: hue는 속성 전용 예약 채널). (라벨 정정 요청: §11-①)
> **★ 왜 빨간불에 자홍인가**: 자홍 = **"이것이 나를 죽인다"의 단일 채널**(적 탄·텔레그래프). 타이머 만료 = **즉사**(`boss.timerExpire: "kill"`, §6.3). 의미가 정확히 일치한다. 새 hue 0, 새 학습 0. 그리고 자홍은 **`ShiftLeft`가 유효한 창의 시각 정의**이기도 하다(§4.3).
> 화면 가장자리 비네트는 **쓰지 않는다** — 그 자리는 호박(내 상태) 전용이다(F-2).

### 4.2 타이머가 멈추는 곳 = 정확히 둘

`PAUSE` + `boss.phaseTransitionSec`(1.5, 최대 2회) — §6.3. **드래프트는 보스전에 발생하지 않으므로 세 번째 후보가 없다**(§6.4). 실효 최대 = **183 게임초**이고 표시는 180→0이며 전환 중 **멈춘 채로 보인다**(멈춘 것이 보이는 것이 곧 "이 3초는 공짜"라는 고지다).

### 4.3 시간 연장 토큰 (`ShiftLeft`)

정본 §11.4 그대로. 이 섹션이 더하는 것은 **UX 경계** 하나뿐:

| 상황 | 화면 |
|---|---|
| 토큰 보유 + **빨간불 아님** | HUD 토큰 아이콘 **은색·비활성**. `Shift` 입력 = **완전 무시, 피드백 없음** |
| 토큰 보유 + **빨간불** | 토큰 아이콘이 **자홍으로 점등** + 1Hz 동기 펄스(타이머와 같은 위상) → **"지금 이 키가 먹는다"** |
| 발동 | 타이머 숫자가 `+0:30` 오버슈트 애니메이션 후 자홍 → 은색 복귀(잔여 > 30이 되므로 정상 상태로 강등). ★ **동시에 상단 띠에 `시간 보너스 ✕` 회색 마크가 점등하고 그 보스전 내내 남는다** |
| 토큰 0 | 아이콘 미표시 |

> ★ **`시간 보너스 ✕` 마크가 `timeTokenForfeitsTimeBonus`(LOCKED, §11.3)를 화면에 존재하게 한다.** 대가가 결과 화면에서 처음 드러나면 그건 함정이지 결정이 아니다. 마크가 보스전 내내 남아 있어야 **"이 판을 포기할래 점수를 포기할래"**가 진짜 결정이 된다.
> **오발 불가**(§11.4): 빨간불이 아니면 입력 자체가 무시되므로, 은색·비활성 아이콘은 **금지 표시가 아니라 상태 표시**다.

**F-3 + `deathAnimSec` 1.5초**가 `Space` 오발을 이중으로 막는다. **카운트다운 없음 — 무기한 대기**(§11.4).

### 5.3 컨티뉴 — 부활

정본 §11.4의 표를 **재정의하지 않는다.** 이 섹션이 더하는 것은 **소급 무효의 화면 표현** 하나:

| 순간 | 화면 |
|---|---|
| 컨티뉴 확정 | 코인 −150 (코인 표시와 **환산 점수 표시가 동시에** 3,000 × 배율만큼 떨어진다) |
| **동시** | ★ **HUD 무피격 pip 6칸이 전부 회색으로 소등**되고 `퍼펙트 ✕` 마크가 켜져 **런 종료까지 남는다** |
| 부활 | 아레나 하단 중앙 · HP 만피 · 적 탄 전량 소거 · `flow.continueIframeSec`(3.0) 점멸 무적 |
| 보스전이면 | 보스 HP 유지 · 타이머 `max(현재 잔여, 60)` → **재개 즉시 빨간불**(잔여 ≤ 30이었다면 60으로 올라 경고 상태) |

> ★ **pip 6칸 소등이 "이중 비용"을 즉시 보여준다.** 정본은 컨티뉴의 대가를 `perfectBonus` + **모든 스테이지의 `stageNoHitBonus` 소급 무효** + 코인 150으로 정했다(§11.4). 그 대가가 20분 뒤 결과 화면에서 처음 보이면 **함정**이다. 지금 6칸이 꺼지는 것을 보고 코인을 내는 것이 **결정**이다.
> 그리고 §6.2의 "무피격은 정산 항목이지 라이브 항목이 아니다"가 여기서 값을 한다 — **회수해야 할 점수가 애초에 지급된 적이 없으므로**, 라이브 점수가 뒤로 가는 일이 발생하지 않는다.

## 6. 점수 상세

### 6.1 항목 — 정본 §11.3이 유일한 소유자

이 섹션은 **키도 값도 판정 규칙도 재정의하지 않는다.** 아래는 정본이 정한 것을 **언제 어디에 어떻게 보여주는가**만 확정한다.

### 6.2 ★ 라이브 점수 vs 정산 점수 (이 섹션의 결정)

| 구분 | 항목 | 확정 시점 |
|---|---|---|
| **라이브** (HUD에 실시간) | 처치 점수 · **상성 처치 보너스** · `midBossClearBonus` · `bossClearBonus` · **남은 시간 보너스** | 발생 즉시 |
| **정산** (RESULTS에서 일괄) | `stageNoHitBonus` · `perfectBonus` · `runClearBonus` · **남은 코인 환산** | 런 종료 |

**★ 왜 나누는가 — 3개의 문제가 한 번에 닫힌다**
1. **소급 무효가 자연스러워진다.** 컨티뉴는 `stageNoHitBonus`를 **런 전체 소급 무효**로 만든다(§11.4). 라이브에 넣으면 점수가 −18,000 뒤로 가는 화면을 만들어야 한다. 정산 항목이면 **회수할 것이 없다.**
2. **남은 코인은 원래 라이브가 될 수 없다.** 잔액 × 20은 런이 끝나야 확정된다 — 상점의 `= −N 점`은 **환산 예고**이지 점수 이벤트가 아니다.
3. **오락실 정산 의식이 성립한다.** RESULTS에서 순차 타이핑으로 붙는 항목이 있어야 결과 화면이 **의식**이 된다(§11.3).

**배율 처리 (산술 동치 확인)**
```
라이브 표시 = floor( (확정 항목 합) × difficultyMul )
최종 점수   = floor( (확정 항목 합 + 정산 항목 합) × difficultyMul )
```
곱셈이 합에 분배되므로 **"전부의 합계에 최종 곱"(§11.3)과 완전히 동일**하다. 배율을 라이브에서 이미 곱하는 이유: Disaster를 고른 이유가 **실시간으로 보여야** 한다. `score.roundMode = "floor"`, **`floor`는 최종 1회만.** (키 요청: §11-③)

### 6.3 상성 처치 보너스 — 화면에서 보이는 곳

정본 §11.3: 개체마다 `총 피해`·`×2 피해` **2개 누산기**, 처치 시 지분 ≥ 0.5면 성립 → 처치 점수 ×1.5.

**막타가 아니라 지분이므로, 이 판정은 UI로 노출할 수 없고 노출해서도 안 된다.** 개체마다 지분 게이지를 띄우면 그건 §11.3이 배제한 **체스식 최적화**를 도로 초대하는 것이다. 이 섹션의 확정:

| 지점 | 표현 |
|---|---|
| 인게임 | **없음.** ×2 히트의 감각(§7.7: 화이트-핫 + 임팩트 프리즈 + 파티클 3 + 저역 크런치)이 곧 지분을 쌓고 있다는 피드백이다 |
| STAGE_CLEAR | 스테이지 소계에 `상성 보너스 +12,400` 1줄 |
| RESULTS | 런 합계 1줄 |

> **최적 플레이 = "제대로 된 스탠스로 싸우기"**(§11.3). 화면이 지분을 보여주면 최적 플레이가 "지분을 관리하기"로 변질된다. **보여주지 않는 것이 설계다.**

### 6.5 ★ 파(par) — 스칼라에서 라이브 고스트를 만드는 사상

정본은 `flow.stagePar[6]`을 **스테이지별 표준 보스 잔여시간**(시뮬 중앙값, `report/stages.csv` → `--certify` 통과 시 기입, §10.4)으로 정의하고, 두 개의 장치가 그것을 소비하도록 요구한다: **라이브 고스트**(보스 HP 바, §11.4)와 **사후 막대**(DEATH·STAGE_CLEAR·RESULTS). 스칼라 하나로 라이브 고스트를 그리려면 **사상이 필요하고 정본에 없다.** 이 섹션이 확정한다.

```
표준 격파 시각        killPar[i] = stage.bossTimerSec(180) − flow.stagePar[i]      // 게임초
파 페이스 기대 코어HP  ghostHpPct(t) = clamp( 1 − t / killPar[i] , 0 , 1 )          // t = 보스 경과 게임초
고스트 마커 위치      = 코어 HP 바 위의 ghostHpPct 지점
```

| 규칙 | 확정 |
|---|---|
| 표현 | **은색 수직 마커 2px**(`palette.element.normal`) + 실제 코어 HP 바 위. 텍스트 0 |
| 읽는 법 | 실제 HP가 고스트보다 **오른쪽 = 언더파**(뒤처짐). **왼쪽 = 오버파** |
| 정지 | 타이머가 멈추면(`PAUSE`·페이즈 전환) `t`도 멈추므로 **고스트도 멈춘다** — 타이머와 일관 |
| ★ **미측정 센티넬** | **`flow.stagePar[i] == 0` = 미측정** → **고스트·막대 모두 숨긴다.** `meta.json`의 기본값이 `[0,0,0,0,0,0]`이므로, 인증 전에는 UI가 **없는 것**이지 **틀린 것**이 아니다 |
| 선형 가정 | `ghostHpPct`가 선형인 것은 **근사이며 의도**다. 정확한 DPS 곡선을 그리려면 시뮬이 시계열을 뱉어야 하고(스칼라 계약 파기), 플레이어가 읽는 것은 **"앞서는가 뒤처지는가"** 한 비트뿐이다 |

> **"20분 뒤에야 알게 되는 언더파를 지금 알려준다"**(§11.4). 4스테이지에서 처음 파선을 밑돌았다면, 그 원인은 **1~3스테이지의 파밍**이다. 파 막대 6개가 그 인과를 그림 한 장으로 보여준다 — **텍스트 0, 판정 0, 잔소리 0.**

**★ `= −N 점`의 계산 (이 섹션의 확정)**
```

```
난이도 배율을 **포함한다.** Disaster에서 30코인짜리 리롤은 −600점이 아니라 **−2,400점**이다. 배율을 빼고 표시하면 **고난이도에서 기회비용이 4배로 커진다는 사실이 화면에서 사라진다** — 그리고 그것이 정확히 이 게임의 중심 긴장(§11.2 "아껴서 점수 vs 써서 생존")이다. 보유 코인 옆의 환산 점수도 같은 식.

### 8.4 ★ 증명 — 어느 속성 카드를 골라도 ×2와 ×0.5가 같은 화면에 나온다

정본 §8.2는 "한 스탠스를 켜고 있으면 ×2(70%)와 ×0.5(10%)가 자연히 같은 화면에 나온다"고 선언한다. 그 선언은 **정답 스탠스를 켰을 때**의 서술이다. 그런데 첫 드래프트의 속성 카드는 **무엇이 뜰지 모르고, 무엇을 고를지는 플레이어의 자유**다. → **3개 경우 전수 확인이 필요하다.**

테마 속성 `T`, `counter`(T를 이김), `prey`(T가 이김). 혼합 = `T .70 / counter .10 / prey .10 / normal .10`(§8.2). 플레이어가 켠 스탠스 `S`:

| `S` | vs `T` (70%) | vs `counter` (10%) | vs `prey` (10%) | vs `normal` (10%) | ×2 있음 | ×0.5 있음 |
|---|---|---|---|---|---|---|
| **`counter`** | **×2** | ×1 | **×0.5** | ×1 | ✔ | ✔ |
| **`T`** | ×1 | **×0.5** | **×2** | ×1 | ✔ | ✔ |
| **`prey`** | **×0.5** | **×2** | ×1 | ×1 | ✔ | ✔ |

**3/3 경우 전부에서 ×2와 ×0.5가 같은 혼합 안에 존재한다** ✔
(검산 — 바다(T=물): `S=풀` → 풀→물 ×2 / 풀→불 ×0.5. `S=물` → 물→물 ×1 / 물→풀 ×0.5 / 물→불 ×2. `S=불` → 불→물 ×0.5 / 불→풀 ×2.)

> **★ 이것이 "속성 2개가 필요하다"(미결 M10)가 소멸한 정확한 이유다.** 대비 웨이브도, 강제 3장도, 두 번째 속성도 필요 없다 — **속성 1개 + 70/10/10/10이면 유레카가 산술적으로 보장된다.** 그리고 이 보장은 **6개 테마 전부**에서 성립한다(혼합 규칙이 counter/prey에서 파생되므로, §8.2 + `check.mjs` S8).
> 부수 효과: `S = prey`(최악의 선택)를 골라도 교육적이다 — 70%에 ×0.5를 맞으며 **"틀린 스탠스"의 감각**을 배우고, `stanceHintPulse`가 정답 키를 가리키고 있다.

### 8.5 ★ `stanceHintPulse`의 정확한 동작 (정본 키의 상세)

정본: *"HUD 키캡 중 지금 화면의 적에게 ×2인 키가 부드럽게 맥동. 스테이지 1 전용, 이후 영구 OFF"* (§9.9). 이 섹션이 확정하는 것은 **대상 선정**과 **투자 0 처리** 둘뿐이다.

| 항목 | 확정 |
|---|---|
| 대상 | 아레나 위 **적 개체 수 기준 최다 속성**에 대해 ×2인 키 **1개**. 동수면 `Q W E R` 순서상 앞선 키. 0기면 맥동 없음 |
| 주기 | `visual.stance` 어휘 재사용, 부드러운 맥동(트위치 유발 금지) |
| **투자 > 0인 키** | **속성색 맥동** |
| ★ **투자 0인 키** | ★ **은색(탈색) 맥동** |
| 범위 | `stanceHintPulseStageMax = 1` → 스테이지 2부터 **영구 OFF** |

> ★ **투자 0인 정답 키를 은색으로 맥동시키는 것이 이 힌트의 절반이다.** 색이 "저 키가 정답"을 말하고, 은색이 "그런데 너는 비어 있다"를 말한다 — 그리고 §7.5의 **탈색 링**이 그 문장을 이미 가르쳤으므로(사슬 #2) 플레이어는 **눌러보지 않고도 안다.** → 다음 드래프트에서 **어느 속성 카드를 고를지가 결정된다.** 이것이 "정보 기반 선택"이다.
> 만약 투자 0인 키를 **속성색으로** 맥동시키면 **I-2(색은 거짓말하지 않는다) 위반**이다 — 화면의 속성색은 "지금 적용할 배율"을 뜻하는데, 그 키는 아무것도 적용하지 않는다. **은색 맥동이 유일하게 정직한 표현이다.**

### 8.6 `Q`(노말)는 no-op이 아니다 — 온보딩이 그것도 가르친다

| 순간 | 무슨 일이 |
|---|---|
| 사슬 #2 | 투자 0인 `W` = **탈색 링 + "빈" 저음** |
| 같은 시점 | `Q` = ★ **은색 링 정상 확장 + 정상 SFX** (탈색 없음, §7.5) |
| → | **두 개가 시각적으로 다르다** = "Q는 비어 있는 게 아니라 **선택이다**" |
| `S = prey`로 70%에 ×0.5를 맞는 중 | `Q`를 누르면 **×1로 개선** — 방패 호가 사라지고 얇은 틱이 돌아온다 |
| → | ★ **"모르겠으면 Q"** = 절대 브릭 없음의 바닥이 몸으로 학습된다 |

이 레슨은 **최종 보스에서 정확히 값을 한다**: 불+4 특화 빌드는 테트라크의 물의 관·불의 관을 **`Q`로 ×1로 민다**(불 스탠스면 ×0.5). §8.16이 "Q의 존재 이유가 여기서 증명된다"고 쓴 것의 **학습 경로가 스테이지 1의 첫 30초에 있다.**

### 8.7 온보딩이 거절당했을 때 (구조적 안전망)

첫 드래프트의 속성 카드는 **보장이지 강제가 아니다.** 플레이어가 무기·패시브를 고르면?

| 안전망 | 동작 |
|---|---|
| ① `guaranteeNewWeaponUntilSlots: 2` | 무기를 골랐다면 슬롯 2가 찬다 → ★ **스트립이 [은색][은색][—][—]** = **"부여 우선순위의 그릇"이 먼저 보인다.** 나중에 속성을 고르면 **①만 물든다** = "앞의 N개"가 **대비로** 드러난다 → **순서가 반대여도 교육적** |
| ② `elementCardPity: 6` | 6드래프트 내 속성 카드 **확정 재제시** |
| ③ 산술 | 런당 레벨업 ~54회 / 6스테이지 ≈ **스테이지 1에만 ~9회** → **피티가 스테이지 1 안에서 반드시 걸린다** |
| ④ `stanceHintPulse` | 그동안 정답 키가 **은색으로 계속 맥동**한다 → 거절이 길어질수록 힌트가 커진다 |

**→ 어떤 선택 순서로도 스테이지 1 안에서 ①②③이 선다.** 강제가 없어도 우회가 없다.

### 8.8 온보딩이 쓰는 텍스트: 0줄

사용 채널: ① 테마 배너의 **수치와 글리프**(문장 아님) ② **탈색 링** ③ **자동 장착 + 키캡 번쩍** ④ **3배율 감각**(시각 + 오디오) ⑤ **슬롯 스트립 pip** ⑥ **은색/속성색 키캡 맥동** ⑦ **카드의 before → after 슬롯 그림.**
**설명문은 어디에도 없다.** 그리고 `Esc` → PAUSE에 **4×4 상성표가 상시 비치**되어 있다(§6.6) — 배움이 실패해도 참조가 실패하지 않는다.
