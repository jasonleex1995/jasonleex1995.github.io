| # | 규칙 | 파생원 |
|---|---|---|
| **F-1** | **클럭이 멈춘 전체화면 상태는 플레이필드를 덮어도 된다.** DRAFT·SHOP·PAUSE·DEATH·STAGE_CLEAR·HEAL·THEME_BANNER는 게임 클럭이 정지하므로 **회피가 존재하지 않는다.** ★ **v1.1 정정**: I-4(적 탄은 절대 가려지지 않는다)는 **§12.3의 레이어 스택 0~13이 물리적으로 보장**한다 — 「I-4는 알파가 아니라 레이어 구조가 보장한다」(§12.3). 이 화면들은 **그 스택 안의 레이어가 아니라 스택 위의 화면**이므로 I-4의 정의역 자체에 들어오지 않는다. **I-4를 "무력화"하는 것이 아니라, 애초에 적용 대상이 아니다.** | §7.1 I-4 · §12.3 · §6.4 · §6.6 |

## 1. 상태 기계 — 화면 단위 전개

정본 §6.5의 다이어그램을 **화면·트리거·입력·클럭** 4열로 전개한 것이다. 상태·전이는 정본과 1:1이며 추가·삭제가 없다.

| 상태 | 클럭 | 나가는 트리거 | 다음 |
|---|---|---|---|
| `TITLE` | 메뉴(§4.4) | `Space` / `O` / 무입력 `flow.attractIdleSec`(20) | DIFFICULTY / OPTIONS / ATTRACT |
| `ATTRACT` | 게임(§2.2) | 아무 키 / 잡몹 페이즈 종료 | TITLE |
| `DIFFICULTY` | 메뉴 | `Enter` / `Escape` | RUN_START / TITLE |
| `RUN_START` | — | **0프레임 논리 상태** (시드 발급 → 테마 5종 일괄 추첨 → 로드아웃 초기화) | THEME_BANNER(1) |
| `THEME_BANNER` | **정지** | `flow.themeBannerSec`(2.5) 경과 또는 `Space` | MOB_PHASE(i) |
| `MOB_PHASE` | **진행** | `stage.mobPhaseSec`(120) 경과 | DRAFT_QUEUE |
| ┗ `DRAFT` | **완전 정지** | 카드 확정 | MOB_PHASE 복귀 |
| ┗ `CRISIS` | **진행** | — (내부 서브구간 95~120, 독립 상태 아님) | — |
| `DRAFT_QUEUE` | **정지** | 큐 소진 (0장이면 0프레임 통과) | BOSS_INTRO / HEAL |
| `BOSS_INTRO` | **진행**(보스 무적·발사 없음·**타이머 정지**) | `stage.bossEntrySec`(3.0) 경과 | BOSS |
| `BOSS` | **진행** | 코어 격파 / HP 0 / 타이머 만료 | STAGE_CLEAR / DEATH |
| `STAGE_CLEAR` | 정지 | `flow.stageClearSec`(2.0) 또는 `Space` | DRAFT_QUEUE |
| `HEAL` | 정지 | `flow.healSec`(1.5) 또는 `Space` | THEME_BANNER(i+1) |
| `SHOP` | 정지 | `Escape` + 확인 1회 | MOB_PHASE(i+1) |
| `PAUSE` | **전 클럭 정지** | `Escape` → `flow.pauseResumeCountdownSec`(1.5) | 직전 상태 |
| `OPTIONS` | 정지 | `Escape` | 직전 상태(TITLE 또는 PAUSE) |
| `DEATH` | 정지 | 컨티뉴 / 포기 / `Space` | 직전 상태 재개 / RESULTS / RUN_START |
| `RESULTS` | 메뉴 | `Space` / `Escape` | RUN_START / TITLE |

**★ 스테이지 루프에 `DRAFT_QUEUE`가 두 번 있다 (정본 §6.4·§6.5의 정확한 독해)**
`MOB_PHASE → DRAFT_QUEUE → BOSS_INTRO` (위기 세션 25초 동안 쌓인 큐를 **보스 인트로 전에** 소화) 와
`STAGE_CLEAR → DRAFT_QUEUE → HEAL` (보스 페이즈 큐 — **구조적으로 항상 0장**, §6.4).
전자가 실질이고 후자는 **안전망**이다. 후자가 0장이므로 §6.5의 `STAGE_CLEAR → DRAFT_QUEUE`는 정상 런에서 0프레임을 통과한다. → **"보스 깨고 나서 카드를 고른다"는 체감은 존재하지 않고, 카드는 항상 보스 *전에* 소화된다** = 보스 = 가져온 빌드의 시험지(§8.8).

**포기(abandon)는 PAUSE에 없다 — 이제 정본의 키다.** ★ **`flow.pauseAllowsAbandon = false`**(§5.5, 05-④ 채택). 정본이 명문화한 근거: ① 포기 키가 있으면 **"지금 죽어서 점수를 확정할까"라는 체스식 최적화**가 생긴다 ② 30분 런의 탈출은 `Escape` 무제한 일시정지 + `blur` 자동 정지가 이미 준다 ③ 오락실에 "포기" 버튼은 없다. **런을 끝내는 방법은 클리어 / 사망(→ DEATH에서 `포기` → RESULTS) 둘뿐**이고, 그래서 `bossTimerSec`가 진짜 허들이 된다. 런 이탈 = 새로고침 = 런 소멸(§6.5).

**TITLE에서 `O`가 열린다 (v1.1 정정).** §5.5의 v1.0 표는 `O`를 PAUSE 전용으로 적었으나 §6.5 다이어그램·§14와 **2:1로 그 표가 오기**였고, v1.1이 정정했다(05-② blocker 채택). 그대로 뒀다면 §14가 "TITLE에서만"으로 확정한 **`기록 삭제`가 도달 불가능한 기능**이 됐다.

### 2.2 ATTRACT — 심사자가 보는 첫 3초

**정의: 인증 봇이 실제로 스테이지 1을 플레이하는 것.** 리플레이도, 스크립트도, 영상도 아니다.

| 항목 | 확정 |
|---|---|
| 구현 | `flow.attract.difficulty = "normal"`, 스테이지 1, 봇 정책 = **`bot.baseline`** — ★ **v1.1에서 4축이 되었다**: `draft: generalist` / `farm: balanced` / `stance: greedyNearest` / **`shop: survivalFirst`**(§10.4 · §10.4.1). 시드는 매 회 새로 발급 |
| ★ `shop` 축 | **어트랙트에서는 도달하지 않는다** — 상점은 스테이지 1 **클리어 후**에 열리고(§6.5) 어트랙트는 **잡몹 페이즈에서 끝난다**(아래). 축은 4개지만 어트랙트가 실제로 소비하는 것은 3개다. **정합이지 예외가 아니다** |
| 루프 | 게임과 **문자 그대로 동일한 `step()`** 에 `bot(world)`의 입력을 주입 (§10.4의 시뮬 루프와 동일) |
| 반응 지연 | `latencyTicks = round(bot.reactionMs / 1000 × 60 × speed)` (§10.4.1). 어트랙트 = Normal(`speed 1.0`) → **15틱.** 봇이 인간처럼 늦게 반응하는 것이 화면에 보인다 |
| 드래프트 | 카드 3장을 **표시하고** `flow.attract.draftDwellSec`(0.8) 후 봇이 선택 → 카드가 뽑히는 것이 보인다 |
| ★ 재정렬 | `bot.slotOrder = "dpsDescending"`(§10.4) → 봇은 드래프트 화면에서 `Tab`으로 **슬롯을 명목 ST DPS 내림차순 정렬**한다. **이것이 화면에 보인다** — 즉 어트랙트가 §11.1의 `elementLevel` 카드가 가르치는 그 최적화를 **시연**한다. 봇의 편의가 아니라 **게임의 산술적 골격**이다(§13.2-③의 전제) |
| 종료 | **잡몹 페이즈 종료 시점**(`flow.attract.endAfterMobPhase = true`) → TITLE. 보스전은 보여주지 않는다 |
| 중단 | **아무 키** → 즉시 TITLE (F-3에 의해 그 키는 TITLE에서 다시 먹지 않는다) |
| 기록 | 어트랙트 런은 **점수를 집계하지 않고 localStorage를 건드리지 않는다** |
| HUD | **전체 HUD 그대로** — 봇의 스탠스 전환이 좌 패널에서 보여야 한다 |

**★ 왜 잡몹 페이즈에서 끊는가**: 잡몹 페이즈 120초의 마지막 25초 = **위기 세션**이고, 그 60기(스테이지 1은 `swarmTotalScale[0]=0.5` → 30기)는 **전원 테마 속성**(`crisisElementRule: "themePure"`, §8.10)이다. 봇이 정답 스탠스를 켜두면 화면이 ×2 화이트-핫 + 저역 크런치로 뒤덮인다. **어트랙트가 게임의 셀링 포인트에서 정확히 끝난다.** 보스전(최대 3분)을 붙이면 어트랙트 1회가 5분이 되고, 클라이맥스가 중간에 파묻힌다.

**★ 왜 봇인가 (비용 논증) — v1.1이 구조를 승인했다**: 봇은 인증 파이프라인(§10.4)에 **이미 존재하는 코드**다. 어트랙트를 위해 만드는 것이 아니라, 만들어진 것을 화면에 띄우는 것이다. 그리고 이것은 공모전 스토리의 실물이다 — **"타이틀 화면에서 지금 플레이하고 있는 그 봇이 이 게임의 밸런스를 인증했습니다."**

정본이 채택한 구조 3건(05-⑤, §10.2):
| 구조 | 확정 |
|---|---|
| **`src/core/bot.js`** | 봇을 `tools/`에서 **core로 승격**. 순수(주입된 rng만 사용, §9.1의 금지 식별자 전부 적용). 봇이 `tools/`에 있으면 배포물이 `tools/`를 import하게 되어 **§9.1의 모듈 경계가 무너진다** |
| **RNG 스트림 8번째 `bot`** | `bot.reactionJitterMs` 용. ★ **동결 파기의 대가가 0이다** — 독립 스트림의 정의상 **인간 런에서 `bot` 스트림은 draw가 0회**이므로 기존 7 스트림의 수열이 **비트 단위로 불변**이다. **이전 인증이 무효가 되지 않는다.** 정본: *"이것이 스트림 분리를 애초에 한 이유이며, 지금 그 배당금을 받는 것"* |
| **`check.mjs` S11** | 스트림 목록 7 → **8종** |

### 2.3 DIFFICULTY

4종 **전부 처음부터 선택 가능**(§6.1, 해금 그라인드 = 기둥 위반). 커서 기본 = **Normal 고정**(마지막 선택을 기억하지 않는다 — `opts`에 그 키가 없고, 만들면 §14 스키마 변경이다).

각 줄에 §6.1 표의 값을 **그대로** 표시한다:

```
NORMAL    ×1.0   보스 타이머 3:00   런 ~30분   점수 ×1.0    최고 ——
HARD      ×1.5   보스 타이머 2:00   런 ~20분   점수 ×1.6    최고 124,300
HELL      ×2.0   보스 타이머 1:30   런 ~15분   점수 ×2.5    최고 ——
DISASTER  ×3.0   보스 타이머 1:00   런 ~10분   점수 ×4.0    최고 ——
```

> **"Hell = 적이 세다"가 아니라 "Hell = 15분 스피드런"임이 선택 시점에 읽혀야 한다.** 난이도의 유일한 축이 배속이므로(§6.1), 화면이 보여줄 것도 **시간**이다. 적 스탯을 언급하는 문구는 거짓말이 된다(적 스탯은 난이도로 변하지 않는다).
> **난이도는 런 시작 후 변경 불가**(§6.1). 이 화면이 그 결정의 유일한 지점이다.
> ★ **표기 규칙 준수 (§11.3, 02-B9 채택)**: `difficulty[].speed`(1.5)와 `scoreMul`(1.6)은 **Hard에서 다른 수**다. **라벨 없는 `×N`은 배속 하나뿐**이고 **점수 배율은 반드시 `점수 ×N`으로 라벨**한다 — 위 4줄이 그 규칙의 형태다(`×1.5` = 배속 / `점수 ×1.6` = 배율). 이 문서에서 라벨 없는 `×N`이 등장하는 곳은 **상단 띠의 배속 배지 하나뿐**이다(§4.1).

**★ v1.1 정정 — 물결표가 이 배너의 정직성이다 (§8.2.1 확정)**

v1.0판 이 문서는 「`mix`의 4개 수치를 **그대로**」 표시하고 「수치를 숨기지 않는다」고 썼다. **그 약속은 지킬 수 없는 것이었다** — 정본 v1.1이 그 이유를 산술로 밝혔다:

| 확정 (정본 소유) | |
|---|---|
| **S8 검사 대상** | ★ **저작 리스트** (`stages[].waves[]` 전체의 개체 수 기준). **실제 스폰이 아니다** |
| 허용 오차 | **±3%p** (저작 리스트 대비) |
| **배너 표기** | ★ **물결표 필수** |
| **실제 조우와의 괴리** | **≤ 5%p** — 검사하지 않으나 **정본이 명문으로 허용**한다 |

- **왜 「그대로」가 거짓말이었나**: 플레이어가 실제로 만나는 것은 **앞 ≤14 웨이브**이고 `waveClearAdvance`(§8.7)가 남은 스케줄을 압축하므로 **조우 비율이 플레이 스타일의 함수**다. 정본의 실측: `sea` s1 소극 파밍 11웨이브 → **물 75.0% (+5.0%p).** `70%`라고 쓰면 **75%를 만난 플레이어가 배신당한다.** `~70%`라고 쓰면 **약속의 정밀도가 약속의 이행 가능성과 일치한다.**
- **정보 기둥은 손상되지 않는다**: `elementCapTotal = 6`의 다이얼(`4/2/0` vs `2/2/2`)은 **70 대 10의 차이**에서 결정되지 70 대 75에서 결정되지 않는다. **정보 기둥이 요구하는 것은 소수점의 진실이 아니라 "무엇이 다수인가"의 진실**이고, 그것은 어떤 파밍 스타일에서도 참이다(70→75는 **더 참이 된다**).
- **그리고 게이트는 여전히 계약이다**: `mixGranularity: "perWave"` + **`check.mjs` S8(저작 리스트 ±3%p)**이 배너를 **광고가 아니라 계약**으로 만든다. 바뀐 것은 **계약의 대상이 명시됐다는 것**뿐이다 — v1.0은 "무엇의 비율인지"를 정하지 않아 **저작 리스트는 0.0%p인데 실제 조우가 5%p 벗어나는** 구멍 위에 서 있었다.
- **물결표는 데이터가 아니라 표기다** — 초안 E의 `70/12/12/6`을 폐기한 근거("규칙에서 파생되지 않는 임의 데이터")는 그대로 유효하다.

### 3.2 MOB_PHASE — HUD 계약

이 상태에는 전용 화면이 없다. 플로우가 성립하기 위해 **상시 표시되어야 하는 것**의 목록은 §9의 계약표에 있다. 이 상태에서 이 섹션이 소유하는 것은 **위기 세션 예고**와 **드래프트 큐 배지** 둘뿐이다.

**위기 세션 예고 (`stage.crisisWarnSec` = 3.0) — ★ v1.1에서 정본이 흡수했다 (§7.12.2, 05 채택 / 02-B7 기각)**

**이 규격은 이제 정본 §7.12.2가 소유한다. 아래는 인용이다.**

| 채널 | 확정 규격 (§7.12.2) |
|---|---|
| ① 상단 띠 | 페이즈 잔여 표시가 **`새떼 3 → 0` 카운트**로 전환 (`hud.fontHeroPx`, 은색) |
| ② 아레나 | ★ **`view.spawnLineY`(−40) 바로 아래에 자홍 점선 수평선 1개**, **전폭 580**, `visual.telegraph.strokePx`/`dashPx` 그대로, **3.0 게임초** 지속 |
| ③ 오디오 | 상승 스윕 + `bgm ×0.5` 덕킹(§7.10) |
| **금지** | 화면 가장자리 호박 펄스 **금지** · 띠의 속성색 펄스 **금지** |

> ★ **②가 이 설계의 핵심**: 자홍 점선 수평선은 §7.4의 **`wall` 이미터 텔레그래프와 문자 그대로 같은 도형**이다. 새 어휘 0, 새 색 0. 플레이어는 이미 "자홍 점선 = 곧 저기가 아프다"를 배웠으므로 **위기 예고를 배울 필요가 없다.** 그리고 3.0 ≥ 모든 텔레그래프 하한(§12.4)이므로 예고 없는 스파이크가 구조적으로 불가능하다.
> ★ **`visual.crisisWarn` 키는 존재하지 않는다** (§7.12.2 · §18.3-02-2). 02가 제안한 「띠 전체가 테마 속성색으로 3회 펄스」는 **§7.2의 가장 강한 예약(「적 탄·텔레그래프에 4속성 색 사용 절대 금지」)의 정면 위반**으로 기각됐다 — 위기 예고는 **위협 예고 = 텔레그래프**다. 정본이 덧붙인 결정적 근거: 02의 우려("예고가 어느 키를 누를지를 말해야 한다")는 정당하나 **그 정보는 이미 화면에 3중으로 있다** — `crisisElementRule = "themePure"`이므로 새떼는 100% 테마 속성이고, 테마 속성은 **배너 · 좌 패널 혼합 바 · 화면의 모든 적 외곽선**이 말한다. **예고가 그것을 네 번째로 말할 이유가 없다.**
> 화면 가장자리 호박 펄스도 **쓰지 않는다** — 호박은 §7.12.4의 **5곳 전수 목록** 전용이며 전부 **"내 몸에 무슨 일이 일어난다/일어났다"** 한 문장이다. 새떼 진입은 내 상태가 아니라 외부 위협이다(F-2).

**드래프트 큐 배지**: 위기 세션 중 레벨업 = 큐 적립(§6.4) → HUD에 **`LV UP ×N`**. 이 배지는 위기 세션 25초 동안만 존재하고, 페이즈 종료 시 DRAFT_QUEUE가 소진한다.

**`무피격 ✔`는 약속이지 지급이 아니다.** STAGE_CLEAR는 무피격 성립을 **표시**하고 HUD 무피격 pip 6칸 중 그 스테이지 칸을 **은색 채움**으로 바꾸지만(§7.12.5의 3상태: 진행 중 = 은색 외곽선 빈 원 / **성립 = 은색 채움** / 깨짐 = `palette.neutralGray` ✕), **점수는 붙지 않는다.** 근거는 §6.2 — 컨티뉴가 무피격을 **런 전체 소급 무효**(§11.4)로 만들 수 있으므로, 이미 지급한 점수를 회수하는 UI가 존재해서는 안 된다.
**실드로 막은 피격은 피격이 아니다** (`shieldPreservesNoHit: true`, §11.3) → pip는 채움을 유지한다.

**스테이지 6은 STAGE_CLEAR → RESULTS**(§6.5). HEAL·THEME_BANNER·SHOP 없음.

## 4. 보스 타이머 UX

### 4.1 3단 상태 — ★ v1.1에서 정본이 흡수했다 (§7.12.1, 05 채택 / 02 기각)

타이머는 상단 띠 중앙(§1.2, y 0~40), 게임시간 `3:00 → 0:00` 표시. 옆에 **배속 배지 `×3.0`** (= 이 문서에서 라벨 없는 `×N`이 등장하는 유일한 자리, §11.3의 표기 규칙).

**이 규격은 이제 정본 §7.12.1이 소유한다. 아래는 인용이다.**

| 상태 | 조건 | 색 | 연출 | 오디오 |
|---|---|---|---|---|
| **정상** | 잔여 > `timerWarnSec`(60) | `palette.element.normal` (은색) | — | — |
| **경고** | 잔여 ≤ 60 | ★ **색 불변 (은색)** | `visual.timer.warnScale` **1.15** + `warnPulseHz` **0.5** | — |
| **빨간불** | 잔여 ≤ `timerRedAlertSec`(30) | ★ **`palette.threat.enemyBullet` (자홍)** | `alertScale` **1.30** + `alertPulseHz` **1.0** | **1Hz 틱**(§7.10) |

> **★ 「노랑」 라벨은 삭제됐다 (05-① 채택)**: v1.0 §6.3의 `타이머 경고 1 (노랑)`은 **팔레트 키가 아니었다** — §7.2의 `palette`에 노랑은 **존재하지 않고**, 노랑 `#FFE24A`는 §7.2가 **호박 상태이상 채널과 명도·색상이 사실상 같다는 이유로 명시적으로 배제**한 색이다. 그대로 뒀다면 **구현자가 팔레트에 없는 색을 발명**했다. v1.1의 §6.3은 이제 `timerWarnSec | 60 (색 라벨 없음 — §7.12)`이다.
> **★ 왜 경고(60초)에 색이 필요 없는가**: 크기(1.15)·맥동(0.5Hz)·그리고 **30초에 오는 것**이 이미 2단 경고다. §7.2 대원칙: **hue는 속성 전용 예약 채널.**
> **★ 왜 빨간불에 자홍인가**: 자홍 = **"이것이 나를 죽인다"의 단일 채널**(적 탄·텔레그래프). 타이머 만료 = **즉사**(`boss.timerExpire: "kill"`). **자홍은 은유가 아니라 참이다.** 정본의 정리: ***"이 게임에 빨강은 없다. 자홍이 이 게임의 빨강이다"*** — 「빨간불」은 정본이 붙인 **이름**이지 색 지정이 아니었다. 그리고 자홍은 **`ShiftLeft`가 유효한 창의 시각 정의**이기도 하다(§4.3).
> **★ 호박은 여기 없다 (02 기각의 근거)**: 02는 「≤60초 = 호박」을 제안했고 정본이 기각했다 — ① **호박의 정의는 "내 상태"**인데 타이머는 **외부 시계**다("코인이 부족한 것도 내 상태가 나쁜 것"이 되어 호박이 무한 확장된다) ② **위치가 결정적이다** — 타이머는 **아레나 오버레이 띠 = 플레이필드 안**이고 §7.2의 예약은 플레이필드 밖 + 온-플레이어다 → **문면 위반.** 화면 가장자리 비네트도 같은 이유로 쓰지 않는다(F-2).

### 4.2 타이머가 멈추는 곳 = 정확히 둘

`PAUSE` + `boss.phaseTransitionSec`(1.5, 최대 2회) — §6.3. **드래프트는 보스전에 발생하지 않으므로 세 번째 후보가 없다**(§6.4). 실효 최대 = **183 게임초**이고 표시는 180→0이며 전환 중 **멈춘 채로 보인다**(멈춘 것이 보이는 것이 곧 "이 3초는 공짜"라는 고지다).

정본 §11.4 그대로(`ShiftLeft` · 빨간불에서만 유효 · **+30 게임초** · 보유 상한 2 · 대가 = 그 보스의 남은 시간 보너스 0). ★ **v1.1에서 이 섹션의 UX 경계가 두 개의 정본 키로 흡수됐다** — `hud.tokenKeycapGatedDisplay`(§9.4.1)와 `시간 보너스 ✕`의 자리(§7.12.5).

> ★ **정본이 자리를 준 근거 (§7.12.5)**: *"토큰은 **타이머의 대가**다. 대가를 타이머 옆에 두지 않으면 `timeTokenForfeitsTimeBonus`가 결과 화면에서 처음 알려지는 함정이 된다."* 그리고 y 30~40은 **`[Shift] +30초` 힌트와 같은 자리이며 배타적이다** — 토큰을 쓰면 힌트가 사라지고 ✕가 온다. **같은 픽셀, 같은 의미축.**
> ★ **`hud.tokenKeycapGatedDisplay` (§9.4.1)**: 정본 §11.4는 "빨간불일 때만 유효, 그 외 **입력 무시**"라고만 했고 **입력이 무시된다는 것을 알리는 채널이 없었다** → *"왜 Shift가 안 먹지"*가 발생 불가해진다. **오발 불가**(§11.4)이므로 은색·비활성 아이콘은 **금지 표시가 아니라 상태 표시**다.

### 5.3 컨티뉴 — 부활

정본 §11.4의 표를 **재정의하지 않는다.** 이 섹션이 더하는 것은 **소급 무효의 화면 표현** 하나:

| 순간 | 화면 |
|---|---|
| 컨티뉴 확정 | 코인 −150 (코인 표시와 **환산 점수 표시가 동시에** `150 × coinToScore(20)` = **3,000 × 난이도 배율**만큼 떨어진다 — §11.4가 "코인 150 = 점수 −3000 자동"이라 부르는 그것) |
| **동시** | ★ **HUD 무피격 pip 6칸이 전부 즉시 `palette.neutralGray` ✕로 바뀌고**(§7.12.5 — **소등이 아니라 ✕다**) `퍼펙트 ✕` 마크가 **같은 행 우측 끝**에 켜져 **런 종료까지 남는다** |
| 부활 | 아레나 하단 중앙 · HP 만피(`continueHealToFull`) · 적 탄 전량 소거(**잡몹은 유지**) · `flow.continueIframeSec`(3.0) 점멸 무적 · **빌드 전부 유지**(레벨·무기·속성·패시브·슬롯 순서·스탠스·스톡) |
| 보스전이면 | 보스 HP **깎은 그대로 유지** · 타이머 = **`max(현재 잔여, flow.continueTimerRestoreSec)`**(60) |
| ★ 재개 직후 타이머 | ★ **「경고」다. 빨간불이 아니다.** `max(·, 60) ≥ 60`이므로 §7.12.1의 빨간불 조건(≤30)에 **산술적으로 걸릴 수 없다.** 잔여가 60을 넘던 중 HP로 죽었다면 그 값이 유지되므로 **정상**일 수도 있다. (v1.0판 이 문서는 "재개 즉시 빨간불"이라 썼다 — **틀렸다.** 같은 칸 안에서 자기모순이었고 정본의 어떤 문장도 그것을 지지하지 않았다.) |

> ★ **pip 6칸이 그 자리에서 ✕가 되는 것이 "이중 비용"을 즉시 보여준다.** 정본은 컨티뉴의 대가를 `perfectBonus` + **모든 스테이지의 `stageNoHitBonus` 소급 무효** + 코인 150으로 정했다(§11.4). 그 대가가 20분 뒤 결과 화면에서 처음 보이면 **함정**이다. 지금 6칸이 꺼지는 것을 보고 코인을 내는 것이 **결정**이다. — ★ **정본 §7.12.5가 02의 「텍스트 1행」안을 기각하며 인용한 것이 정확히 이 논거다**: *"05가 계약으로 요구한 것은 6칸이고 그 이유가 명확하다 — 컨티뉴의 대가를 그 자리에서 6칸이 동시에 꺼지는 것으로 보여준다. 텍스트 1행은 그 장면을 만들지 못한다."*
> ★ **타이머를 3:00으로 리셋하지 않는 것이 핵심**(§11.4): 리셋하면 컨티뉴는 "6분 보스전"이 되어 **게임의 유일한 허들이 코인으로 삭제**된다. **60초 = 마무리할 수 있는 시간이지, 다시 싸울 수 있는 시간이 아니다.**
> 그리고 §6.2의 "무피격은 정산 항목이지 라이브 항목이 아니다"가 여기서 값을 한다 — **회수해야 할 점수가 애초에 지급된 적이 없으므로**, 라이브 점수가 뒤로 가는 일이 발생하지 않는다.

## 6. 점수 상세

### 6.1 항목 — 정본 §11.3이 유일한 소유자

이 섹션은 **키도 값도 판정 규칙도 재정의하지 않는다.** 아래는 정본이 정한 것을 **언제 어디에 어떻게 보여주는가**만 확정한다.

### 6.2 ★ 라이브 점수 vs 정산 점수 (이 섹션의 결정)

| 구분 | 항목 | 확정 시점 |
|---|---|---|
| **라이브** (HUD에 실시간) | 처치 점수 · **상성 처치 보너스** · `midBossClearBonus` · `bossClearBonus` · **남은 시간 보너스** | 발생 즉시 |
| **정산** (RESULTS에서 일괄) | `stageNoHitBonus` · `perfectBonus` · `runClearBonus` · **남은 코인 환산** | 런 종료 |

**★ 왜 나누는가 — 3개의 문제가 한 번에 닫힌다**
1. **소급 무효가 자연스러워진다.** 컨티뉴는 `stageNoHitBonus`를 **런 전체 소급 무효**로 만든다(§11.4). 라이브에 넣으면 점수가 −18,000 뒤로 가는 화면을 만들어야 한다. 정산 항목이면 **회수할 것이 없다.**
2. **남은 코인은 원래 라이브가 될 수 없다.** 잔액 × 20은 런이 끝나야 확정된다 — 상점의 `= −N 점`은 **환산 예고**이지 점수 이벤트가 아니다.
3. **오락실 정산 의식이 성립한다.** RESULTS에서 순차 타이핑으로 붙는 항목이 있어야 결과 화면이 **의식**이 된다(§11.3).

**배율 처리 (산술 동치 확인)**
```
라이브 표시 = floor( (확정 항목 합) × difficultyMul )
최종 점수   = floor( (확정 항목 합 + 정산 항목 합) × difficultyMul )
```
곱셈이 합에 분배되므로 **"전부의 합계에 최종 곱"(§11.3)과 완전히 동일**하다. 배율을 라이브에서 이미 곱하는 이유: Disaster를 고른 이유가 **실시간으로 보여야** 한다.

★ **`score.roundMode = "floor"`는 이제 정본의 키다** (§11.3, 05-③ 채택): *"배율은 항목 합계에 **1회** 곱하고 `floor`는 **최종에 1회만**."* 정본이 명문화한 근거 — 배율이 비정수(1.6 / 2.5)라 정수화 규칙이 **반드시 필요**하고, 규칙이 없으면 구현자가 항목마다 반올림해 **라이브 점수와 정산 점수가 갈린다** → §11.3이 요구하는 "놓친 보너스를 회색으로 함께 표시"가 **1점 어긋난 숫자를 보여준다 = 신뢰 붕괴.**
> **주의**: 위 두 식에 `floor`가 각각 하나씩 있는 것은 **두 벌의 반올림이 아니다** — 라이브 표시는 **표시일 뿐 누산되지 않는다.** 누산되는 것은 항목 합(정수)이고, `floor`는 **화면에 숫자를 그리는 순간에만** 적용된다. 최종 점수의 `floor`가 정본이 말하는 "최종 1회"다.

### 6.3 상성 처치 보너스 — 화면에서 보이는 곳

정본 §11.3: 개체마다 `총 피해`·`×2 피해` **2개 누산기**, 처치 시 지분 ≥ 0.5면 성립 → 처치 점수 ×1.5.

**막타가 아니라 지분이므로, 이 판정은 UI로 노출할 수 없고 노출해서도 안 된다.** 개체마다 지분 게이지를 띄우면 그건 §11.3이 배제한 **체스식 최적화**를 도로 초대하는 것이다. 이 섹션의 확정:

| 지점 | 표현 |
|---|---|
| 인게임 | **없음.** ×2 히트의 감각(§7.7: 화이트-핫 + 임팩트 프리즈 + 파티클 3 + 저역 크런치)이 곧 지분을 쌓고 있다는 피드백이다 |
| STAGE_CLEAR | 스테이지 소계에 `상성 보너스 +12,400` 1줄 |
| RESULTS | 런 합계 1줄 |

> **최적 플레이 = "제대로 된 스탠스로 싸우기"**(§11.3). 화면이 지분을 보여주면 최적 플레이가 "지분을 관리하기"로 변질된다. **보여주지 않는 것이 설계다.**

> **"20분 뒤에야 알게 되는 언더파를 지금 알려준다"**(§11.4). 4스테이지에서 처음 파선을 밑돌았다면, 그 원인은 **1~3스테이지의 파밍**이다. 파 막대 6개가 그 인과를 그림 한 장으로 보여준다 — **텍스트 0, 판정 0, 잔소리 0.**

**★ 위 목업의 가격 검산** (C-7: 목업의 숫자는 예시. 검산은 §11.2의 공식 `price(item, n) = ceil(basePrice × growth^n)`과 정합함을 보이기 위한 것 — **`n` = 그 항목의 런 내 구매 횟수**이므로 **구매 `n`회를 마친 항목의 「현재가」는 `price(·, n)`이지 `basePrice`가 아니다**)

| 항목 | `basePrice` / `growth` | 구매 `n` | 현재가 `ceil(b×g^n)` | 다음가 `ceil(b×g^(n+1))` | `= −N 점` (`×20 ×1.0`) |
|---|---|---|---|---|---|
| `reroll` | 30 / 1.60 | 1 | `ceil(48.0)` = **48** | `ceil(76.8)` = **77** | −960 |
| `potion` | 40 / 1.50 | 2 | `ceil(90.0)` = **90** | `ceil(135.0)` = **135** | −1,800 |
| `bomb` | 50 / 1.35 | — | 스톡 `3/3` = `bombStockMax` 도달 → **회색** | — | — |
| `shield` | 60 / 1.50 | 0 | `ceil(60.0)` = **60** | `ceil(90.0)` = **90** | −1,200 |
| `timeToken` | 80 / 1.80 | 0 | `ceil(80.0)` = **80** | `ceil(144.0)` = **144** | −1,600 |
| `defense` | 45 / 1.45 | 2 | `ceil(94.61)` = **95** | `ceil(137.19)` = **138** | −1,900 |

- ★ **v1.0판의 오류 2건을 정정했다**: ① `reroll`을 「1회 구매함 + 현재가 30」으로 그렸다 — **구매했으면 현재가는 48이다.** 스톡도 `2/3`이었으나 1회 구매 = 스톡 1. ② `defense`를 `94 → 136`으로 그렸다 — **`ceil(45×1.45²) = ceil(94.61) = 95`, `ceil(45×1.45³) = ceil(137.19) = 138`이다.** `ceil`을 빼먹고 절삭한 값이었다. 환산 점수도 −1,880 → **−1,900**으로 정정.
- **`⌐회색⌐` 조건 확인**: 코인 210이므로 표의 5항목 전부 구매 가능하다 → 회색의 유일한 사유는 `bomb`의 **스톡 상한**(`bombStockMax: 3`)이다. 코인 부족이 아니라 **상한 도달**이며, 이 구분이 §5.2의 컨티뉴 회색(= 코인 부족)과 다른 의미축임에 주의.

| 요소 | 규칙 |
|---|---|
| 헤더 | 방문 시점 · **★ 다음 스테이지 테마 뱃지(구성비 4수치, ★ 물결표 표기 — §8.2.1)** · 보유 코인 + **환산 점수**(`hud.coinShowsScoreValue`, §9.4.1) · 현재 HP / 최대 HP |
| 각 줄 | 아이콘 · 이름 · **효과 1줄** · **보유/상한** · **현재가 → 다음가** · ★ **`= −N 점`** (§11.2 표시 필수) |
| 구매 불가 | 코인 부족 / `maxPurchases` 도달 / 스톡 상한 도달 → **회색 + `Enter` 무시** |
| 구매 | `Enter` 즉시 (확인 없음, §5.4). **구매 불가 항목의 Enter가 무시되므로 오발의 대부분이 애초에 발생하지 않는다** |
| 나가기 | `Escape` → **확인 1회**(§5.4) → MOB_PHASE(i+1) |

**★ 헤더에 다음 테마를 다시 띄우는 이유**: §6.5의 전이 순서 근거가 **"테마가 상점 뒤면 유틸 구매가 장님"**이다. 배너는 2.5초 뒤 사라지므로, **상점에서 결정하는 순간에 테마가 화면에 없으면 그 순서의 의미가 반쯤 사라진다.** 배너(사전 고지) + 상점 헤더(결정 시점 재고지) = 순서 근거의 완성. 새 정보 0, 재배치뿐.

**★ `= −N 점`의 계산 (이 섹션의 확정 — 「정산의 순서」 위임 범위, §17)**
```
표시 = price × score.coinToScore(20) × difficulty[].scoreMul
```
난이도 배율을 **포함한다.** Disaster에서 리롤 첫 구매(30코인)는 −600점이 아니라 **−2,400점**이다. 배율을 빼고 표시하면 **고난이도에서 기회비용이 4배로 커진다는 사실이 화면에서 사라진다** — 그리고 그것이 정확히 이 게임의 중심 긴장(§11.2 "아껴서 점수 vs 써서 생존")이다. 보유 코인 옆의 환산 점수(`hud.coinShowsScoreValue`)도 같은 식.

**★ 이 식이 §11.3의 「배율은 합계에 최종 1회」와 충돌하지 않는 증명**: 남은 코인 환산은 **정산 항목**이므로 최종 점수는 `floor((… + 잔액×20) × mul)`이다. 코인을 `p` 쓰면 잔액이 `p` 줄어 최종 점수가 정확히 `p × 20 × mul` 준다 — **분배법칙.** 즉 `= −N 점`은 **새 점수 규칙이 아니라 §11.3의 식을 미분한 값**이며, 화면에 배율을 곱해 보여주는 것이 **실제 손실과 일치하는 유일한 표기**다. (`floor`는 최종에 1회만 걸리므로 이 예고값은 ±1점 오차를 가질 수 있다 — 예고이지 정산이 아니다.)

### 8.4 ★ 증명 — 어느 속성 카드를 골라도 ×2와 ×0.5가 같은 화면에 나온다

정본 §8.2는 "한 스탠스를 켜고 있으면 ×2(70%)와 ×0.5(10%)가 자연히 같은 화면에 나온다"고 선언한다. 그 선언은 **정답 스탠스를 켰을 때**의 서술이다. 그런데 첫 드래프트의 속성 카드는 **무엇이 뜰지 모르고, 무엇을 고를지는 플레이어의 자유**다. → **3개 경우 전수 확인이 필요하다.**

테마 속성 `T`, `counter`(T를 이김), `prey`(T가 이김). 혼합 = `T .70 / counter .10 / prey .10 / normal .10`(§8.2). 플레이어가 켠 스탠스 `S`:

| `S` | vs `T` (70%) | vs `counter` (10%) | vs `prey` (10%) | vs `normal` (10%) | ×2 있음 | ×0.5 있음 |
|---|---|---|---|---|---|---|
| **`counter`** | **×2** | ×1 | **×0.5** | ×1 | ✔ | ✔ |
| **`T`** | ×1 | **×0.5** | **×2** | ×1 | ✔ | ✔ |
| **`prey`** | **×0.5** | **×2** | ×1 | ×1 | ✔ | ✔ |

**3/3 경우 전부에서 ×2와 ×0.5가 같은 혼합 안에 존재한다** ✔
(검산 — 바다(T=물): `S=풀` → 풀→물 ×2 / 풀→불 ×0.5. `S=물` → 물→물 ×1 / 물→풀 ×0.5 / 물→불 ×2. `S=불` → 불→물 ×0.5 / 불→풀 ×2.)

> **★ 이것이 "속성 2개가 필요하다"(미결 M10)가 소멸한 정확한 이유다.** 대비 웨이브도, 강제 3장도, 두 번째 속성도 필요 없다 — **속성 1개 + 70/10/10/10이면 유레카가 산술적으로 보장된다.** 그리고 이 보장은 **6개 테마 전부**에서 성립한다(혼합 규칙이 counter/prey에서 파생되므로, §8.2 + `check.mjs` S8). 정본 §8.2.1이 같은 결론을 **온보딩의 관점에서 독립적으로** 쓴다: *"스테이지 1은 `introOk` 테마 + ~70/10/10/10이므로, 한 스탠스를 켜고 있으면 ×2(~70%)와 ×0.5(~10%)가 자연히 같은 화면에 나온다."*
> ★ **물결표는 이 증명을 흔들지 않는다** (v1.1 확인): 위 표는 **존재 명제**("×2와 ×0.5가 **같은 혼합 안에 있는가**")이지 비율의 명제가 아니다. §8.2.1이 허용한 **실제 조우 ≤5%p 괴리**(최악: 물 70.0 → 75.0)는 **어떤 칸의 부호도 바꾸지 않는다** — 0이 아닌 비율은 5%p로 0이 되지 않기 때문이다(최악 케이스에서도 소수 속성은 8.3%로 살아 있다). **표의 3/3 전수 확인은 저작 리스트에서도 실제 조우에서도 참이다.**
> 부수 효과: `S = prey`(최악의 선택)를 골라도 교육적이다 — 70%에 ×0.5를 맞으며 **"틀린 스탠스"의 감각**을 배우고, `stanceHintPulse`가 정답 키를 가리키고 있다.

| ★ **투자 0인 키** (이 섹션) | ★ **은색(탈색) 맥동** |

> ★ **「중간보스 제외」가 사슬 #8과 정합한다**: 중간보스는 `midBossAtSec [35]`에 오고 속성은 `notThemeAndNotNormal`이다(§8.9). 만약 중간보스를 셌다면 **1기짜리 중간보스가 30기짜리 잡몹의 최빈을 뒤집을 수 없어** 어차피 영향이 없거나, 잡몹이 비는 순간 **힌트가 중간보스를 가리켜 사슬 #8의 레슨("전환의 필요")을 대신 풀어버린다.** 정본의 제외 규칙이 옳고, 그 근거("개체 위 HP 바가 이미 답을 말한다")는 §7.6의 엘리트·중간보스 판독 규격에서 나온다.

> ★ **투자 0인 정답 키를 은색으로 맥동시키는 것이 이 힌트의 절반이다.** 색이 "저 키가 정답"을 말하고, 은색이 "그런데 너는 비어 있다"를 말한다 — 그리고 §7.5의 **탈색 링**이 그 문장을 이미 가르쳤으므로(사슬 #2) 플레이어는 **눌러보지 않고도 안다.** → 다음 드래프트에서 **어느 속성 카드를 고를지가 결정된다.** 이것이 "정보 기반 선택"이다.
> 만약 투자 0인 키를 **속성색으로** 맥동시키면 **I-2(색은 거짓말하지 않는다) 위반**이다 — 화면의 속성색은 "지금 적용할 배율"을 뜻하는데, 그 키는 아무것도 적용하지 않는다. **은색 맥동이 유일하게 정직한 표현이다.**

### 8.6 `Q`(노말)는 no-op이 아니다 — 온보딩이 그것도 가르친다

| 순간 | 무슨 일이 |
|---|---|
| 사슬 #2 | 투자 0인 `W` = **탈색 링 + "빈" 저음** |
| 같은 시점 | `Q` = ★ **은색 링 정상 확장 + 정상 SFX** (탈색 없음, §7.5) |
| → | **두 개가 시각적으로 다르다** = "Q는 비어 있는 게 아니라 **선택이다**" |
| `S = prey`로 70%에 ×0.5를 맞는 중 | `Q`를 누르면 **×1로 개선** — 방패 호가 사라지고 얇은 틱이 돌아온다 |
| → | ★ **"모르겠으면 Q"** = 절대 브릭 없음의 바닥이 몸으로 학습된다 |

이 레슨은 **최종 보스에서 정확히 값을 한다** — 그리고 **v1.1이 그 값을 숫자로 확정했다**:

> §8.16: *"4/2/0 특화는 한 관에 ×2, 한 관에 ×1.62(2투자), 남은 한 관에 **노말 스탠스로 ×1**(잘못된 스탠스로 ×0.5를 맞느니 **Q가 낫다** — **Q의 존재 이유가 여기서 증명된다**)."*
> §13.6.3의 실측: **특화 142.0s (여유 38초로 성공)** / 무투자 195.6s (15.6초 초과로 실패) / 코어 직행 209.7s (완전 불가).

**★ `Q`의 값을 이 섹션이 직접 계산한다 (정본 §13.5·§13.6의 수치로)**
```
dpsRef[6] = 708            ← 명목 · 단일표적 · 무속성 (정본 §13.5 소유, 사용자 확정 1)
× uptimeRef 0.60           ← 실효 가동률 (정본 §10.4.3 소유 — 「520 vs 708」의 진짜 원인이 이 단위였다)
= 424.8                    ← 실효 DPS. 아래 모든 시간이 이 수에서 나온다
```
| 4/2/0 특화가 남은 관(투자 0)을 때리는 법 | 실효 DPS | 그 관 1개(25,800) 소요 | 전체 |
|---|---|---|---|
| ★ **`Q` = 노말 스탠스 → ×1** | 424.8 | **60.7s** | 30.4 + 37.5 + 60.7 + 13.4 = ★ **142.0s ✔ 성공** (여유 38초) |
| **`Q`가 없어 틀린 스탠스 → ×0.5** | 212.4 | **121.5s** | 30.4 + 37.5 + 121.5 + 13.4 = ★ **202.7s ✗ 22.7초 초과 = 탈락** |

★ **`Q` 한 키가 특화 빌드의 최종 보스 클리어 가부를 가른다**(142.0 vs 202.7). 정본 §13.6.3의 특화 142.0s는 **`Q`를 누른다는 전제 위에 서 있고**, `specialistPass ≥ 0.85` 게이트(실측 **0.994**)가 그 전제 위에 서 있다. → **그 학습 경로가 스테이지 1의 첫 30초에 있다** — 사슬 #2의 탈색 링과 위 표의 `Q` 대비가 그것이다. **온보딩이 게이트를 지탱한다**는 말이 은유가 아니라 산술이다.

### 8.7 온보딩이 거절당했을 때 (구조적 안전망)

첫 드래프트의 속성 카드는 **보장이지 강제가 아니다.** 플레이어가 무기·패시브를 고르면?

| 안전망 | 동작 |
|---|---|
| ① `guaranteeNewWeaponUntilSlots: 2` | 무기를 골랐다면 슬롯 2가 찬다 → ★ **스트립이 [은색][은색][—][—]** = **"부여 우선순위의 그릇"이 먼저 보인다.** 나중에 속성을 고르면 **①만 물든다** = "앞의 N개"가 **대비로** 드러난다 → **순서가 반대여도 교육적** |
| ② `elementCardPity: 6` | 6드래프트 내 속성 카드 **확정 재제시** |
| ③ 산술 | ★ **정본 §13.5가 이 수를 소유한다**: 「보스전 진입 레벨 = `[12, 19, 26, 35, 44, 54]` → 픽 수 = `[11, 18, 25, 34, 43, 53]`」 → **스테이지 1의 보스전 진입 시점까지 11픽.** **11 > `elementCardPity` 6** → **피티가 스테이지 1 안에서, 그것도 보스를 만나기 전에 반드시 걸린다** |
| ④ `stanceHintPulse` | 그동안 정답 키가 **은색으로 계속 맥동**한다 → 거절이 길어질수록 힌트가 커진다 |

**→ 어떤 선택 순서로도 스테이지 1 안에서 ①②③이 선다.** 강제가 없어도 우회가 없다.

> ★ **v1.0판의 산술을 정정했다**: 이 문서는 「런당 레벨업 ~54회 / 6스테이지 ≈ 스테이지 1에만 ~9회」라고 썼다 — **레벨업을 스테이지에 균등 분배한 것이며 틀렸다.** XP 배분은 `xpScale[i] × spawnDensityScale[i]`에 비례하고(§13.5) 그 곱은 `[0.70, 1.36, 2.40, 3.91, 5.98, 9.00]`으로 **후반이 13배 무겁다.** 정본의 실제 계산은 **스테이지 1 = 11픽**(균등 가정보다 오히려 **많다** — 초반 레벨이 싸기 때문). **결론은 유지되고 근거만 정본의 것으로 교체한다** — 그리고 정정된 수가 논거를 **더 강하게** 만든다(11 > 6이 9 > 6보다 여유가 크다).
> ★ **같은 정본 계산이 사슬 #7의 전제도 준다**: §13.5의 제너럴리스트 궤적에서 **s1 (11픽) = 4무기 · 6레벨 · 2속성** → 스테이지 1 보스전 진입 시점에 **무기 4칸이 이미 차 있고 속성 투자가 2**다. 즉 사슬 #7의 스트립 `[속성색][은색][—][—]`은 **중간 상태**이고, 보스를 만날 때쯤엔 `[속성색][속성색][은색][은색]`이 된다 — **"앞의 N개"의 대비가 스테이지 1 안에서 두 번 보인다.**

### 8.8 온보딩이 쓰는 텍스트: 0줄

사용 채널: ① 테마 배너의 **수치와 글리프**(문장 아님) ② **탈색 링** ③ **자동 장착 + 키캡 번쩍** ④ **3배율 감각**(시각 + 오디오) ⑤ **슬롯 스트립 pip** ⑥ **은색/속성색 키캡 맥동** ⑦ **카드의 before → after 슬롯 그림.**

## 10. 이 섹션이 제안했고 **정본이 채택한** 키 — 거처 (v1.1 전면 개정)

> ★ **v1.0판 이 절의 제목은 「이 섹션이 새로 정한 값 (전부 `meta.json`)」이었다. 두 군데가 틀렸다.**

> ② ★ **「전부 `meta.json`」은 오기였다** — **`visual.timer.*`는 `rules.json > visual` 소유다**(§9.2 · §18.3-05-1). 정본의 통보: ***"`meta.json`에 넣으면 로드 실패"*** — §9.3의 「미지 키 = 에러」 + `rules.json` 루트 키 17종 목록에 `visual`이 있고 `meta.json`에는 없기 때문이다. **깨끗이 승복한다.**

**거처 (§9.2의 9파일 매니페스트 기준)**

```jsonc
// data/meta.json > flow          ← 플로우 (정본 §0.2.1 · §5.5)
{ "menuSpeed": 1.0, "edgeTriggerOnStateEnter": true, "deathAnimSec": 1.5,

// ★ data/rules.json > visual     ← 시각 규격 (정본 §7.12.1) — meta.json이 아니다
{ "timer": { "warnScale": 1.15, "warnPulseHz": 0.5, "alertScale": 1.30, "alertPulseHz": 1.0 } }
```

### 11.3 남은 요청 — **2건** (둘 다 이 섹션 소관 밖이며, 최소화의 결과)

> 이 섹션이 v1.1을 읽으며 발견한 것 중 **자기 권한으로 고칠 수 없는 것만** 남긴다. 둘 다 **02/정본의 영역**이며 이 문서는 인용만 한다(C-1).

| # | severity | 요청 | 근거 (검증함) |
|---|---|---|---|
| **⑩** | ★ **major** | **`hud.noHitRow`를 §9.4.1의 `hud` 블록에 선언해 달라.** §7.12.5가 무피격 pip 6칸의 자리를 **「우 패널, `hud.noHitRow`」**로 확정하며 **백틱 키로** 적었고 §18.3-02-5가 그것을 02에게 통보했다. 그러나 **§9.4.1의 `hud` 블록에 그 키가 없다** — §9.4는 그 블록을 `"hud": { "...§9.4.1 전 키..." }`, 즉 **전 키**라고 선언한다 | ★ **A1과 정확히 같은 결함 구조다.** A1(blocker)의 근거는 「§2.3이 `hud.hitboxAlwaysVisible`을 확정 키로 참조하는데 §9.4의 블록 목록에 `hud`가 없다 → §9.3의 「미지 키 = 에러」/「누락 키 = 에러, 폴백 금지」로 **부팅 실패**」였다. 지금 §7.12.5가 `hud.noHitRow`를 **같은 방식으로 참조**하고 있다. **처분은 둘 중 하나여야 하고 정본이 정해야 한다**: ⓐ 키로 선언(값 = 우 패널 내 y 좌표) ⓑ 키가 아니라 **02의 레이아웃 영역 이름**임을 명시(백틱 제거). ★ **major인 이유**: ⓑ로 읽으면 부팅은 되므로 blocker는 아니나, **그대로 두면 02가 키로 읽고 선언해 로더가 「미지 키」로 거부하거나, 정본이 키로 의도했는데 02가 안 만들어 「누락 키」로 거부한다.** 즉 **어느 쪽으로 읽어도 절반의 확률로 로드 실패**이며, 이것이 v1.1이 A1·`render`·`passiveHooks`에서 세 번 반복해 닫은 바로 그 결함이다 |
| **⑪** | **minor** | **§7.2 「예약 규칙」 3번째 불릿의 문면을 개정해 달라** — 현재 `- 호박은 **플레이필드 밖(화면 가장자리 밴드) + 온-플레이어 아이콘에만**. 탄·적·배경 금지.` | ★ **§7.12.4가 스스로 그 개정을 지시했으나 §7.2에 적용되지 않았다**: *"→ §7.2의 예약 문구를 **「호박은 §7.12.4의 5곳」**으로 개정한다(위치 기반 → 의미 기반)."* 그런데 §7.12.4-④⑤(상태이상 탄의 텔레그래프 링 · 탄 자체의 테두리)는 **플레이필드 안**이므로 **§7.2의 현 문면과 정면 충돌**한다 — 그리고 §7.12.4는 v1.0의 §7.2가 **"자기 §7.4를 위반하고 있었다"**고 명시적으로 진단했다. **minor인 이유**: §7.12.4가 자신을 「전수 목록」으로 선언했고 §7.2보다 후속·구체적이므로 **실무상 모호함은 없다.** 그러나 C-1(**한 항목에 답은 하나**)의 문면 위반이 정본 안에 남아 있고, **이 문서의 F-2가 §7.2가 아니라 §7.12.4를 인용하도록 바꾼 이유**가 그것이다 |

**요청이 2건뿐이고 둘 다 「이 섹션이 아니라 정본/02의 문면」인 것이 v1.1의 성과다.** v1.0판의 9건은 **전부 "정본에 자리가 없는 것"**이었고, 지금 남은 것은 **"정본 안에서 두 문장이 갈라진 곳"** 2건뿐이다.
